1. Executive Summary
This document describes the Loopboard — a proof of concept (POC) that integrates AI 
capabilities directly into the team's existing Agile workflow. The system uses GitHub Copilot 
(powered by Claude) and the Model Context Protocol (MCP) to automate the most time-
consuming parts of sprint management: writing tickets, reporting on sprints, and running daily 
standups.
The POC targets three specific pain points identified by the team:
• Ticket quality is inconsistent — descriptions lack acceptance criteria, context, and clear 
scope.
• Sprint reports are written manually, taking significant time each sprint cycle.
• Creating separate PO and Dev tickets for the same feature is repetitive and error-prone.
Goal:  By the end of Phase 1, a Product Owner or Scrum Master can type a plain-English feature 
description and have properly structured PO and Dev tickets created in Jira automatically — without 
leaving VS Code or a web browser.
2. Problem Statement
The following issues were identified during team retrospectives and daily standups:
2.1  Poor ticket descriptions
Jira tickets are often created with minimal context. Developers spend time in standup or Slack 
clarifying requirements that should have been in the ticket. Acceptance criteria are either 
missing or too vague to test against.
2.2  Manual sprint reports
At the end of each sprint, the Scrum Master manually collates completed tickets, velocity data, 
and blockers into a report. This process typically takes 1-2 hours and is prone to missed items.
2.3  Duplicate effort for PO and Dev tickets
The same feature must be described from two perspectives: a user-value story for the PO board 
and a technical task for the Dev board. These are currently written separately, leading to 
inconsistencies between the two.

The MCP (Model Context Protocol) server is the key architectural component. It exposes a set 
of tools — such as create_po_ticket and get_daily_huddle — that Claude can call automatically 
based on a natural language prompt. No manual API calls or form-filling is required.
Example interaction:  User types: "Create a PO story and dev task for password reset via email" — 
Claude calls create_po_ticket and create_dev_ticket via MCP, both tickets appear in Jira within 
seconds.

4.3  Data flow — ticket creation
When a PO or SM triggers ticket creation, the following sequence occurs:
1. User types a plain-English feature description in Copilot Chat or the React UI.
2. The prompt is sent to Claude (via GitHub Copilot extension + MCP configuration).
3. Claude generates two structured ticket drafts: one PO story, one Dev task — each with 
acceptance criteria.
4. Claude calls create_po_ticket and create_dev_ticket via the Jira MCP server.
5. The MCP server sends POST requests to the Jira REST API using the service account 
token from .env.
6. Both tickets are created in Jira and linked to each other as parent/child.
7. The React UI (or Copilot Chat) displays the ticket keys and URLs as confirmation.
4.4  Authentication flow
All Jira API calls are authenticated using HTTP Basic Auth with a Jira API token. The token is 
stored only in the .env file on the developer's machine and is never sent to the browser or 
committed to source control.

8. Technical Decisions
The following decisions were made deliberately and are documented here to prevent revisiting 
them without cause.
Why MCP instead of a custom API?
MCP is the standard protocol for connecting AI models to external tools. By building MCP-
compliant servers, the same tools work with GitHub Copilot today and with any future MCP-
compatible model or client. A custom REST API would lock us into one integration path.
Why GitHub Copilot instead of a direct Claude API integration?
The team already has GitHub Copilot licenses. Using Copilot avoids additional subscription 
cost, keeps the toolchain inside the existing GitHub ecosystem, and allows developers to 
interact with the AI companion without leaving VS Code.
Why skip the Teams Calendar MCP for now?
Microsoft Graph OAuth for calendar write access requires app registration, admin consent, and 
redirect URL configuration — significant setup overhead for a POC. Calendar automation will be 
added in Phase 3 once the core ticket and sprint features are validated.

Why Node.js for the MCP server?
The official @modelcontextprotocol/sdk is first-class in Node.js/TypeScript. The Jira REST API 
client (axios) and schema validation (Zod) also have mature TypeScript support. The entire 
stack is in one language, reducing context switching for the development team.
Why a monorepo?
The React app and MCP servers share type definitions and configuration. A monorepo with npm 
workspaces keeps them in sync without publishing to a package registry. Splitting into separate 
repos would create version drift on shared interfaces.
