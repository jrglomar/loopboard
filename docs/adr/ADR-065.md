# ADR-065: Offset History shows earned history + a manual-adjustment log (v1.54)

## Status: Accepted (2026-07-13) — implemented in v1.54

## Context

The Offset Tracker (ADR-038/044/061) let a developer's balance move three ways: banked earned per sprint,
spend derived live from Offset leaves, and a single `manualAdjust` the UI calls the **opening balance**. The
per-developer **Offset History** dialog only surfaced the *usage* side — every Offset leave (a spend). Two
gaps the user asked to close:

1. **Earned wasn't visible per sprint.** The dialog showed a summed `earned` tile but not *which* sprints
   banked it, even though the ledger already stored `bySprint[sid].earned` (ADR-061).
2. **Only one manual number.** Beyond the one-time opening balance there was no way to make ad-hoc manual
   corrections to a balance (comp for weekend on-call, a one-off deduction, …) — and no record of them.

The user chose (confirmed): a **log of ± entries** (each dated, optionally noted, removable), distinct from
the opening column, with **all of it managed from the History button**.

## Decision

- **Store: a manual-adjustment log.** Each assignee's ledger entry gains
  `adjustments?: OffsetAdjustment[]` where `OffsetAdjustment = { id, amount: int (±, non-zero), note?,
  createdAt }`. Kept separate from the one-time `manualAdjust` (opening). Optional field → back-compatible
  with pre-v1.54 store files. **`balance = Σ earned − Σ spent + manualAdjust + Σ adjustments.amount`**
  (`summarizeOffset`, which also returns the log newest-first). Pure `addAdjustment` / `removeAdjustment`
  helpers; the server assigns `id` + `createdAt`.
- **Two new MCP tools** (jira **40 → 42**): `add_offset_adjustment { assignee, amount, note? }` (appends;
  `400 VALIDATION` on a zero/non-integer amount) and `delete_offset_adjustment { assignee, id }` (removes by
  id, no-op if absent). Both return the updated ledger summary. The existing `set_offset_adjustment`
  (opening) is unchanged.
- **`get_offset_ledger` now returns `adjustments`** per summary (alongside `bySprint`) so the UI can plot
  earned + list adjustments without another call.
- **The History dialog is the one place for all of it.** `buildOffsetHistory` now also returns
  `earnedBySprint` (each sprint that banked earned, newest sprint first, from `bySprint`) and `adjustments`
  (the log). The dialog renders three sections — **Earned** (per sprint, +N), **Used** (each Offset leave,
  −1), **Manual adjustments** (each dated ± entry with its note, a delete button, and an **add form**:
  amount ± and an optional note) — plus a 5-tile standing (Earned / Used / Opening / Adjust / Balance). The
  offset table's **Opening** column is untouched (the one-time carry-in stays there).
- Wallet balance folds the log in: `OffsetWalletEntry` gains `adjustmentsTotal`; the wallet card's per-dev
  line shows `· adj ±N` when non-zero so the balance stays explainable.

## Consequences

- The History button is now a complete per-developer ledger: how the balance was earned, spent, and
  manually adjusted — with adjustments editable in place, no new table columns.
- Ad-hoc corrections are first-class and auditable (dated, noted, individually removable), distinct from the
  opening carry-in.
- Additive + back-compatible: older store files (no `adjustments`) read as an empty log; existing
  balances/opening are unchanged. No migration.
- Two more write tools to keep keyless-tested; they reject empty input (smoke) like the other offset writes.
- Gate: typecheck + build clean, mcp-jira 537 → 542 (offset store/tools) and react-app 767 → 774 (wallet +
  dialog), github 57 → **1,373 total**, smoke 53 → **55/55** (the two new write tools each add an
  empty-input validation check; jira tools 42). Pill → 1.54.0. **Not live-eyeballed** — the Offset Tracker
  is behind the app login gate; verified via the pure `summarizeOffset`/`buildOffsetHistory` tests + the
  jsdom dialog tests (add/delete/earned-list).

## Alternatives considered

- **A second single number beside Opening.** Rejected by the user — no per-entry log or audit trail; the
  "history" framing calls for dated entries.
- **A unified chronological timeline** mixing earned, spent, and adjustments. Banking has no per-event
  timestamp (only per sprint), so earned can't be honestly placed among dated spends; three clear sections
  avoid fabricating dates.
- **Reuse `set_offset_adjustment` for the log.** It sets one absolute value (the opening); a log needs
  append/remove semantics with ids — distinct tools are clearer and keep the opening intact.
