# ADR-071: Multi-sprint report + Trends & KPIs mode — one tool, one fetch, presentation on top (v1.59)

## Status: Accepted (2026-07-17) — implemented in v1.59

## Context

Two user requests, planned together (F2 + F3): a report across **multiple sprints** — selected
sprints or a date range, "usually the previous 10" — and a **KPI report** for the whole team or a
specific developer. The user locked the KPI scope to **velocity & trend only** (reliability,
throughput, and capacity-adjusted variants were offered and deliberately not selected).

What existed: `get_sprint_report` covers exactly one sprint; `get_velocity` loops sprints but
returns team-level points only (no per-assignee data); the two of them (and `list_sprints`)
carried a **duplicated closed-sprint sort** (getVelocity.ts ≡ listSprints.ts, ten lines each).
The Reports page had one mode and a tab bar already at capacity (ADR-060: Guide became a header
button because an 8th tab didn't fit).

## Decision

**One new read tool, `get_multi_sprint_report` (contract §4.29), and F3 is pure presentation over
F2's payload** — a single fetch feeds the per-sprint table, the team KPIs, and the per-developer
view. `byAssignee` per sprint is free CPU on issues already fetched (`computeByAssignee` reused
verbatim), so the per-developer feature costs zero extra Jira calls.

**Selection has two mutually exclusive paths** (base ZodObject + refine-in-handler — the
set_leaves/updateTicket precedent, keeping the schema JSON-Schema-generable for the AI allowlist):
- *Pool path* (default): the `get_velocity` template verbatim — closed sprints (+ active when
  `includeActive`), `sortClosedSprintsLatestFirst`, optional strict-before `beforeSprintId`
  anchor, `slice(0, sprintCount)` (default 10, max 26).
- *Explicit path*: `sprintIds` (1..26) → `getSprintMeta` each, chronological.

**Date range is a client concern.** `list_sprints` already returns every sprint's dates; the UI
(`lib/sprintRange.ts`) filters them and sends concrete `sprintIds`. The backend never grows a
date-parsing surface for something the client can resolve exactly.

**Averaging conventions follow `get_velocity`**: `averageCompleted` = completed totals over the
window; the cross-sprint `byAssignee` aggregate's `avgDonePoints` divides by the **FULL window**,
not the sprints the person was active in (`sprintsActive` is exposed alongside so the distinction
is visible, not hidden). Empty window → the empty-but-valid zero shape, never an error.

**Refactor before feature:** `sortClosedSprintsLatestFirst` was extracted into `sprintSelect.ts`
and swapped into both existing call sites *first*, with the v14/v15 test files passing unchanged
as the behavior-preservation proof — then the new tool consumed the shared function.

**Frontend: a mode inside Reports, not a new tab** (ADR-060 precedent). A page-local
`"sprint" | "trends"` segmented control; the existing sprint-report path is untouched (the diff is
purely additive), and all new UI lives in `components/trends/*` rather than growing the
1,581-line Reports.tsx: `TrendsView` (orchestrator; Last-N default 10 / pick-sprints / date-range),
`SprintRangePicker`, `MultiSprintTable`, one generic `MultiSprintBarChart` (the VelocityChart CSS
technique lifted, primary/secondary series; VelocityChart itself untouched), `TeamKpiSection`
(avg completed + avg completion rate + completed-vs-committed bars), `DeveloperKpiSection`
(native `<select>` — Radix Select is jsdom-hostile, ADR-009 — avg tile, "active in N of M
sprints", per-sprint bars). Deliberately **no** forecast/best-worst tiles and no
reliability/throughput/capacity KPIs — out of the locked scope; the payload supports them later
without a contract change. Export = markdown/CSV via `buildMultiSprintMarkdown/Csv` reusing the
existing csv helpers; the styled-xlsx/print treatments are deferred.

**Cross-cutting counts moved atomically**: tools/index 42→43, askService READ_TOOLS 18→19, smoke
expected-tools +1 (not in the empty-input-validation loop — all-optional schema), toolCatalog
47→48 with its anti-drift test counts (jira 43, jira-read 13, READ_TOOLS copy 19, Reports &
velocity group 3) updated in the same change, plus Guide/USER-GUIDE prose and table rows.

## Consequences

- "How did the last 10 sprints go, and how is Alice trending?" is one tool call and one screen.
- Cost profile matches `get_velocity`: N+1..2 sprint-pool calls + N issue fetches, all parallel;
  no changelog fetches (aging stays `get_active_sprint`-only, ADR-070).
- Per-assignee identity is **display name**, inherited from every existing aggregation
  (byAssignee, leaves, offset). A renamed developer splits across the window; two developers with
  identical display names merge. Accepted as a known, pre-existing limitation — the fix (accountId
  keying) would touch every surface at once and belongs to its own ADR if it ever matters.
- No new Jira rate-limit exposure class: the parallel fan-out mirrors get_velocity's existing
  pattern (max 26 sprints). Backoff remains a pre-existing, not-newly-introduced gap.
- The mutual-exclusion rule lives in the handler, so `{}` stays valid (`{}` = last 10 closed) and
  the AI assistant can call the tool with no arguments.
- Tests: mcp-jira 554 → 577 (multiSprintReport.test.ts, 23 — pool/explicit/refine/parity/
  aggregate units), react-app 822 → 887 (sprintRange 16, trends components 35, reportMarkdown +11,
  Reports mode-toggle +3, toolCatalog count updates). Total 1,433 → **1,521**; smoke 55/55.
  Pill → 1.59.0.
- One pre-existing Reports test was re-scoped (not re-asserted): its blind
  `querySelector("button[aria-pressed]")` grabbed the new header mode toggle instead of the
  leaves-table toggle it meant; it now scopes to the leaves table like its sibling tests.
- **Not live-eyeballed**: all surfaces sit behind the login gate; proven by unit/jsdom tests
  against contract-shaped fixtures.

## Alternatives considered

- **N × `get_sprint_report` from the frontend.** No new tool, but N sequential round-trips per
  render, no shared window math, and the AI assistant couldn't answer trend questions in one call.
- **Extend `get_velocity`** with byAssignee + counts. Would silently fatten a payload consumed by
  existing velocity charts and the AI; a separate tool keeps both contracts stable.
- **Backend date-range parameters.** The client already holds every sprint's dates from
  `list_sprints`; a server-side range would duplicate that selection logic and invite timezone
  parsing questions for zero savings.
- **A "Trends" tab.** The tab bar is the reason Guide is a header button (ADR-060); a mode toggle
  inside Reports costs no chrome.
- **Recharts or another chart dependency** for the KPI bars. The repo's CSS-bar technique
  (VelocityChart) already covers grouped bars; no new dependency for two bar charts.
