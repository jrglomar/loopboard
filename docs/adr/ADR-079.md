# ADR-079: PO draft capacity plan — dev roster, capacity, and drag-to-draft on the PO Planning page (v1.68)

## Status: Accepted (2026-07-21) — implemented in v1.68

## Context

The team's planning flow is: the PO grooms tickets on the PO board's future sprint FIRST; the
developers then plot leaves/offsets (Leaves plotter, ADR-016/038) which yields each developer's
capacity for the sprint (ADR-047: `capacity = max(0, requiredPoints − working leave days)`, in
points). Only later, on the DEV board (Planning / Linking), do dev tickets get really assigned.

Two gaps blocked the PO from sanity-checking load against capacity during grooming:

1. **The PO board's "Manage team" never shows the developers.** Not a bug: team rosters are
   deliberately curated PER BOARD (ADR-019 — `{ boardId: TeamMember[] }`), and the PO board's
   dialog edits the PO roster with candidates drawn from PO-board activity. The developers live on
   the Dev board's roster, which the PO Planning page simply never displayed.
2. **No way to tentatively lay PO tickets against developers.** Real assignment (`assign_issue`)
   is a Jira write and belongs to the Dev-board flow; the PO needs a DRAFT — visible, persistent,
   shareable with the team, but with zero effect on Jira — to see "if these stories land on these
   people, who is over capacity?"

A wrinkle: capacity is derived from leaves, and leaves are keyed by the **Dev** sprint (separate
board → separate sprint ids), so the PO sprint being planned must be paired with a Dev sprint to
read capacity from.

## Decision

**New per-PO-sprint store + tools (`get_draft_plan` / `set_draft_plan`, §4.30).** Same bridge-side
store pattern as impediments/post-scrum (ADR-027/031), through the ADR-077 storage port (json |
sqlite, name `draft-plan`, env `JIRA_DRAFT_PLAN_FILE`, registered in `storage/registry.ts` for the
override + sqlite auto-import). Doc shape per PO sprint: `{ devSprintId: number | null,
assignments: Record<issueKey, { accountId, displayName }> }` — full-replace semantics like every
sibling store; empty assignments deletes the entry. Persisting server-side (not localStorage) makes
the draft durable across devices and visible to whoever opens the Planning page — it is a team
artifact, like leaves. The tools NEVER write to Jira; `assign_issue` remains the only assignment
path. `displayName` is snapshotted alongside `accountId` so a member later removed from the roster
still renders meaningfully (leaves/capacity are display-name-keyed, ADR-016).

**Draft Capacity Plan card on Planning, PO board only.** Renders when `boardKey === "po"` and a
sprint is selected, between the leaves plotter and the assignment list:

- **Dev roster, in place.** The card shows the DEV board's curated team (`useTeamMembers(devBoardId)`,
  devBoardId = `boards.dev[projectIdx].id`) as tiles, and embeds the existing `TeamManager` bound
  to the Dev board id ("Manage dev team") — reusing ADR-019's one-roster-per-board model instead of
  merging rosters or widening the PO board's candidate search.
- **Dev sprint pairing.** A pure `pairDevSprint(poSprint, devSprints)` picks the default capacity
  source: max date-overlap with the PO sprint, ties prefer future then earliest start; fallback to
  exact name match, then first future, then first active. A native `<select>` (ADR-009) lets the
  PO override; the choice persists in the draft (`devSprintId`) so the view restores.
- **Capacity reuse, verbatim.** Per-tile capacity = ADR-047's `computeDevCapacity(policy.requiredPoints,
  computeCapacity(...).byAssigneeLeaveDays)` over the paired Dev sprint's working days and leaves
  (`useLeaves(devSprintId)`), joined to tiles by displayName. No dev sprint dates → capacity shown
  as unknown, drafting still works.
- **Drag to draft + accessible fallback.** PO sprint tickets (same flattened buckets as the
  assignment list) render as chips: native HTML5 drag onto a developer tile drafts the ticket;
  every chip ALSO carries a native "Draft to…" `<select>` (keyboard/screen-reader path, ADR-009),
  and drafted chips have a remove (×). Tiles show drafted-points totals with an over/under delta
  chip — advisory colour only, NEVER blocking (explicit user requirement: over/under allowed).
  Saves are optimistic full-replace with rollback (the `useLeaves` save pattern).
- **Honest edges.** Draft entries whose ticket left the sprint, or whose member left the roster,
  are surfaced as removable "stale" rows instead of being silently dropped — the store is only
  mutated by an explicit user action.

## Consequences

- The PO can pull the real developer roster + live capacity into PO-board grooming and lay out a
  tentative plan against it, without touching Jira assignees or duplicating the roster model.
- The "why don't devs show in Manage team" confusion is answered structurally: the PO board keeps
  its own roster; the card surfaces the DEV roster read-through, with dev-roster editing one click
  away via the same TeamManager.
- Two more MCP tools (43 → 45) ride the existing registry (stdio + bridge for free); the AI Q&A
  assistant can read the draft plan (allowlist 19 → 20).
- The draft is advisory by design — nothing enforces capacity, and nothing syncs it to later real
  assignments; it's a planning overlay the Dev-board flow is free to diverge from.
- Tests: mcp-jira 626 → **644** (+18: 17 draft-plan store/tool cases — empty shape, round-trip,
  full-replace, both delete-on-empty paths, key/size/field validation, corrupt-file tolerance,
  registry presence — plus 1 AI read-allowlist growth); react-app 958 → **1031** (+73: 13 sprint
  pairing, 15 draft rollup math, 7 client, 10 useDraftPlan hook, 25 DraftPlanCard, 3 Planning
  PO-only wiring); mcp-github unchanged at 57. Total **1,732**; smoke 55 → **60** (both tools in
  the empty-input VALIDATION loop + 3 explicit draft-plan behavior checks).

## Alternatives considered

- **localStorage-only draft** — rejected: not shared between PO and team, dies with the browser
  profile; the repo already has a first-class store pattern (ADR-027/077) that makes server
  persistence the cheaper option, not the dearer one.
- **Real Jira assignment from the PO board** (just reuse `assign_issue`) — rejected: the user
  explicitly wants a non-executing draft; real assignment stays on the Dev board (Linking/Planning),
  and a draft that silently wrote assignees would be worse than no draft.
- **A drag-and-drop library (dnd-kit / react-dnd)** — rejected: native HTML5 DnD covers the mouse
  path with zero new dependencies, and the a11y path is the per-chip native `<select>` anyway
  (ADR-009's native-controls stance) — a library would add bytes to solve a problem the fallback
  already solves.
- **Merging dev members into the PO board's roster** (or auto-seeding the PO team with devs) —
  rejected: ADR-019's per-board rosters are load-bearing (leaves plotter + assignment lists key off
  them); polluting the PO roster would put developers into PO-board assignment dropdowns. Reading
  the Dev roster where needed keeps one source of truth per board.
- **Storing the draft keyed by accountId list per member** (member → keys) — rejected: keying by
  issueKey (`issueKey → member`) makes "a ticket is drafted to at most one person" structural
  instead of an invariant to police.
