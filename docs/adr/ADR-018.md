# ADR-018: Sprint Preparation (Planning/Grooming) page + ticket-to-developer assignment

## Status: Accepted (2026-06-15)

## Context

The sprint-prep actions had grown scattered: the **New Sprint** button lived on the
Dashboard, **ticket generation** was its own tab, and the **leaves/capacity plotter** lived
on Reports. Grooming/planning a sprint meant hopping between three places, and there was no
way to **assign tickets to developers** from the app at all. The product owner asked for a
single **Sprint Preparation (Planning/Grooming)** workspace that consolidates these and adds
assignment.

## Decision

1. **A new Planning page** becomes the prep hub, scoped to a **board + the sprint being
   planned**, defaulting to the **next future sprint** on the selected board (the one about
   to start), falling back to the active sprint. Tab nav becomes **Dashboard · Planning ·
   Reports**.
2. **Move (not duplicate)** three features into Planning:
   - **New Sprint** (`CreateSprintDialog`) moves off the Dashboard.
   - **Ticket generation** — the "Ticket Generator" tab is removed and the existing
     `TicketGen` component is **reused** inside Planning, with target sprints pre-seeded from
     the planning context (still overridable). No logic is rewritten.
   - **Editable leaves/capacity plotter** moves off Reports; **Reports keeps a read-only**
     Leaves column + capacity (derived from the same `get_leaves`). Editing happens only in
     Planning, viewing in Reports — a clean prepare-vs-review split.
3. **Add ticket-to-developer assignment.** Two new MCP tools: `get_assignable_users`
   (`GET /user/assignable/search`, active users, by **accountId** — Jira Cloud requires
   accountId to assign) and `assign_issue` (`PUT issue/{key}/assignee`, a real write,
   `null` unassigns). Planning lists the planned sprint's tickets with an assignee `<select>`
   that calls `assign_issue`.
4. **Roster from the team, not the sprint.** The leaves plotter and the assignment selects
   are populated from `get_assignable_users` (the project's assignable developers), not just
   the people already on the sprint — essential when planning a near-empty future sprint.

## Consequences

- Planning/grooming is one screen: pick/create the sprint, draft tickets into it, plot the
  team's leaves (capacity), and staff the tickets — matching how the ceremony actually runs.
- Assignment introduces a second real Jira **write** path (after `create_sprint` and
  add-to-sprint). It's optimistic with inline error handling; tests mock the client so the
  suite stays keyless/offline.
- Removing the Ticket Generator tab is a deliberate consolidation; the component is reused,
  so there's no behavioral regression, only a relocation. Anyone who wants standalone ticket
  drafting now does it in Planning.
- The Dashboard slims to a pure live-board view (no New Sprint button); Reports slims to a
  read-only leaves view.

## Alternatives considered

- **Add Planning but keep the originals** (the other offered option) — more access points
  but duplicated UI and two sources of truth for the same action; the owner chose "move."
- **Assign by display name / email** — Jira Cloud assigns by `accountId`; listing assignable
  users first is the correct, instance-portable approach.
- **A bespoke team-roster store** — unnecessary; `get_assignable_users` is the authoritative
  list straight from Jira and needs no persistence.
- **Per-feature sub-tabs inside Planning** — kept it a single scrollable workspace (context
  picker → new sprint → ticket gen → leaves → assignment) so the whole prep flow is visible;
  revisit if it grows.
