# ADR-041: Forward, multi-sprint leave planner

## Status: Accepted (2026-06-30) — **revised same day (see Update)**

> **Update (v1.31, per user feedback):** the planner now shows **only the selected sprint** (not the
> multi-sprint window); you switch to any sprint — including a **future** one — via the Leaves-page
> sprint `<select>`, which is **grouped by state (Active / Future / Closed)** so future sprints are easy
> to find. `LeavesPlannerCard`, `get_all_leaves`, `useAllLeaves`, weekend-exclusion, and per-day
> sprint-attribution are unchanged. `selectCalendarSprints` remains in `leavePlanner.ts` (and tested) but
> the Leaves page now passes a one-sprint window `[selectedSprint]`. The rest of this ADR documents the
> original window design for history.

## Context

The Leaves page (ADR-038) plots typed leaves (VL/EL/Holiday/Offset) on **one selected sprint's**
working days. To record a leave that falls in a future sprint, you must first select that sprint — there's
no way to plan leaves across the upcoming weeks in one view. The user wants to **plot leaves/offset on a
forward calendar (weekends blocked)** and have each plotted day **matched to the sprint that contains it**
(current and future, plus recent past for backfill).

Constraints discovered:
- `sprintWorkingDays(start,end)` already returns **Mon–Fri only**, so weekend-blocking is free.
- Leaves are stored **per sprint id** (`{ sprintId: { assignee: { date: type } } }`); `get_leaves` /
  `set_leaves` each take one sprint. The offset engine is per-sprint (needs that sprint's done points).
- `list_sprints` (`useSprintList("all", boardId)`) already returns `active`/`future`/`closed` with date
  ranges. The Dev board has ~30 never-closed "active" sprints, so "all active" is not a usable window.

## Decision

Keep the per-sprint storage + offset model; make only the **calendar** span multiple sprints.

- **New read tool `get_all_leaves {}`** → the whole store in one read (legacy dates → `VL`). Avoids one
  `get_leaves` per sprint. Writes stay per-sprint via `set_leaves`.
- **Sprint window (pure `selectCalendarSprints`)**: filter to dated sprints, sort by `startDate`, find the
  **anchor** (the sprint containing today, else the latest one already started, else the first upcoming),
  and take `[anchor − before … anchor + after]` (default 2 before / 4 after). This naturally yields
  "a few recent closed + active + next few" and is robust to the 30-active quirk (it anchors on the
  latest-started, not on the state label).
- **Calendar model (pure `buildLeaveCalendar`)**: concatenate each window sprint's `sprintWorkingDays`
  into a flat Mon–Fri day list, each day tagged with its `sprintId`; dedupe by date (first sprint wins);
  emit per-sprint segments for the grouped header. A `dateToSprintId` map drives attribution.
- **UI (`LeavesPlannerCard`)**: a person-row × day-column matrix grouped by sprint (labelled dividers),
  weekends absent. Clicking a cell paints the selected leave type and saves via `set_leaves` **for that
  day's sprint** (full-replace of that assignee's dates within that one sprint). One `useAllLeaves` hook
  loads the store and routes saves.
- The **offset table stays scoped to the selected sprint** (sprint `<select>`), since earned/spent need
  that sprint's done points; selecting a sprint also highlights it in the planner.

## Consequences

- Leaves can be planned weeks ahead in one scrollable, weekend-free calendar; every plotted day is
  attributed to the right sprint automatically, so the existing per-sprint offset accounting is unchanged.
- One extra read tool (`get_all_leaves`); no change to `set_leaves` or the offset tools.
- The existing single-sprint `LeavesCalendarCard` is untouched (still used read-only/editable on
  Reports + Planning); only the Leaves page swaps to the multi-sprint planner.
- Overlapping sprint date ranges (a data quirk on the Dev board) are handled by first-wins date dedupe;
  the bounded window keeps the blast radius small.

## Alternatives considered

- **N× `get_leaves` (one per window sprint).** Works, but chatty and grows with the window; a single
  `get_all_leaves` is cleaner and symmetric with `get_offset_ledger`.
- **A literal month-grid calendar with disabled weekend cells.** Doesn't scale to a team (one grid per
  person) and obscures sprint boundaries; the team matrix grouped by sprint fits this tool better.
- **Render every future sprint.** The PO board has ~54 future sprints — unusable width. The anchored
  window bounds it while still reaching forward.
