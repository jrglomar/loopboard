# ADR-039: "Has linked PR" badge on the Huddle board + Reports

## Status: Accepted (2026-06-30)

## Context

The team links GitHub PRs to Jira issues automatically (an uppercase issue key in the branch/commit/PR
title populates the issue's Development panel). v1.22 (ADR-034) surfaced those linked PRs in the Huddle
**code-review card** via `get_issue_pull_requests`. But scanning the board, you cannot tell **which
specific ticket** already has a PR / code review in flight without cross-referencing that card. The user
wants that signal **on the card itself** (Huddle board) and on the **Reports** issue rows — "ensure
everything has better visibility."

## Decision

Frontend-only; **no tool/IO change** — reuse `get_issue_pull_requests` (§4.25) and the existing
`useIssuePullRequests(keys)` hook.

- **Pure summary (`src/lib/prBadge.ts`).** `summarizePrBadge(prs: LinkedPr[] | undefined)` → `null` when
  none, else `{ count, newest, tone, openCount }`. `newest` = the PR with the latest `lastUpdate`
  (stable fallback = first). `tone` is an aggregate over the **still-open** PRs: `changes` if any
  changes-requested, else `done` if none are open (all merged/closed), else `approved` if any approved,
  else `review`.
- **Shared component (`src/components/PrBadge.tsx`).** A compact, clickable badge (`GitPullRequest` +
  count) that **links to `newest.url`** (new tab) and whose `title`/`aria-label` lists every linked PR
  when >1. The tint maps from `tone` (red / amber / green / muted) so a glance conveys review state.
  Renders nothing when there are no linked PRs.
- **Huddle board.** `Dashboard` lifts `useIssuePullRequests(sprintKeys)` **once** and passes the
  resulting `Record<key, LinkedPr[]>` to `SprintBoard` (new optional `prsByKey` prop), threaded down to
  each `IssueCard`, which renders `<PrBadge prs={prsByKey[issue.key]} />`. The same lifted map is passed
  to the code-review card (`PullRequestsCard`, new optional `issuePrs` prop) so the page makes **one**
  dev-status fetch instead of two.
- **Reports.** `SprintReportView` lifts `useIssuePullRequests` over the report's Completed + Carryover
  issue keys and badges each row.

## Consequences

- A ticket with an in-flight PR is now obvious at a glance on the board and in reports, with review
  state encoded in the badge colour and one-click access to the newest PR.
- No new backend surface; the only new code is one pure helper + one presentational component + prop
  threading. `prsByKey`/`issuePrs` are optional, so `SprintBoard`/`PullRequestsCard` standalone tests are
  unaffected.
- Inherits the v1.22 caveat: linked-PR data comes from Jira's **undocumented** `/rest/dev-status`
  Development-panel endpoint; the badge shows whatever that returns (and nothing when it returns none).

## Alternatives considered

- **Badge from the single-repo GitHub `list_prs`.** Rejected — misses other repos; the dev-status
  source (v1.22) is already multi-repo and per-issue.
- **Let each `IssueCard` call the hook itself.** Rejected — N fetches per board; lifting once is cheaper
  and keeps a single source of truth for the page.
- **A non-clickable dot/indicator.** Rejected — the user values quick access; linking to the newest PR
  is more useful and costs nothing extra.
