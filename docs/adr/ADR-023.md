# ADR-023: Per-draft "comment & regenerate" (reuse existing AI endpoints)

## Status: Accepted (2026-06-17)

## Context

When drafting tickets — both in **TicketGen** (the PO+Dev pair) and on the **Linking** page
(one Dev draft per PO) — the user wants to leave a **comment on a specific draft** and click a
button to **regenerate that draft** incorporating the comment, without retyping the whole thing.

TicketGen already supports refinement through its AI chat (a follow-up message re-drafts the
pair), but that isn't discoverable as a per-draft control, and the Linking plan has no
refinement at all.

## Decision

Add one shared presentational control, `components/RefineDraftControl` — a comment textarea +
"Regenerate" button — shown **only when AI is enabled** (a comment can't meaningfully steer a
deterministic template). Each caller wires it to the AI endpoint it already uses; **no new
backend surface**:

- **Linking** (per plan item): on regenerate, re-call `POST /api/ai/plan-dev-tickets` for **that
  single PO**, passing the reviewer comment **and the current draft** inside the existing
  `instructions` field, and replace that item's `devSummary`/`devDescription`. A per-item busy
  state spins just that row.
- **TicketGen** (the draft pair): on regenerate, append the comment as a user turn to the
  existing conversation and re-call `POST /api/ai/draft-tickets` (exactly what a chat follow-up
  does), refreshing the PO+Dev draft.

`draft-tickets` already takes `messages[]` and `plan-dev-tickets` already takes `instructions`,
so both refinement flows fit the current endpoints with zero IO change.

## Consequences

- One reusable control, consistent UX in both places; the AI does the refinement using the
  endpoint each surface already speaks to.
- Linking refinement stuffs "comment + current draft" into `instructions` (free text) rather
  than a structured `currentDraft` field — acceptable for a POC; a structured field can be
  added later if refinement quality needs it.
- Regenerate is AI-only by design; with AI off the user edits the draft fields directly (as
  today).
- No contract/tool/endpoint changes — purely additive frontend.

## Alternatives considered

- **A structured `currentDraft` on `plan-dev-tickets`.** Cleaner prompt, but a backend + schema +
  test change for marginal POC benefit; the `instructions` blob already carries the context.
- **A new `refine-draft` endpoint.** Unnecessary — `draft-tickets` (conversation) and
  `plan-dev-tickets` (instructions) already refine.
- **Rely on TicketGen's chat alone.** Doesn't give Linking any refinement and isn't a
  per-draft affordance, which is what was asked for.
