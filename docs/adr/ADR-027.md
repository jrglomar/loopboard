# ADR-027: Huddle daily visibility — impediments log + pending-PR list (manual stores)

## Status: Accepted (2026-06-24)

## Context

The Scrum-Master review flagged two missing daily-standup affordances: a place to record
**impediments/blockers** for visibility, and a place to track **pending PRs** under code review. The
Dashboard is the team's daily surface, so it is renamed **Huddle** and gains both.

Neither maps cleanly to a Jira object: impediments are free-form notes, and PR links live in GitHub.
The team wants a lightweight, always-available scratchpad keyed to the current sprint — not a new Jira
field or a live GitHub integration (the user chose a **manual bridge store** over GitHub-derived PRs).

The codebase already has a proven pattern for exactly this: the **leaves** and **team** stores —
per-key JSON files on the Jira bridge, read/write helpers, config-driven path, tolerant of a
missing/corrupt file.

## Decision

### Two manual per-sprint JSON stores (mirror leaves/team)

- `impedimentsStore.ts` → `{ [sprintId]: Impediment[] }`,
  `Impediment = { id, text, ticketKey?, createdAt, resolved? }`.
- `prsStore.ts` → `{ [sprintId]: PullRequest[] }`,
  `PullRequest = { id, url, title?, ticketKey?, status?, addedAt }`.
- Tools: `get_impediments`/`set_impediments`, `get_pull_requests`/`set_pull_requests`. `set_*` are
  full-replace (like `set_team_members`) and fill `id`/timestamps when the client omits them, so the
  UI can add an item with just text/url.
- Paths from `JIRA_IMPEDIMENTS_FILE` / `JIRA_PRS_FILE` (defaults in the package dir, git-ignored).

### Huddle page

- Rename the **Dashboard** tab + heading to **Huddle**.
- The board's filter line shows the **filtered points total** alongside the issue count.
- Two new sidebar cards — an Impediments log and a Code-review PR list — keyed to the Huddle's
  selected sprint, backed by the stores via `useImpediments` / `usePullRequests` hooks (mirror
  `useLeaves`).

## Consequences

- Daily blockers and review queue are visible at standup without leaving the app, and Copilot can
  read/write them too.
- The stores are **ephemeral, bridge-local** (like leaves/team) — not synced to Jira/GitHub. That is
  acceptable for a standup scratchpad and keeps the surface tiny.
- jira tools 23 → 27.

## Alternatives considered

- **GitHub-derived PRs** (live, via mcp-github). More accurate but heavier (a list/search endpoint,
  auth, repo config); the user chose the manual store for simplicity. Can be added later behind the
  same card.
- **Store impediments as Jira issues/labels.** Heavier and pollutes the backlog; a free-form note is
  what standup needs.
- **A new combined "huddle store".** Two focused stores match the existing one-store-per-concern
  pattern and keep the tool IO obvious.
