# ADR-061: Offset banking is a confirmed action + a clear opening balance (v1.50)

## Status: Accepted (2026-07-11) — implemented in v1.50

## Context

The Offset Tracker (ADR-038/044) **auto-banked** each sprint's earned offset the moment you opened that
sprint — a signature-guarded effect fired `set_offset_for_sprint` on view. Two problems the user hit:

1. **No control / no confirmation.** Balances changed on their own just by browsing a sprint; there was no
   deliberate "record this" step, which is uncomfortable for numbers people reconcile.
2. **No obvious way to seed a prior balance.** They had a pre-existing offset balance (from before
   Loopboard) and needed the wallet total to match. The store already had a per-developer `manualAdjust`
   (a signed number folded into the balance) — but it was an unlabelled `Manual` input buried in the offset
   table, so it wasn't recognised as "set your starting balance here".

## Decision

- **Bank on confirm, not on view.** Removed the auto-bank effect. The Offset Tracker now has a **"Bank
  earned offsets"** button that opens a confirmation dialog (`BankOffsetsDialog`) listing each developer's
  computed earned for the sprint and what's already banked; `set_offset_for_sprint` runs only on confirm.
  The button reads "Banked ✓" (and confirm is disabled) once everything for the sprint already matches, so
  re-banking is safe and obvious. Banking stays idempotent per sprint.
- **Surface the manual field as the "opening balance".** Per the user's choice, we **reuse the existing
  `manualAdjust`** rather than add a redundant second additive field — no data migration, existing values
  kept. It's relabelled **"Opening"** in the table and **"opening"** in the wallet, with copy explaining
  it's each developer's prior/carry-in balance to make the total match. Math is unchanged:
  `balance = Σ banked earned − Σ Offset leaves + opening`.
- **Expose per-sprint banked state.** `get_offset_ledger` (and `summarizeOffset`) now include
  `bySprint: { [sprintId]: { earned, spent } }` in each summary, so the UI can tell whether a sprint is
  already banked and show the banked→new transition in the dialog. Additive, backward-compatible.

## Consequences

- Balances only move when the user deliberately banks a sprint — reconciliation is in their hands.
- Seeding a prior balance is now a recognised, labelled action (the Opening column), fixing the "my total
  doesn't match" gap without a new concept.
- Because banking is manual, a sprint viewed but never banked contributes **0 earned** to the wallet until
  banked — intended, and the button's "Banked ✓ / Bank earned offsets" state makes the status legible.
- `manualAdjust` keeps its internal name (store + `set_offset_adjustment` tool) while the UI says "opening
  balance". A minor name/label divergence, accepted to avoid migrating live offset data.
- Gate: 1,354 tests (mcp-jira 532, react-app 759→765, github 57), typecheck + build clean, smoke 53/53.
  Pill → 1.50.0.

## Alternatives considered

- **Keep auto-bank, add an "undo"** — still surprises the user first; a confirm-first flow is cleaner.
- **A separate `openingBalance` field alongside `manualAdjust`** — two knobs that both just add to the
  balance; the user chose to reuse the one field (no migration, no redundancy).
- **Per-developer bank buttons** — `set_offset_for_sprint` already takes the whole sprint's entries;
  banking the sprint in one confirmed action matches the tool and the mental model ("close out this sprint").
