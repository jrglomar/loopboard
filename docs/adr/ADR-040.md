# ADR-040: Dual fly-in tracking (PO + Dev) with PO→Dev alignment

## Status: Accepted (2026-06-30)

## Context

The Huddle "Fly-in" tracker (v1.23/v1.24) lists the **selected** board's current-sprint tickets whose
title reads like a fly-in. But fly-ins are coordinated across the PO and Dev boards: a PO fly-in should
have a **corresponding Dev fly-in**. With a single-board view you cannot see both sides at once, nor tell
whether a PO fly-in is actually mirrored on the Dev board. The user wants **separate PO + Dev tracking**
and a check that the **PO fly-in is aligned with the Dev board**.

## Decision

Frontend-only; **no new tool** — reuse `get_active_sprint` (both boards) and `get_linked_issues`.

- **Both boards' fly-ins.** `Dashboard` already fetches the **selected** board's active sprint; it now
  also fetches the **opposite** board's active sprint (`useActiveSprint(otherBoardId, null)`, default
  project on that side). `selectFlyIns` runs over each board's issues → `devFlyIns` + `poFlyIns`. One
  extra read per Dashboard view; a no-op when the other board has no sprint.
- **Alignment via links.** For the PO fly-ins, `useLinkedIssues(poFlyInKeys)` (a new hook over the
  existing `get_linked_issues`, default `projectKey` = Dev) returns each PO key's linked Dev issues. A PO
  fly-in is **aligned** when one of its linked Dev issues is **itself a fly-in** (`matchFlyIn` on the
  linked issue's summary). `Dashboard` builds `poAlignment: Record<poKey, LinkedIssue | null>` (the
  matching Dev fly-in, or null).
- **UI.** `FlyInCard` now takes `{ devFlyIns, poFlyIns, poAlignment? }` and renders **two labelled
  groups** (Dev board / PO board). Each PO fly-in shows an alignment badge: green **"✓ aligned"** linking
  to the Dev fly-in, or amber **"⚠ No Dev fly-in"**.

## Consequences

- The team sees both boards' fly-ins in one place and immediately spots a PO fly-in that has no
  corresponding Dev fly-in — the exact gap the user wanted surfaced.
- One extra `get_active_sprint` read (the opposite board) + one `get_linked_issues` call scoped to the PO
  fly-in keys only (usually a handful). Both are read-only and resilient (failures → empty → the card
  simply shows fewer/zero badges).
- `FlyInCard`'s props changed (`flyIns` → `devFlyIns`/`poFlyIns`/`poAlignment`); its only caller is the
  Dashboard, updated in the same change.
- Alignment is a **title-match heuristic** on the linked Dev issue — it does not require the Dev fly-in to
  be in the current Dev sprint. This is intentional (a Dev fly-in may live in a different sprint) and
  matches how `matchFlyIn` already classifies fly-ins elsewhere.

## Alternatives considered

- **Align by checking the linked Dev issue is in `devFlyIns`** (current Dev sprint). Rejected — too
  strict; a legitimately-linked Dev fly-in in another sprint would be flagged as missing.
- **A dedicated `get_fly_in_alignment` backend tool.** Rejected — `get_linked_issues` + the pure
  `matchFlyIn` already provide everything; no new surface needed (contract-first: don't invent surface).
- **Keep one board and add a PO/Dev filter.** Rejected — the point is to see both at once and compare.
