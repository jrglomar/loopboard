# ADR-052: P2 batch — burndown, persisted retro, attention nudges (+ dependency-link direction fix)

## Status: Accepted (2026-07-07) — implemented; retro/burndown/attention gated offline, link direction verified live

## Context

Following the P1 batch (ADR-050), the "P2" roadmap items were: a **burndown chart**, a
**persisted retrospective** that stops the team retyping the retro at export time, and
**attention nudges** on the Huddle. While working the linking flow the team also reported that
PO↔Dev links read the wrong way round — so the dependency-link **direction** is fixed here too.

## Decision

### Burndown (Reports)
- `IssueSummary` gains optional `resolvedAt?` / `updatedAt?` (Jira `resolutiondate` / `updated`).
  The sprint-issue endpoint already returned these — only the mapper + types changed (no new
  request, no new tool).
- Pure `lib/burndown.ts` `computeBurndown(committed, completedIssues, workingDays, today)`:
  per working day `remaining = committed − Σ points resolved on/before that day`; weekend/
  pre-sprint resolutions roll forward; days after `today` are `null` (active sprints show a
  partial line); plus the straight ideal line. Rendered as a dependency-free SVG (`BurndownCard`).
- Documented limits: baseline = CURRENT committed (scope creep is not re-based); code-review-
  complete items burn only when Jira marks them resolved (DoD nuance).

### Persisted retro (Reports)
- The established per-sprint JSON-store pattern: `retroStore.ts`, `JIRA_RETRO_FILE`
  (default `.loopboard-retro.json`). Fields `{ reasonForDelays, whatWorkedWell, whatDidNotWork,
  plannedImprovements, kudos }` (each ≤ 4000, trimmed); `set_retro` stamps `updatedAt`; all-empty
  clears. Tools `get_retro` / `set_retro` (§4.28). **jira tools 38 → 40; smoke 50 → 52.**
- `RetroCard` on the Reports page writes it once; the **Full-report export dialog pre-fills** the
  five retro fields from the store and **persists typed values back** on export — so the retro is
  authored once, not re-typed. `get_retro` joins the assistant read-allowlist (**18** read tools).

### Attention nudges (Huddle)
- Pure `lib/attention.ts` `buildAttention({ issues, prsByKey, today, staleDays=3 })` →
  prioritized items: **stale in-progress** (in-progress + no `updatedAt` in ≥ staleDays days),
  **unassigned** (unfinished + no assignee; not double-flagged when already stale), and
  **PRs awaiting review** (`status==="open" && decision==="review_required"`). `today` is injected
  (pure/testable). `AttentionCard` sits at the top of the Huddle sidebar, reusing the already-lifted
  `sprintIssues` + `issuePrs` (no extra fetch). Empty state = "All clear".

### Dependency-link direction fix (Linking)
- `create_dev_ticket` swapped the `issueLink` payload to **`inwardIssue = PO, outwardIssue = Dev`**
  so the story reads **"PO depends on Dev"**. Root cause: in the team's Jira the **inward** issue
  displays the link type's **outward** description ("depends on") — the opposite of the assumption
  baked into v1.36/v1.40. `JIRA_LINK_TYPE` default `"Depends"` → **`"Depends on"`** (the exact link-
  type name in the instance, so a missing `.env` can't produce an invalid type). Verified live
  against real Jira: the PO now reads "depends on VRDB-x"; a unit test locks the inward=PO payload.

## Consequences

- Two new mcp-jira tools (40 total); one new store file; smoke updated to 52 checks. No new deps
  (burndown SVG is hand-rolled; retro reuses the store/hook/card pattern).
- The version pill (react-app `package.json`) was realigned to **1.42.0** — it had drifted at
  1.13.0 since ADR-024 while the contract/feature version advanced.
- The link fix changes the **direction of newly created** links only; pre-existing backwards links
  (incl. the junk test tickets) are unaffected and can be relinked/deleted by the team.
- Burndown/retro/attention are read-only or local-JSON, so tests stay keyless/offline. Retro and
  attention are pure-function-first; the burndown y-axis now renders numeric labels, so one Reports
  test assertion moved from `getByText` to `getAllByText` (numbers can appear more than once).

## Alternatives considered

- **Retro as a new external system / Jira field** — rejected; the JSON-store pattern is already
  proven for meeting notes/goal/impediments and keeps the retro portable + greppable.
- **Business-day staleness math for attention** — rejected for now; calendar-day age is simpler,
  deterministic, and good enough for a daily nudge. Can be revisited if noise warrants.
- **Re-basing the burndown on scope changes** — deferred; the honest "current committed" baseline
  with documented caveats is clearer than a moving target for a POC.
