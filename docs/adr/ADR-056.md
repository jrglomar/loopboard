# ADR-056: Shared-credential ("point-of-view") users + admin user CRUD

## Status: Accepted (2026-07-10) — implemented in v1.46

## Context

Loopboard became multi-tenant in v1.45 (ADR-055): every feature runs on the signed-in user's own
Jira/GitHub/AI tokens. That raised the floor for onboarding — a teammate who just needs to *see* the
board (a stakeholder, a new hire, a PO watching Dev) must first obtain three tokens. The team wanted a
user that can be created by an admin, holds **no tokens at all**, and still sees the boards — sharing
the admin's (or a nominated user's) connections and config. The admin also needed real **CRUD** over
users, not just role/config edits.

The hard part is not plumbing, it's **attribution**: a Jira mutation made with a borrowed token is
recorded in Jira as the *token owner's* action. Silently letting a borrower create and transition
issues would corrupt the team's audit trail.

## Decision

- **One-hop delegation.** `StoredUser.credentialSourceUserId` names a user to borrow from.
  `getEffectiveConnection(userId, provider)` returns the user's OWN connection if present, else the
  source's. A source must own its credentials (enforced on write), so delegation never chains and
  resolution cannot cycle. A borrower who later connects their own token silently takes over.
- **Shared point-of-view, not just shared secrets.** A borrower's per-user JSON stores resolve to the
  **source's** directory (`RequestContext.storeUserId`), and the config merge inserts the source's
  overrides beneath the user's own: `.env ← admin global ← source overrides ← own overrides ← creds`.
  Without this, a borrower would see the right Jira but an empty retro / meeting notes / offset ledger.
- **Read-only against Jira by default.** `RequestContext.canWriteJira` is false when the Jira connection
  in play is borrowed and the admin has not set `allowWrites`. `POST /api/tools/:name` rejects the eight
  **Jira-mutating** tools with `403 READ_ONLY_USER`. The remaining `set_*` tools write **local team JSON**
  (leaves, retro, notes, impediments, offset), which is collaborative app state rather than Jira history,
  so borrowers keep those. Admins can opt a borrower into writes per user, accepting the attribution.
- **Admin user CRUD** (`POST/PUT/DELETE /api/admin/users[/:id]`) with guards: no self-disable/self-delete,
  ADMIN_EMAILS accounts are undeletable/undisablable, and a credential source in use by others cannot be
  deleted or converted into a borrower (`409 IN_USE`). **Disable** blocks login and existing sessions
  (`403 ACCOUNT_DISABLED`) while preserving the record; **delete** is a hard removal of the account, its
  encrypted connections, and its config overrides.
- **Honest UI.** Borrowed connections render as “Shared from <email>” (never the owner's token hint) and
  offer no Disconnect — only the option to connect your own. A `Read-only` pill sits in the header.

## Consequences

- A stakeholder can be onboarded in one admin action and immediately see the team's board, huddle,
  reports, and Task Helper, on the admin's tokens, without being able to alter Jira.
- Jira's history stays truthful: nobody writes as someone else unless an admin explicitly allowed it.
- `JIRA_WRITE_TOOLS` is a hand-maintained set — a **new Jira-mutating tool must be added there**, or a
  borrower could call it. This mirrors the existing `READ_TOOLS`/`WRITE_TOOLS` allowlists in `askService`
  and is deliberate (audit-friendly, greppable) rather than inferred from tool names.
- Local team stores are shared by design; a read-only borrower can still edit meeting notes and retro.
  If that becomes unwanted, gate those tools on `canWriteJira` too.
- Gate: 1,294 tests (mcp-jira 478→498, react-app 732→739, github 57), smoke 53/53, build clean. Pill → 1.46.0.

## Alternatives considered

- **Block every non-`get_`/`list_` tool for borrowers** — fail-safe for new tools, but it breaks the
  offset wallet's idempotent auto-bank and the huddle's shared notes for exactly the people meant to
  observe them. Rejected in favor of the explicit Jira-write set.
- **Copy the admin's tokens onto the new user** — trivially simple, but the secret is then duplicated at
  rest, revocation is manual, and the audit problem remains without the guard.
- **Read-only enforced in the frontend** — trivially bypassable; the guard must live at `/api/tools`.
