# ADR-072: Huddle attention card retired; Trends per-developer KPIs are leave-adjusted (v1.60)

## Status: Accepted (2026-07-17) — implemented in v1.60

## Context

Three pieces of user feedback from using v1.58/v1.59 live:

1. The Huddle's **"Needs attention"** card (ADR-052 Phase C) should go. Its headline signal —
   "stale" = days since `updatedAt` ≥ 3 — was superseded by the v1.58 Aging card, which measures
   real time-in-column against a points-scaled expectation instead of last-touch recency.
2. The Aging card's **"+N more" was dead text** — with a deep WIP list ("+25 more") most of the
   list was simply unreachable. "We should be able to get all list of it."
3. In Trends & KPIs, developers showed **below the required 8 points for sprints where they were
   on leave** — the per-dev numbers ignored plotted VL/EL/Holiday/Offset days, so a person with
   3 leave days and 5 done points looked like a miss when they had met their adjusted target.
   Also: the **date range should be the default** selection mode on the page.

## Decision

**Retire the attention card entirely** — component, lib, and tests deleted, not just unmounted.
Its two calendar-day helpers (`daysSince`/`toUtcMidnight`) relocate into `lib/aging.ts`, their only
remaining consumer (they were already shared there since v1.58 precisely to avoid a third copy).
Consciously accepted loss: the card's *unassigned* and *PR-review* nudges leave the Huddle with it —
the user asked for removal, and both remain visible elsewhere (the board's Unassigned filter option;
the Code review sidebar card / PR badges). Dashboard's `useIssuePullRequests` fetch stays — the
board's PR badges and the PR card still consume it.

**Aging card: "Show all N / Show less" expander** replacing the static trailer. Per-visit state
(deliberately NOT persisted like the card's collapse — a 30-row list is something you open during
standup, not a layout preference), `aria-expanded`, and `max-h-80 overflow-y-auto` only when
expanded so the default 6-row view stays static. No button at ≤ 6 entries.

**Leave-adjusted per-developer KPIs, computed client-side** (`lib/kpiAdjust.ts`) — zero backend or
contract surface change. The math reuses the capacity convention the user confirmed back in the
offset arc (capacity.ts / offset.ts, ADR-038/048): **every plotted leave day of any type
(VL / EL / Holiday / Offset) relieves one point**, so per dev per sprint:
`adjustedTarget = max(0, requiredPoints − leaveDays)`, `met = donePoints ≥ adjustedTarget` —
including `met` at a zero target (a fully-covered sprint is met by definition). Membership is the
union of the window's `byAssignee` names and the leaves store's names, so a developer fully on
leave with zero tickets still appears (done 0, target reduced, met) instead of vanishing.
"Unassigned" is excluded from the leaves side only. Data: one `get_all_leaves` read (existing tool,
existing `useAllLeaves` hook — the store is bridge-local JSON, no Jira cost) joined with the
already-fetched `MultiSprintReport`; `requiredPoints` flows from Reports' existing `usePolicy()`
call as a prop. The per-dev section renders donePoints against the adjusted target (secondary bar
series "Target (adj)"), a "Met target: N of M sprints" tile, and Leaves/Target/Met table columns
with accessible met/missed labels.

**Date range is the default Trends selection mode**, pre-filled to the span of the last 10 closed
sprints (start = 10th-most-recent closed sprint's `startDate`, end = today —
`defaultRangeFromClosed` in `sprintRange.ts`). The pre-fill is a one-shot effect keyed on the
sprint list's arrival that never clobbers a user-touched input and re-arms on board switch; the
Last-N and pick-sprints modes remain. This keeps the original "usually previous 10" default while
making it visible and editable as dates, per the user's choice.

## Consequences

- The Huddle sidebar is one card lighter and the aging list is fully reachable; the standup
  question "what's been sitting too long" now has the complete ordered answer.
- A developer's trend now reads against what they actually owed each sprint. The team-level table
  and tiles are deliberately unchanged — committed-vs-completed is a Jira-truth metric; the leave
  adjustment applies where the complaint lived: the individual view.
- Client-side join means the AI assistant's `get_multi_sprint_report` payload is still leave-naive;
  the assistant can call `get_all_leaves` itself (both are allowlisted reads). Accepted trade for
  zero new backend surface; backend enrichment remains possible later without breaking anything.
- Leave data is display-name-keyed (as everywhere); the known rename/collision fragility (ADR-071)
  extends to the adjustment.
- Tests: react-app 887 → **911** (−12 with the attention module: attention.test 8 +
  AttentionCard.test 4; +36: kpiAdjust 12, sprintRange +8, aging +4 relocated day-helper pins,
  AgingCard +3 net expander, DeveloperKpiSection +5 net, Reports +4), files 75 → 74. Total
  1,521 → **1,545**; smoke 55/55 (no tool changes). Pill → 1.60.0.
- CONTRACTS changelog v1.60 (items 169–172) records the release as UI-only.
- **Not live-eyeballed** (login gate) — proven by the suite above; the leave math additionally
  pins the exact confirmed convention in kpiAdjust.test.ts fixtures.

## Alternatives considered

- **Unmount the attention card but keep the code.** Dead code with tests that still run; deletion
  is honest and the ADR/CONTRACTS history preserves the design if it's ever wanted back.
- **Backend enrichment** (leaves/targets inside `get_multi_sprint_report`). One richer payload and
  AI-visible adjustments, but a contract change + backend tests for something two existing local
  reads already provide; rejected for scope — revisit only if the AI's trend answers prove to need it.
- **Offset LEDGER (banked balances) in the adjustment.** The wallet is bookkeeping about *earned*
  offsets; what excuses missing points in a sprint is the *plotted* Offset day off, which is
  already a leave type. Mixing the ledger in would double-count.
- **Persisting the aging expander state.** A per-visit disclosure, not a preference; persistence
  is what the card-collapse already does.
- **"Show all" navigating to a dedicated page.** Overkill for a sidebar card; inline expansion
  keeps the standup flow.
