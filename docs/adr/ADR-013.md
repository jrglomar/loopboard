# ADR-013: Reports redesign (vertical layout, story-points focus, 2-decimal) + remove "My Issues" filter

## Status: Accepted (2026-06-12)

## Context

After live use of the v1.4 Reports page, the product owner gave focused feedback:
1. The report's flex-row layout reads poorly — sections sit side-by-side and feel cramped;
   a UI expert should decide proper placement.
2. Committed/completed points sometimes render with many decimals (floating-point sums).
3. The report should be measured in **story points, not issue counts**.
4. The Dashboard's "My Issues" quick filter (v1.3, ADR-010) isn't wanted.

## Decision

1. **Reports layout → vertical, sectioned.** Replace the top-level flex-row with a
   single-column stack of full-width section Cards in reading order (picker + export →
   completion summary → velocity → by-assignee → issue lists → optional AI summary). A
   UI-expert (Frontend Specialist) places the sections and records a short rationale in
   `docs/REPORTS-LAYOUT.md`. Responsive grids *inside* a section (e.g. summary stat tiles)
   are fine; top-level sections do not sit side-by-side.
2. **Story-points focus.** The report's headline metrics are points only: committed,
   completed, completion rate (points-based), and **carryover points** (committed −
   completed). Remove "issues done/total" as a metric and the done/total **issue** columns
   from the by-assignee table (points columns only). Blocked may remain as a small risk
   flag (an impediment signal, not a velocity metric). The Completed / Carryover **lists**
   still enumerate individual issues (with their points).
3. **2-decimal formatting.** A shared pure `formatPoints(n)` renders ≤2 decimals with
   trailing zeros trimmed (`13.5`, `29.75`, `30`), used everywhere points appear; replaces
   the velocity `.toFixed(1)`. Unit-tested.
4. **Remove "My Issues".** Delete the remembered "My Issues" toggle and its `localStorage`
   key from the Dashboard; keep the plain Assignee `<select>` as the filter. Supersedes
   that part of ADR-010.

## Consequences

- The report scans top-to-bottom like a real sprint review document and prints cleanly.
- Velocity/capacity conversation stays in points — consistent with how the team estimates.
- One fewer control on the dashboard filter row (simpler); assignee filtering is unchanged.
- `get_sprint_report` still returns issue counts in its payload (other consumers/Copilot
  may use them); this ADR only changes what the **UI** displays — no tool/contract-output
  change, so backend tests are unaffected.

## Alternatives considered

- **Keep issue counts as a secondary metric** — rejected; the owner wants a points-only
  report, and mixing the two muddies the headline.
- **Strip issue counts from the backend output too** — unnecessary and riskier (other
  callers); a display-only change is sufficient and reversible.
- **Round points server-side** — formatting is a presentation concern; keep raw precision
  in the data, format in the UI (and in `reportMarkdown`).
- **Keep "My Issues"** — the owner explicitly asked to remove it; the assignee select
  already covers per-person filtering.
