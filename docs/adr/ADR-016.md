# ADR-016: Per-sprint leaves/offset tracker (bridge-side JSON) + capacity-adjusted velocity + full-width report

## Status: Accepted (2026-06-12)

## Context

To plan a realistic *committed* velocity, the team needs to account for who is **out** during
a sprint (leaves / offsets). Jira has no native leave concept, and the MCP bridges were
stateless with no datastore. The product owner asked to track each assignee's leaves per
sprint and feed that into a capacity-adjusted velocity, surfaced on the Reports page — and,
separately, to make the report a **full-width** layout rather than the v1.4.1 single column.

The product owner explicitly chose **bridge-side JSON file** persistence (over browser
localStorage) so leaves survive browser clears and are shared by anyone pointing at that
mcp-jira server.

## Decision

1. **First stateful store — a JSON file on the mcp-jira host.** New MCP tools `get_leaves`
   / `set_leaves` read/write `JIRA_LEAVES_FILE` (default `<mcp-jira pkg>/.loopboard-leaves.json`,
   git-ignored). Shape: `{ [sprintId]: { [assigneeName]: ["YYYY-MM-DD", …] } }`. A
   `leavesStore.ts` does read-modify-write, tolerating a missing/corrupt file as `{}`. This
   is a deliberate, user-chosen exception to the stateless-bridge norm; it is contained to
   two tools + one helper, and tests point `JIRA_LEAVES_FILE` at a temp file so the suite
   stays keyless/offline.
2. **Leaves keyed by assignee display name + sprintId.** We have display names from issues,
   not stable accountIds; for a single-team POC that's adequate (documented limitation).
3. **Capacity-adjusted "possible committed velocity"** — a pure `capacity.ts`:
   `totalPersonDays = assignees × sprintWorkingDays` (Mon–Fri within the sprint dates),
   `available = total − Σ leaveDays`, `capacityFactor = available/total`,
   `possibleCommitted = averageCompleted × capacityFactor`. Shown beside the raw forecast,
   **labeled a heuristic** (same honesty stance as ADR-010/012 — we don't dress a planning
   estimate up as a promise), with its inputs visible.
4. **Leaves calendar UI + by-assignee Leaves column** on the Reports page: a working-day
   day-grid per assignee for the selected sprint, editable, persisted via `set_leaves`; the
   by-assignee table gains a Leaves (days off) column.
5. **Reports go full-width** (supersedes ADR-013's single-column): a UI-expert full-width
   responsive dashboard grid — not a simple vertical column, not the old cramped flex-row.

## Consequences

- The app gains real persistence (one JSON file). It must be git-ignored and is host-local;
  not a multi-tenant store — fine for the POC, revisit (a real DB) if this grows.
- Capacity math is a heuristic planning aid; clearly labeled so no one treats it as a
  commitment. Working days are Mon–Fri (holidays not modeled — a documented limitation).
- Leaves tie to display names; renames or duplicate names would need accountIds later.
- The report layout reverses ADR-013's "single vertical column" in favor of full-width;
  the UI-expert records the new rationale in `docs/REPORTS-LAYOUT.md`.

## Alternatives considered

- **Browser localStorage** (the other offered option) — zero backend and offline, but
  per-browser and not shared; the owner chose the shared JSON file.
- **A real database / Jira custom field** — overkill / instance-fragile for a POC; a JSON
  file is the smallest thing that delivers shared persistence.
- **Modeling holidays + part-time factors** — out of scope; Mon–Fri working days + explicit
  leave dates is enough to estimate capacity now.
- **Server-side capacity computation** — the inputs (assignees, sprint dates, leaves) are
  all available client-side and the math is trivial + pure; keeping it in the UI avoids a
  new tool and keeps velocity raw/server-authoritative.
