# ADR-005: Contract-First Parallel Build with File-Ownership Boundaries; Minimal Type Duplication Accepted Over Shared Package

## Status: Accepted

## Context

The POC is built by three specialized agents (Backend, Fullstack, Frontend) working in parallel on three packages (`mcp-jira`, `mcp-github`, `react-app`). Parallel development requires that each agent can make progress without waiting for another agent's package to compile or export types.

The most obvious solution is a shared types package (e.g., `@loopboard/shared` or `packages/shared`) that all three packages import. This is the standard graduated-monorepo pattern. However, it creates a **bootstrap ordering problem** for parallel development:

- The shared package must be authored before the dependent packages can import from it.
- Any change to `@loopboard/shared` forces all three packages to re-resolve types.
- If the shared package is owned by one agent, that agent becomes a bottleneck for the other two.
- npm workspace symlinks are set up at install time; a new package added during development requires `npm install` at the root — which is prohibited by the build constraints.

The alternative — duplicating a small number of types across packages — is considered an anti-pattern in production systems but is pragmatically acceptable in a time-boxed POC where:

1. The types are few and simple (`TicketRef`, `IssueSummary`, `HuddleItem`, `PrSummary` — defined once in `CONTRACTS.md`).
2. `CONTRACTS.md` is the authoritative source for these types. Duplication between packages is **disciplined duplication**: each package copies from the contract, not from each other.
3. The POC's quality gate is `typecheck + build + test` per package, not cross-package type compatibility (which is verified implicitly by the HTTP bridge contract).
4. The graduation path (extract `@loopboard/shared`) is clear and well-understood.

## Decision

**No `packages/shared` (or `@loopboard/shared`) package is created for the POC.**

Each package defines its own local copies of the shared domain types, copied verbatim from `docs/CONTRACTS.md`:

- `packages/mcp-jira/src/lib/types.ts` — defines `TicketRef`, `IssueSummary`, `HuddleItem`.
- `packages/mcp-github/src/lib/types.ts` — defines `PrSummary` (may also import `TicketRef` locally if needed for `link_pr_to_ticket` output).
- `packages/react-app/src/lib/types.ts` — defines client-side copies of all types it consumes from the HTTP bridge responses.

`docs/CONTRACTS.md` is the **single authoritative source** for all type shapes. The implementation files are copies derived from the contract, not the other way around.

**File-ownership boundaries** are strictly enforced:

| Package | Owning agent | Must not edit |
|---|---|---|
| `packages/mcp-jira` | Backend agent | `mcp-github/`, `react-app/` |
| `packages/mcp-github` | Fullstack agent | `mcp-jira/`, `react-app/` |
| `packages/react-app` | Frontend agent | `mcp-jira/`, `mcp-github/` |
| `docs/` | Architect agent | `packages/` |

No package imports from another package's `src/` directory. The only cross-package communication at runtime is via HTTP (bridges at `:4001`/`:4002`).

## Consequences

**Positive:**
- All three agents can work in parallel from day 1 with no coordination required for type definitions.
- Each package typechecks independently without requiring other packages to be built first.
- No `npm install` required after the initial workspace install (no new workspace packages added).
- The build is deterministic: each package's type correctness depends only on its own source and the contract document.

**Negative / trade-offs:**
- Type duplication: `TicketRef` appears in at least two `types.ts` files. If the contract changes (during development, before finalization), all copies must be updated.
- No compile-time enforcement of cross-package type compatibility. The HTTP bridge envelope is the runtime boundary; mismatches surface as runtime errors, not TypeScript errors.
- The duplication is intentionally minimal (4-5 interfaces). Any future addition of shared utility functions or enums would require this ADR to be revisited.

**Mitigation:** The Architect finalizes `CONTRACTS.md` before builders start. The contract is not changed during the build phase. This eliminates the "type drift during development" risk.

## Graduation path

When the POC graduates to a supported product:

1. Extract `packages/shared` (or `packages/core`) containing all shared types, zod schemas (for HTTP response validation on the client), and any shared utility functions (e.g., `jiraKeys.ts` regex if needed in both servers).
2. Add `"@loopboard/shared": "workspace:*"` to each package's `package.json`.
3. Replace local type copies with imports from `@loopboard/shared`.
4. The TypeScript compiler enforces cross-package type compatibility automatically from that point forward.

This migration is a mechanical find-and-replace with no behavioral changes, making it low-risk.

## Alternatives considered

### Alternative A: Create `packages/shared` before starting parallel build

Author the shared types package first (by the Architect or Backend agent), then have all agents start their work.

**Rejected because:** (a) Creates a sequential dependency that defeats the goal of parallel development. (b) The shared package is trivial for the POC (a handful of interfaces) — the coordination overhead exceeds the benefit. (c) Requires adding a new workspace package, which may trigger `npm install` steps that are blocked by build constraints.

### Alternative B: React app imports types from `mcp-jira` package via workspace reference

`react-app` adds `"@loopboard/mcp-jira": "workspace:*"` and imports types from there.

**Rejected because:** (a) Creates a compile-time dependency from `react-app` on `mcp-jira`; if the backend agent's build is broken, the frontend agent's typecheck fails too. (b) Imports a Node.js package (which may include `express`, `axios`) into a browser bundle — build tools would need explicit exclusions. (c) Violates file-ownership boundaries; the frontend agent cannot rely on an artifact owned by the backend agent.

### Alternative C: Centralize types in `CONTRACTS.md` only, with no TypeScript type files at all

Use `unknown` for all cross-package values in the React app; validate at runtime only.

**Rejected because:** The spec explicitly requires `no any` and full TypeScript typing. Runtime-only validation without type files would require `any` or `unknown` casts throughout the frontend hooks, violating the quality gate.
