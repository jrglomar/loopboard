# ADR-064: AI status is read per-user from /api/me/context, not global /api/health (v1.53)

## Status: Accepted (2026-07-13) — implemented in v1.53

## Context

Same bug class as ADR-062, the last remaining instance. Multi-tenancy (ADR-055) runs every AI call on the
signed-in user's own token (context-resolved). But the UI's *gate* for whether AI features show — the
"is AI enabled?" check — still read the **global, unauthenticated `GET /api/health .ai`**, which reflects
only the `.env` `AI_PROVIDER`. So a user who connected **their own** AI token, on an install with **no**
global `.env` AI, was told **"AI disabled"**: TicketGen/Linking fell back to deterministic templates and the
assistant refused, even though their token worked. The four consumers — `TicketGen`, `Linking`, `Reports`,
`AssistantWidget` — all call `getAiStatus()`, which fetched `/api/health`.

## Decision

- **Serve AI status from the per-user context endpoint.** `GET /api/me/context` now also returns
  `ai: { enabled, provider, model }`, computed in the same request context as boards/policy:
  `resolveUser(userId)` → `runWithUser({ … resolved.config … }, () => getAiStatus())`. Because the backend
  `getAiStatus()` reads `getConfig()` (ALS-aware), inside the context it reports the user's EFFECTIVE AI
  (their own token, else inherited/global). Falls back to the global `getAiStatus()` when the user can't be
  resolved.
- **`getAiStatus()` (client) reads `.ai` from the context.** It now calls `getMyContext()` (authenticated
  `/api/me/context`) and returns `.ai`, instead of fetching the global `/api/health`. Any failure (not
  signed in, bridge down, missing field) still returns `{ enabled: false, provider: null, model: null }`, so
  the page never crashes. Its signature is unchanged, so all four consumers — including TicketGen's
  re-check after "Use AI drafting" — keep working with no change.
- **`/api/health .ai` stays global + keyless.** Still returned for smoke/health; just no longer the UI's
  source.

## Consequences

- A user on their own AI token now sees AI drafting and the assistant enabled, matching every other
  per-user surface — the last "global `/api/health` leaks into a per-user decision" bug is closed.
- No component or contract-endpoint change: the fix is one added response field + one client function body.
  The signature-preserving approach means TicketGen's refresh-after-toggle semantics are retained (a hook
  reading `AuthContext` would have lost them).
- Minor: `getAiStatus()` now fetches the fuller `/api/me/context` (which `AuthContext` also fetches) rather
  than the tiny `/api/health`. Acceptable — it's a light endpoint and already the per-user source of truth;
  the alternative (a `useAiStatus()` hook off `AuthContext`) was rejected to avoid touching four components
  and dropping the manual re-check.
- Gate: typecheck + build clean, 1,361 tests (github 57, mcp-jira 537, react-app 767), smoke 53/53.
  Pill → 1.53.0. New tests: delegation.test.ts +2 (a user on their own AI token → `getAiStatus()` enabled
  in-context while global is off); aiClient.test.ts getAiStatus block re-pointed at the context envelope.

## Alternatives considered

- **A `useAiStatus()` hook reading `AuthContext.context.ai`** (mirrors the ADR-062 `useBoards` move). Most
  consistent and avoids the extra fetch, but touches all four consumers and, in TicketGen, loses the
  deliberate re-check after "Use AI drafting" (the context loads once). Not worth the churn/regression for a
  focused bug fix; can revisit if the extra fetch ever matters.
- **Make `/api/health` context-aware.** Rejected for the same reason as ADR-062 — health is intentionally
  global/unauthenticated (keyless smoke, liveness). Per-user questions belong on `/api/me/context`.
- **A dedicated `GET /api/me/ai-status`.** Redundant — `AuthContext` already fetches `/api/me/context`; AI
  status belongs alongside boards/policy there.
