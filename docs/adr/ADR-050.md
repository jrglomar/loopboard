# ADR-050: P1 hardening — live-verified writes, assistant memory & reach, Huddle freshness

## Status: Accepted (2026-07-04) — implemented; write paths and assistant memory verified LIVE

## Context

A feature review picked four P1 items: (1) several shipped write features had never run against real
Jira; (2) the assistant couldn't see dev-panel PRs or offset balances, forgot everything between
questions, and couldn't file leaves; (3) the Huddle only refreshed on navigation; (4) the Reports
Completion Summary had dead space. (A fifth idea — defaulting Reports to the latest closed sprint —
was rejected by the team: they deliberately keep sprints open for monitoring.)

## Decision

1. **Live verification (no code change; one env fix).** Executed against the real instance:
   PO create, 3× Dev create with link + points + assignee (the PO→2-Dev breakdown), update_ticket
   points, assign_issue. Finding: `.env` still pinned `JIRA_LINK_TYPE=Relates` (overriding the v1.36
   default), and the instance's dependency type is named **"Depends on"**. After the env fix the link
   verifies as **"VBPO-x depends on VRDB-y"** — the exact chosen semantics; the payload direction
   (inward=dev, outward=po) was already correct. Test tickets left clearly marked for deletion.
2. **Assistant reach (askService):** READ_TOOLS += `get_issue_pull_requests`, `get_all_leaves`,
   `get_offset_ledger`; WRITE_TOOLS += `set_leaves` (proposal-only, modal-confirmed like all writes;
   the system prompt warns the model that set_leaves REPLACES a person's sprint entries, so it must
   read-merge first). ConfirmActionDialog renders non-primitive args as JSON.
3. **Assistant memory:** `/api/ai/ask` gains optional `history` (≤8 `{role, content}` turns ≤2000
   chars). The service folds it into the system prompt ("Conversation so far") — provider-agnostic,
   zero adapter changes. ChatPanel keeps its Ask-mode turns in a ref and sends them automatically.
   Verified live: a follow-up "who owns it?" resolved the ticket named only in history.
4. **Huddle freshness:** pure `useAutoRefresh(cb, ms, enabled)` hook; the Dashboard refetches
   sprint + huddle every 5 minutes and shows "Auto-refreshes every 5 min · Updated HH:MM".
5. **Completion Summary:** pure `remainingByStatus(notCompleted)` splits open points into
   To Do / In Progress tiles (code review counts as completed per the ADR-014 DoD).

## Consequences

- No new tool; jira tools stay 36. The ask endpoint's input widened (optional field) — old clients
  unaffected. Memory is prompt-folded, so provider adapters stay single-shape.
- History caps (8 turns × 500 chars folded) bound prompt growth; the ref resets with the widget.
- Auto-refresh is fixed at 5 min (constant in Dashboard) — make it configurable only if asked.

## Alternatives considered

- **True multi-turn message history through the provider port** — richer, but requires adapter
  changes for plain-assistant messages in both providers; prompt folding delivers 90% for 10% cost.
- **Auto-executing "safe" writes (set_leaves) without confirmation** — rejected; the ADR-030
  invariant (the AI never mutates, humans confirm every write) stays absolute.
