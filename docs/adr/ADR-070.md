# ADR-070: Ticket aging on the Huddle — changelog-derived Work Item Age, scaled by story points (v1.58)

## Status: Accepted (2026-07-17) — implemented in v1.58

## Context

The user asked to track how long in-progress Jira tickets have been sitting, **based on their points**,
visible on the **Huddle**, and consistent with Scrum practice. Scrum's metric here is **Work Item Age** —
elapsed time since work *started* on an item that is not yet done.

What existed was not that. The Attention card's "stale" rule (ADR-052) counts days since `updatedAt` ≥ 3:
a comment or a field edit resets it, and it says nothing about how long the ticket has been *in its
column*. **No status-transition history existed anywhere in the codebase** (grep-verified) — `IssueSummary`
carried no timestamps beyond `updatedAt`, so age could not be derived from any data already on hand.

A flat "3 days is stale" threshold is also wrong for a pointed backlog: a 13-point ticket at 5 days is
healthy, a 1-pointer at 5 days is stuck. The expectation has to scale with the estimate.

## Decision

**Source: the dedicated per-issue changelog endpoint** — `GET /rest/api/3/issue/{key}/changelog?startAt&maxResults`
(`getIssueChangelogRaw`), **not** `expand=changelog` on a search. This repo was already burned by a
search-family endpoint change (v1.44.1: `/rest/api/3/search` → 410), and the per-issue endpoint is stable
and paginated. Cost is bounded to the **inprogress + codereview** buckets only (typically 5–20 per sprint),
fanned out with `Promise.all` and a per-key `try/catch` → `null` (the `getIssuePullRequests.ts` pattern).

**Age definition: `inProgressSince` = the LATEST changelog entry with a `status` item whose `toString`
equals the issue's CURRENT status.** This mirrors Jira's own "days in column" affordance, resets naturally
on bounce-backs (a ticket kicked back from Code Review to In Progress is young again — which is what the
huddle wants to see), and needs no historical status-name → category inference. The same rule serves both
in-flight buckets, so "aging" means *time in the current column*, not time since the first start.

**Bounded 2-page fetch:** page 1 (`startAt=0, maxResults=100`); if `!isLast && total > 100`, also fetch the
tail page at `startAt = total − 100`. `resolveInProgressSince` takes the **max over all pages** rather than
the last entry seen, so it is order-independent — it does not depend on an unverified assumption about Jira's
page ordering. No match, a gap in the middle of a very long history, or a failed fetch → **`null`, never a
guess**: the ticket simply carries no age and is omitted from the aging UI.

**Opt-in, so the read paths never pay.** `get_active_sprint` gains `withAging?: boolean` (default **false**);
enrichment runs only when true. `get_velocity`, `get_sprint_report`, and the huddle call `getSprintIssues`
directly and are byte-identical to v1.57. Only the Dashboard's **primary** sprint call opts in — the
opposite-board fly-in fetch does not (it needs keys and summaries, not ages).

**Policy plumbed as a new sibling `aging` key**, copying the offset-policy chain exactly (ADR-062/064):
`JIRA_AGING_BASE_DAYS` (default 1) and `JIRA_AGING_DAYS_PER_POINT` (default 1) → `getAgingPolicy()` →
exposed on **both** `GET /api/health` (global, sentinel fallback) and `GET /api/me/context` (per-user, inside
the same `runWithUser` destructure). The frontend reads it via `useAgingPolicy()` off `/api/me/context` —
**not** `/api/health` — which is the whole point of ADR-062/064: a per-user board override must not read a
global value.

**The math is pure and injected:** `expectedDays = baseDays + daysPerPoint × (storyPoints ?? 0)`;
`ratio = ageDays / expectedDays`; tiers **ok < 100% ≤ watch < 150% ≤ overdue**. Calendar days via
attention.ts's `daysSince`/`toUtcMidnight` (now exported rather than copied a third time — ADR-052's
convention is shared, not duplicated). Unpointed tickets get the base only and are flagged `unpointed`;
a degenerate all-zero policy yields `Infinity` rather than `NaN`.

**Two surfaces, one computation:** an `AgingCard` on the Huddle sidebar (worst-first, tier counts, own
collapse key) and an age chip on the board's `IssueCard`. The board takes `agingPolicy` as an optional
**prop with a default**, not a hook — `useAuth` throws without a provider, and `SprintBoard` is rendered
provider-less in tests.

## Consequences

- Huddle answers "what's been sitting too long *for its size*" — a 1-pointer at 5 days is overdue while a
  13-pointer at 5 days is fine, from the same policy.
- Cost: ≤ 2 changelog calls per in-flight issue, only on the Dashboard's primary sprint. `withAging:false`
  is free, which is why the 11 existing jiraClient-mocking test files needed **zero** changes — the new mock
  export is unnecessary where the flag is never set.
- Tickets whose history is unreadable show **no chip** rather than a wrong one. Silent-by-omission is the
  deliberate trade: a fabricated age on a huddle is worse than a missing one.
- Age resets on bounce-back. If total-time-since-first-start is ever wanted, that is a different resolver
  over the same fetched pages (min over entries into any in-progress-category status) — the payload already
  supports it.
- Admin gains a "Ticket aging" field group; both fields validate as int ≥ 0, max 365.
- Tests: mcp-jira 543 → 554 (new `test/aging.test.ts`, 11), react-app 796 → 822 (`lib/aging.test.ts` 14,
  `AgingCard.test.tsx` 7, +4 SprintBoard chip, +1 Dashboard withAging-threading). Total 1,396 → **1,433**;
  smoke 55/55. Pill → 1.58.0.
- **Not live-eyeballed**: every new surface sits behind the login gate, so this is proven by jsdom/unit
  tests against fixtures, not by a real Jira changelog in a browser.

## Alternatives considered

- **`expand=changelog` on the sprint-issues search.** One call for the whole sprint instead of N, but it
  rides the search family this repo has already been broken by (v1.44.1, 410), and it returns only the
  *first* changelog page per issue with no tail access — exactly the ancient-ticket case that needs it.
- **Derive age from `updatedAt`** (no new fetch). Free, and wrong: any edit or comment resets it. That is
  the Attention card's existing "stale" rule, which is why it stays separate rather than being retitled.
- **A flat day threshold.** Ignores the estimate, which is the substance of the request.
- **Always-on enrichment.** Would put N changelog calls on the velocity and report paths that never render
  a chip, and would force a mock-export edit across 11 test files to prove a cost nobody asked for.
- **Age since first entry into any in-progress status** (rather than the current status). Truer to
  literal Work Item Age, but a bounced ticket would show its old age with no way to see the current column's
  clock — less actionable in a daily huddle, and it needs status-name → category inference for historical
  names the board may no longer have.
