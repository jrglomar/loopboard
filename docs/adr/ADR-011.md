# ADR-011: Sprint management — select future sprints, create sprints, place tickets in a sprint

## Status: Accepted (2026-06-12)

## Context

The dashboard could only show *active* sprints (ADR-007 picked the latest active). Teams
plan ahead: they need to look at the **next (future)** sprint, **create** a new sprint with
a goal and dates, and **prepare tickets** into it. These are first-class Jira Agile
capabilities the POC had not yet surfaced — including a genuine **write** (creating a
sprint and adding issues to it).

## Decision

1. **Future-sprint selection.** `get_active_sprint` now fetches `state=active,future`,
   returns both `activeSprints` (latest-first) and `futureSprints` (earliest-first, so the
   *next* sprint leads), and `sprintId` may select an active OR future sprint. Default is
   unchanged (latest active); when no active sprint exists it falls back to the next future
   one. A future sprint usually has no issues — the board renders an empty planning view
   with a "Future sprint" badge. One tool, one extra fetch param — no new round trip for
   the selector.
2. **`create_sprint` is a real MCP write tool.** `POST /rest/agile/1.0/sprint`
   (name, goal, dates → a future sprint). Unlike the AI endpoints (bridge-only, ADR-006),
   this is a legitimate Jira action Claude/Copilot should be able to perform, so it is
   registered as an MCP tool *and* exposed on the bridge for the UI. The UI gates it behind
   an explicit dialog whose copy states it creates a real sprint.
3. **Tickets into a sprint via an optional `sprintId`.** `create_po_ticket` /
   `create_dev_ticket` take an optional `sprintId`; after creating the issue the server
   calls `POST /rest/agile/1.0/sprint/{id}/issue`. This is **non-fatal** (mirrors the
   `linkWarning` pattern): a ticket whose project isn't on that sprint's board still gets
   created, with a `sprintWarning`. Covers "prepare a ticket for that sprint" without a
   separate move tool (moving *existing* tickets is deferred — the user chose create-into).
4. **`list_sprints`** backs the create/target/report pickers (active/future/closed).

## Consequences

- The dashboard becomes a planning surface, not just a live-sprint view.
- A real write path now exists. Tests stay keyless/offline (mocked client); the live write
  is only exercised intentionally from the UI dialog or Copilot. Smoke tests must NOT call
  `create_sprint` against a real board — they assert validation/registration only.
- `sprintId` on creation is best-effort by design; the UI surfaces the warning so users
  understand board/project constraints rather than seeing a hard failure.
- `get_active_sprint`'s output grew (`futureSprints`); the frontend selector and types
  follow. Backward compatible — `activeSprints` is unchanged.

## Alternatives considered

- **A separate `get_future_sprints` tool** — a second round trip for the selector; folding
  future sprints into the existing board fetch is cheaper and keeps the selector populated
  from one call.
- **`create_sprint` as a bridge-only endpoint** (like the AI ones) — rejected; creating a
  sprint is a normal Jira operation Claude should be able to do via MCP, not an LLM-shaped
  concern. Gating lives in the UI, not by hiding the tool.
- **A dedicated `move_issue_to_sprint` tool now** — deferred; the user scoped this round to
  *creating new* tickets into a sprint. Easy to add later (the helper already exists).
- **Hard-failing ticket creation when the sprint add fails** — rejected; the ticket is
  still valuable, so we keep creation succeeding and warn (consistent with `linkWarning`).
