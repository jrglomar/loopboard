# ADR-008: Code Review as a first-class board bucket + client-side assignee filter

## Status: Accepted (2026-06-11)

## Context

Live use surfaced two dashboard gaps:

1. The team's Jira workflow has a distinct **Code Review** status between In Progress and
   Done. v1.0/v1.1 mapped it to Jira's `inprogress` status category, so review work was
   invisibly merged into the In Progress column and the huddle's "in progress" count.
2. On a shared sprint board, a developer wants to see **only their own tickets** — the
   Jira board has per-assignee filtering; our dashboard had none.

## Decision

1. **Code Review bucket.** A status is "code review" iff its `statusCategory` is
   `inprogress` AND its lowercased status name matches `JIRA_CODE_REVIEW_STATUSES`
   (env-configurable, default `code review,in review,peer review,review`). The
   category guard means a done-category "Reviewed" or a todo-category "Ready for review"
   can never be reclassified. `get_active_sprint` gains `issuesByStatus.codereview` +
   `totals.codereview`; `get_daily_huddle` gains a `codeReview` bucket (precedence
   done > blocked > codeReview > inProgress > upNext) and a new `summaryText` segment.
   `IssueSummary.statusCategory` still reports Jira's raw category — this is bucketing,
   not a field change, so Copilot/MCP consumers see the underlying truth.
2. **Assignee filter is client-side and presentational.** `IssueSummary.assignee` already
   ships on every issue; the Dashboard derives the distinct-assignee list (plus
   "Unassigned") and filters the rendered cards across all four columns in the browser —
   no new tool, no refetch, no contract change to the tools themselves. The huddle digest
   is never filtered (it is a whole-sprint standup artifact). The filter resets on sprint
   change.

## Consequences

- The board reads true to the team's workflow; review bottlenecks are visible at a glance
  and countable in the standup digest.
- Teams whose review status is named differently set one env var — no code change.
- The filter adds zero server load and works offline against already-loaded data; it is a
  view concern owned by the UI (consistent with stateless bridges, ADR-001).
- A blocked ticket that is also in review shows under blocked in the huddle (impediments
  outrank review) but stays in its real column on the board with a blocked badge — board
  mirrors Jira columns; huddle prioritizes what blocks the team.

## Alternatives considered

- **Hard-coding the code-review status name** — brittle across instances; env list is
  cheap and testable.
- **A server-side `assignee` tool parameter** — a round trip per filter change and more
  tool surface, for data the client already holds. Rejected for the POC; revisit if board
  sizes outgrow a single fetch.
- **Filtering the huddle too** — breaks the "standup digest = whole team" intent; the
  standup should still call out everyone's blockers.
- **Treating code review as its own `statusCategory`** — Jira only has three categories;
  inventing a fourth on `IssueSummary` would lie about the source data. Bucketing keeps
  the field honest.
