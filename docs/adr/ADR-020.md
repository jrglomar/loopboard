# ADR-020: Board-wide recent assignees, full search-all in Manage Team, and no off-team assignment lock

## Status: Accepted (2026-06-16)

## Context

v1.8 (ADR-019) introduced the curated per-board team roster, seeded from
`get_recent_assignees`. Live use surfaced three follow-up issues from the product owner:

1. **Only ~3–4 people surfaced as "recent."** `get_recent_assignees` sampled the latest
   `sprintCount` (default 3) sprints **closed-first, then active**, sliced to `sprintCount`.
   On the team's board (≥3 closed sprints) the **active sprint was never reached**, so anyone
   whose recent work was only in the active sprint was missed. The seed list was both too
   small and skewed to older closed work.
2. **No way to add someone who wasn't recently assigned.** The roster could only be built from
   the (truncated) recent list — there was no full-roster search, so the recent list acted as
   a hard ceiling.
3. **The assignment dropdown locked off-team assignees.** A ticket whose current assignee was
   not on the curated team showed a **disabled** "(not on team)" option. The owner found this
   an unwanted restriction.

## Decision

1. **`get_recent_assignees` scans the whole board, not N sprints.** Use
   `GET /rest/agile/1.0/board/{boardId}/issue` with
   `jql = assignee IS NOT EMPTY AND updated >= -{withinDays}d ORDER BY updated DESC` and
   `fields=assignee`, paging up to `maxResults` issues, then collect distinct assignees by
   `accountId` with ticket counts. This covers the **active sprint + backlog + recently-closed
   work** in one query. Input changes: `sprintCount` → **`withinDays` (default 90)**;
   `maxResults` default **100 → 200**. New jiraClient helper `getBoardAssigneesRaw(boardId,
   jql, maxResults)` (the only place that pages the agile board-issue endpoint).
2. **"Search all people" in Manage Team.** The dialog gains a search box backed by
   `useAssignableUsers(boardId)` → `get_assignable_users` (the full assignable list, ~50 on
   this site), filtered **client-side by display name**. Any person can be added regardless of
   recent activity. The unfiltered list is opt-in (you must search), so the v1.8 "no strangers
   dumped in by default" property is preserved — the strangers were a problem only when they
   were the *default* roster, not when explicitly searched for.
3. **No off-team assignment lock.** In the assignment dropdown, a current assignee who is not
   on the curated team is rendered as a **normal, selectable** `<option>` (plain display name),
   so the assignment is both preserved and re-selectable. The dropdown remains **Unassigned +
   curated team (+ the current off-team assignee when present)** — it still does not enumerate
   the org-wide roster inline; to assign a brand-new person, add them via "Manage team → Search
   all people" first.

## Consequences

- The "usual members" seed now reflects everyone actually working the board (the team's ~6
  members incl. active-sprint people), not a 3-closed-sprint sample.
- `get_recent_assignees` no longer depends on the sprint sort helpers; it depends on the board
  issue endpoint. Output shape is unchanged (`{ boardId, assignees[] }`), so the frontend
  contract for the recent list is stable; only the input param renamed.
- `get_assignable_users` graduates from "optional typeahead" to the explicit search source for
  roster building — but is still never the silent default roster.
- Dropping the off-team lock means a non-team assignee can be left assigned or re-picked; this
  matches how teams actually carry occasional cross-team help without first editing the roster.

## Alternatives considered

- **Keep sprint sampling but raise `sprintCount` / put active first.** Fragile — still tied to
  sprint boundaries and counts; a board-wide JQL with a date window is simpler and complete.
- **No date window (scan all board issues).** Unbounded on a large board (the team's board has
  78+ sprints of history); `updated >= -90d` bounds cost and matches "recently assigned."
- **Make the assignment dropdown list all assignable users again.** Reintroduces the exact
  v1.7 "strangers" problem this line of work fixed; the search-all add box is the controlled
  way to reach non-team people.
- **Server-side `get_assignable_users?query=` typeahead.** Jira's assignable search does accept
  a `query`, but the full list is small (~50) and fetched once; client-side filter is simpler,
  instant, and needs no per-keystroke round-trips.
