# ADR-077: Working-day sprint timeline + pluggable storage (json | sqlite) (v1.65)

## Status: Accepted (2026-07-18) ŌĆö implemented in v1.65

## Context

Two user requests, one release:

1. **The Huddle's day counter lied on weekends.** "Day 4 of 13 ┬Ę 10 days left" was pure
   calendar/millisecond math (`computeTimeline`, v1.3) ŌĆö on a Saturday it showed phantom progress
   through days nobody works. The user expected working days: "Day 3 of 10". The repo already had
   the working-day convention (`sprintWorkingDays` feeds burndown and capacity) ŌĆö the timeline
   simply never used it.
2. **Production should be database-ready while dev stays JSON.** The earlier persistence review
   (ADR-075) set the ladder: JSON+atomic ŌåÆ SQLite when needed ŌåÆ Postgres if ever multi-instance.
   The user asked for the environment split now. Decisive architectural fact: **the entire store
   call graph is synchronous** ŌĆö every store, tool handler, and route consumes store reads
   directly. `better-sqlite3` is synchronous; `pg` is not. Choosing Postgres here would have
   forced an async rewrite of all 11 stores and every consumer ŌĆö a phase three times the size for
   no present need. User choices (AskUserQuestion): **SQLite** as the production driver;
   **auto-import on first boot** as the migration path.

## Decision

**Timeline (P1)**: `computeTimeline` rebuilt on `sprintWorkingDays` ŌĆö one working-day convention
for timeline, burndown, and capacity; no third implementation. `dayOfN` = working days Ōēż today
(weekends clamp to Friday's count; pre-start = 0 ŌĆö fixing a latent always-"Day 1" quirk),
`daysLeft = totalDays ŌłÆ dayOfN` (uniform, unlike the old boundary-inconsistent formula), pace
consumes the working-day fraction unchanged. The rendered format stays "Day N of M ┬Ę X days
left" with a "working days (MonŌĆōFri)" tooltip.

**Storage port (P2)**: `StorageDriver { readDoc(scope, name); writeDoc(scope, name, data) }` ŌĆö
drivers take scope literally and never read request context themselves; a `currentScope()` helper
derives it from the SAME AsyncLocalStorage source the config's path resolution already used. The
three pre-existing scope patterns are preserved exactly: nine team stores = user-context-or-shared,
the users store = always shared, the journal = always the real user id. The **json driver** is the
old behavior extracted, not re-implemented: identical paths (per-store `*_FILE` overrides
honored; the generic default formula was verified to reconstruct all 10 `DEFAULT_*_FILE`
constants byte-for-byte), `writeJsonAtomic` writes, identical shapes ŌĆö proven by all 584
pre-existing tests passing unchanged. Driver selection: `STORAGE_DRIVER` env, default `json`.

**SQLite driver + auto-import (P3)**: one `docs(scope, name, data, updated_at)` table,
`INSERT OR REPLACE`, WAL, lazy singleton, `STORAGE_SQLITE_FILE` (compose points it at `/data`).
On first open with an EMPTY table, existing JSON docs (shared files by the json driver's own
resolution + a directory scan of the per-user stores) import in one transaction with a loud log;
JSON files stay untouched as the natural backup; the import never runs again. Contract tests run
the SAME suite against both drivers; sqlite tests use `:memory:` (offline, real driver).

**Also fixed in passing**: `.env.docker.example` had NEVER been committed ŌĆö the `.gitignore`
`.env.*` pattern swallowed it and only `.env.example` was negated (which is also why the v1.64
rebrand never saw it). Negation added; the template's first commit ships with placeholder board
IDs (its on-disk copy carried the developer's real ones) and the new storage vars documented.

## Consequences

- The standup counter now matches how the team actually counts: today's live case reads
  Day 3 of 10 ┬Ę 7 left instead of Day 4 of 13 ┬Ę 10 left; the pace chip follows the same clock.
- Production can flip to a real database with one env var and zero data steps; dev workflows,
  tests, and the `*_FILE` override ecosystem are untouched by default.
- SQLite remains per-instance ŌĆö the one-replica rule (ADR-075) stands. Postgres stays possible
  behind the same port; its true cost (async-ifying the store call graph) is now written down
  rather than discovered mid-phase.
- **Alpine ŌåÆ Debian slim (the fallback became the default).** better-sqlite3 ships no musl
  prebuilds (upstream #619/#1382), so the alpine images had to compile it from source via
  `apk add python3 make g++`. That path first proved wider than scoped ŌĆö `github.Dockerfile` and
  `web.Dockerfile`'s build stage also `npm ci` the whole workspace, so both compiled
  better-sqlite3 too despite never using storage, and all three got the apk line ŌĆö and then
  **failed in the field on the very first real `docker compose up --build`**: `apk` could not
  resolve g++/make/python3 (the build host couldn't reach Alpine's package CDN ŌĆö three standard
  packages missing together = the index never loaded, an environmental network issue, not the
  repo's). Rather than make image builds depend on a working path to `dl-cdn.alpinelinux.org`,
  all three Node stages switched to **`node:20-slim`** (Debian/glibc), where prebuild-install
  downloads the prebuilt binary and no `apk`, compiler, or Python is involved at all. The
  bridges' HEALTHCHECK moved from busybox `wget` (alpine-only) to a `node -e` http probe
  (Node is always present). The web serve stage stays `nginx:alpine` (no Node there). Cost:
  a larger base image, for a build that no longer compiles anything and no longer needs Alpine's
  package mirror. **Still not run through a real `docker build` in-house** (no Docker daemon
  available here) ŌĆö but it removes the exact step that failed, rather than hoping the mirror
  comes back.
- New dependency: better-sqlite3 12.11.1 (+types). Native module; Windows prebuild installed
  cleanly. `@types/better-sqlite3` sits in `dependencies` (cosmetic).
- Tests: react-app 948 ŌåÆ **956** (+8 timeline incl. the reproduced live case), mcp-jira 584 ŌåÆ
  **611** (+27: dual-driver contract suite, auto-import, config). Total **1,624**; smoke 55/55.
  Pill ŌåÆ 1.65.0.

## Alternatives considered

- **Postgres now** ŌĆö offered with its cost stated; declined for the sync-rewrite reason above.
- **Port only, no driver** ŌĆö would have made "production-ready" a promise instead of a switch.
- **Explicit migration script** instead of auto-import ŌĆö offered; the zero-step Docker upgrade won.
- **A third working-day helper** for the timeline ŌĆö the whole point was to stop at two.
