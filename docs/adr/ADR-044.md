# ADR-044: Offset wallet — main tracker with live spend + auto-banked earn

## Status: Accepted (2026-07-01)

## Context

The Offset Tracker (ADR-038) recorded each sprint's earned/spent snapshot only when the user clicked
"Record this sprint", and the balance came from the stored `spent`. The user wants a **main tracker of
every developer's total offset points** where **using an offset (plotting an `Offset` leave) auto-deducts
live** and **earning auto-adds**, plus (Phase 2) a per-user usage history.

Constraint: computing "earned" for a sprint needs that sprint's done-points (`get_sprint_report`), so
fully-automatic earning across *all* sprints would fetch a report per sprint (heavy). The user accepted
**auto-bank on view** instead.

## Decision

Frontend-only. A **wallet** model, computed from data we already load:

- **Spend is derived, live.** `countOffsetLeaves(get_all_leaves)` sums each developer's `Offset`-type
  leave days across every sprint. Plotting/removing an Offset leave (via the shared `useAllLeaves`)
  changes the balance immediately — no separate write, no refund logic.
- **Earned is banked per sprint, on view.** An effect on the Offset Tracker page idempotently records the
  selected sprint's computed earned (`set_offset_for_sprint`, `spent: 0`) when its report + leaves load,
  guarded by a `{assignee:earned}` signature so it writes only when earned changes and never loops (the
  signature is independent of the ledger the write refreshes). The manual "Record this sprint" button is
  removed.
- **Balance = earned − spent + manual**, computed by `computeOffsetWallet(ledger, allLeaves)` using ONLY
  the ledger's `earned` + `manualAdjust` (never its stored `spent`/`balance`, which are now vestigial).
  Old stored `spent` is therefore ignored (no migration, no double-count).
- **UI:** a new `OffsetWalletCard` at the top of the page shows each developer's balance (earned/used),
  live. The per-sprint offset table's Balance column switches to the wallet; the manual-adjust input
  (`set_offset_adjustment`) is unchanged. (Phase 2 adds a per-row History button + modal — see ADR-044
  continued.)

## Consequences

- One live, correct running balance per developer with automatic add (banked earn) and deduct (Offset
  leaves), from existing tools — no backend/contract/tool change (`get_offset_ledger` /
  `set_offset_for_sprint` / `get_all_leaves` reused; smoke unchanged).
- "Earned" only banks for sprints you open on the tracker (the accepted trade-off vs. fetching every
  sprint's report). Idempotent, so re-opening is safe.
- The ledger's stored `spent` is dead; a future cleanup could drop it from `set_offset_for_sprint`.

## Alternatives considered

- **Keep the manual "Record" button + stored spent.** Rejected — the user wants automatic add/deduct.
- **Spend as explicit transactions written on plot (with refunds on un-plot).** Rejected — dual-store
  coordination + refund logic is error-prone; deriving spend from the leaves the user already plots is
  simpler and always correct.
- **Fully-automatic earn across all sprints.** Rejected — a `get_sprint_report` per sprint is too heavy.
