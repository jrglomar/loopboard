# ADR-075: Crash-atomic store writes + deployment hardening checklist (v1.63)

## Status: Accepted (2026-07-18) — implemented in v1.63

## Context

A pre-deployment review (user-requested) assessed two things: the JSON-file stores versus a
database, and token safety. Verdicts: the JSON stores are the right call at this scale and the
token handling is sound (AES-256-GCM per value, fresh IV + auth tag, key in env separate from
ciphertext, masked hints only to the client, no secret logging) — but the review surfaced one
concrete code flaw and several deployment-side requirements:

- **Store writes were not atomic.** All 11 store modules wrote with plain `fs.writeFileSync`;
  a crash or power loss mid-write could truncate the file — and the file is the whole store.
- The remaining risks were configuration, not code: TLS, `NODE_ENV` for the cookie `secure`
  flag, backup/key separation, token scopes, and the single-replica constraint — none of which
  DEPLOYMENT.md's hardening section spelled out with the *why*.
- One unknown: whether login had brute-force protection (unverified at review time).

## Decision

**One shared `writeJsonAtomic(filePath, data)` helper** (`lib/atomicFile.ts`): serialize with the
stores' exact existing formatting (`JSON.stringify(data, null, 2)`, utf8) → write a same-directory
`<file>.tmp` → `fs.renameSync` over the target. rename(2) is atomic on the same filesystem, so a
reader (or a crash) sees either the old complete file or the new complete file — never a
half-write. All 11 stores (leaves, team, impediments, prs, post-scrum, meeting-goal, offset,
meeting-notes, retro, journal, users) swapped to it; read paths, filenames, and JSON shapes are
byte-identical, which is why every existing store test passes unchanged. `.gitignore` += `*.tmp`
(a hard crash can strand a temp file; readers ignore it, and the next write replaces it).

Deliberately NOT done: fsync before rename (power-loss-during-page-cache-flush is beyond this
tool's threat model and would complicate the helper for marginal gain on team-scale data), and
in-process write-queueing (Node's single thread + synchronous writes already serialize the
file operation itself; the read-modify-write last-writer-wins race across async request
boundaries is a separate, accepted semantic — ADR-074-era assessment).

**DEPLOYMENT.md §5 became a concrete 12-item checklist** carrying the review's findings with
their reasons: TLS mandatory (the Connections token paste + session cookie are the exposed hop);
`NODE_ENV=production` for the cookie `secure` flag — **verified already baked into both bridge
Dockerfiles** (`ENV NODE_ENV=production`), so the shipped images are correct by default; scheduled
data-volume backups with the hard rule that `.env` never travels in the same artifact
(`TOKEN_ENC_KEY` must not accompany the ciphertext it unlocks); fine-grained GitHub PATs and
Atlassian scoped API tokens with expiries (rotation = reconnect in Connections); exactly ONE
replica per bridge (per-instance local stores silently diverge otherwise). §3.3 notes the new
write atomicity.

**Login throttling: verified and documented, not changed.** The review's open question is now
answered in code: `POST /api/auth/login` has a per-email in-memory limiter (10 fails / 15 min →
429, cleared on success). Its limits are documented in the checklist: signup has none, the key is
the email (IP spraying isn't throttled), and the counter is per-instance and resets on restart.
Recommendation on record: if ever internet-facing, add an IP-based limiter at the reverse proxy,
not in the app.

## Consequences

- The one code-level durability flaw from the review is closed; the JSON-store decision stands on
  firmer ground (SQLite remains the documented next rung if the tool outgrows a single team).
- Deploying by the book now produces a correct setup: TLS terminated in front, secure cookies
  (default images), separated backups, scoped tokens, one replica.
- Tests: mcp-jira 577 → **582** (atomicFile.test.ts: formatting parity, full overwrite, no
  leftover tmp, leavesStore round-trip); react-app 948 and github 57 untouched. Total **1,587**;
  smoke 55/55. Pill → 1.63.0.
- Windows note — the theoretical risk turned real at the first gate: one full-suite run failed 2
  tests exactly once (never reproduced across 7 subsequent runs), consistent with an AV scanner
  transiently locking the freshly-created temp file on first sight. Rather than ship an
  unexplained flake, the helper's rename now retries up to 5× on EPERM/EACCES with a ~5 ms
  spin-wait (the graceful-fs mitigation, synchronous variant); any other error — or a persistent
  lock — still throws. Pinned by tests (transient-EPERM-then-success lands the write; ENOENT
  propagates without retry). The deployment target is Linux containers where rename is
  POSIX-atomic and the retry path never triggers.

## Alternatives considered

- **Jump to SQLite now.** Solves atomicity too, but swaps a one-helper fix for a store-layer
  migration; the review's ladder (JSON+atomic → SQLite when needed → Postgres if multi-instance)
  stands.
- **Per-store bespoke temp handling.** Eleven copies of the same four lines is how one of them
  ends up wrong; one helper, one test file.
- **Implement signup/IP rate limiting now.** Out of the approved scope (report-only), and the
  right place for IP throttling is the reverse proxy in front, not Express — documented instead.
