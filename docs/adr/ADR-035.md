# ADR-035: Huddle Fly-in tracker + bigger assistant + theme refresh

## Status: Accepted (2026-06-26)

## Context

Three user requests for the Huddle: (1) a **Fly-in tracking** section listing current-sprint tickets
whose title is LIKE "FLY IN"; (2) the floating chatbot popup should be **bigger**; (3) revisit the
Huddle and **apply a cleaner, more compact UI theme** (per a reference Kanban mockup — light/airy,
rounded white cards, tinted tag/priority pills, soft shadows, an azure-blue accent) while keeping all
existing components.

## Decision

- **Fly-in card** (`FlyInCard`): a sidebar card listing sprint tickets matched by a pure
  `matchFlyIn(summary)` (word-boundary `/\bfly[\s_-]*in\b/i`, so "fly in" / "fly-in" / "flyin" match
  but "butterfly inside" does not) + `selectFlyIns(issues)`. It reads the **already-loaded**
  `get_active_sprint` issues (the Dashboard computes the flat issue list once, reused for `sprintKeys`
  and `flyIns`) — no new tool, route, or fetch. Each row links the issue key + shows summary/assignee.
- **Bigger assistant**: `AssistantWidget` panel `w-[360px] → w-[440px]`; `ChatPanel` height
  `h-[480px] → h-[min(640px,calc(100vh-9rem))]` so it's larger but never taller than the viewport.
- **Theme refresh** (presentation-only): a token pass in `globals.css` — primary shifts from indigo
  to azure blue (`243 75% 59% → 228 80% 60%`, with the focus ring + dark-mode primary following),
  background goes cooler/lighter, borders softer, card radius `0.5rem → 0.75rem`, shadows softer and
  more diffuse. Because these are design tokens, the new look applies **app-wide** (the user asked to
  "apply the UI theme to our app"), and every shadcn card/button/input picks it up without markup
  changes. No DOM or copy changes, so the content/behavior test suite stays the regression guard.

## Consequences

- The daily Huddle gains a fly-in view for free from existing data; the assistant is roomier; and the
  whole app reads as the cleaner, rounder, bluer reference design without touching component structure.
- Token changes are subjective and not asserted by tests — they should be eyeballed in a live preview;
  the build + full suite confirm nothing structural regressed.
- The fly-in match is title-based (what the user asked) — if they later want a label/field instead,
  only `matchFlyIn` changes.

## Alternatives considered

- **A backend `get_fly_ins` tool / JQL search.** Unnecessary — the sprint issues are already loaded;
  a pure client-side filter is simpler and offline-testable.
- **Per-component restyle for the theme.** Higher effort and inconsistent; the shared token pass gives
  an app-wide result from one file, consistent with ADR-032's density pass.
