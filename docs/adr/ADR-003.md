# ADR-003: Dev-to-PO Ticket Linking Uses Generic Jira Issue Link, Not Parent/Child Hierarchy

## Status: Accepted

## Context

Spec §4.3 step 6 states:

> "Both tickets are created in Jira and linked to each other as **parent/child**."

In Jira, "parent/child" has two distinct meanings depending on instance configuration:

1. **Subtask hierarchy**: A "subtask" issue type whose parent is set via the `parent` field in the issue creation payload. Subtasks appear indented under their parent on the board. This requires the child issue type to be configured as a subtask type in the project's issue type scheme — not universally true for "Task" issue types on standard project configs.

2. **Epic link / Epic hierarchy** (Next-gen / Team-managed projects): In Jira's newer "next-gen" (team-managed) project type, parent/child is achieved by setting `parent.key` in the issue create body. In classic (company-managed) projects this field is rejected or ignored for non-subtask types.

3. **Portfolio / Advanced Roadmaps hierarchy**: Parent links managed by the Advanced Roadmaps add-on (not universally licensed).

None of these forms of parent/child is portable across Jira instance configurations. A Jira instance that uses classic company-managed projects, or one that has subtask types restricted, will reject the attempt to set a parent/child relationship between a "Story" (PO) and a "Task" (Dev). The resulting error would fail the entire `create_dev_ticket` call.

**Generic issue links** (`POST /rest/api/3/issueLink`) are portable. Any Jira instance with Jira Software supports generic issue links regardless of project type, issue type scheme, or licensed add-ons. Link types are configurable but the Jira default always includes at minimum "Relates to" (or the configurable equivalent). The link type is parameterized via `JIRA_LINK_TYPE` (default `Relates`) so operators can override it.

This is a **deliberate spec deviation** made on portability grounds.

## Decision

`create_dev_ticket` links the Dev task to its PO story using `POST /rest/api/3/issueLink` with `type.name` set to `JIRA_LINK_TYPE` (default: `"Relates"`). It does **not** attempt to set a `parent` field or use the Epic link mechanism.

Additionally, the link operation is **non-fatal**: if `POST /rest/api/3/issueLink` fails for any reason (link type not configured, permissions insufficient, Jira network error), `create_dev_ticket` returns the created Dev ticket successfully and includes `linkWarning: "<error message>"` in its output. The ticket is not rolled back.

The `JIRA_LINK_TYPE` environment variable allows operators to configure a more semantically precise link type (e.g., `"is implemented by"`, `"relates to"`) without code changes.

## Consequences

**Positive:**
- Works on any standard Jira Software instance without requiring specific project type, issue type scheme, or add-on configuration.
- Link failure does not prevent ticket creation — the PO and Dev tickets exist in Jira even if the link cannot be established.
- `JIRA_LINK_TYPE` provides a configuration escape hatch for teams with custom link type names.
- The Jira UI shows generic links clearly under the "Link" section of each ticket.

**Negative / trade-offs:**
- The relationship between the PO story and Dev task is visible in Jira as a generic link, not as a hierarchical parent/child. The Dev task does not appear as a child of the PO story on the board hierarchy view.
- If the Jira instance has `"Relates"` renamed or removed, the link will fail; `linkWarning` surfaces this without blocking ticket creation.
- Spec §4.3 says "parent/child" — this deviation must be documented (here and in `CONTRACTS.md`) so that the PO/SM stakeholders are aware the visual representation differs from the spec's wording.

**Spec deviation rationale summary:** The spec's intent ("link the tickets together") is fulfilled. The implementation ("generic issue link") differs from the spec's label ("parent/child") specifically to ensure the POC works across the widest range of Jira configurations. This is the correct trade-off for a POC aimed at rapid validation across heterogeneous team environments.

## Alternatives considered

### Alternative A: Use `parent` field in issue create payload (Jira next-gen style)

Set `parent.key = <poTicketKey>` in the `POST /rest/api/3/issue` body for the Dev task.

**Rejected because:** Works only on Jira team-managed (next-gen) projects. On classic (company-managed) projects the `parent` field is either ignored or rejected for non-subtask types. Fails silently or with a confusing error on the most common enterprise Jira setup.

### Alternative B: Create Dev ticket as a subtask

Set `issuetype.name = "Sub-task"` and `parent.key = <poTicketKey>`.

**Rejected because:** Subtask issue types are not universally configured. Many teams have disabled subtasks or use them exclusively for a different purpose (e.g., QA tasks). Creating all Dev tickets as subtasks would impose a Jira project configuration requirement on the user and would not match the spec's intent of a "Dev task" on the Dev board.

### Alternative C: Use Epic link custom field

Set `customfield_10014 = <poTicketKey>` (the standard Epic Link field).

**Rejected because:** Epic Link is a classic-project-only custom field that has been deprecated in newer Jira versions. Next-gen projects use `parent` instead. The field ID varies across Jira instances. Using it would create a dual portability problem (field name may differ, and it doesn't work on next-gen projects).

### Alternative D: Fail the entire create_dev_ticket call if linking fails

Throw an error and roll back (delete) the created Dev ticket if the link call fails.

**Rejected because:** Jira has no atomic create-and-link operation. A rollback (DELETE) would require another API call that could also fail, leaving the system in an uncertain state. The better UX is to surface the created ticket (the primary goal) and warn about the link failure, which the user can resolve manually in the Jira UI.
