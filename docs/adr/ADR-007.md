# ADR-007: Latest-active-sprint default with explicit sprint selection

## Status: Accepted (2026-06-11)

## Context

The v1.0 contract resolved "the active sprint" as `values[0]` of Jira's
`GET /board/{id}/sprint?state=active` response. Live testing on the team's real Dev board
showed that boards routinely run **multiple active sprints** (overlapping sprints,
carry-over sprints not yet closed), and Jira returns them oldest-first — so the dashboard
pinned itself to the stalest sprint. The product owner asked for (a) the latest active
sprint by default and (b) a UI dropdown to switch between active sprints.

## Decision

1. `get_active_sprint` and `get_daily_huddle` fetch **all** active sprints, sort
   latest-first (descending `startDate`, ISO string compare; `null` startDate last; ties
   by descending `id`), and default-select the **first** — the most recently started
   sprint, which matches the team's intuition of "the current sprint".
2. Both tools accept an optional **`sprintId`** input. A `sprintId` that is not an active
   sprint on the board is an `UPSTREAM` error (active-only keeps the POC read paths on
   the sprint Agile API endpoints and the dropdown honest — closed/future sprints are
   Phase 3 reporting territory).
3. `get_active_sprint` output gains **`activeSprints: ActiveSprintRef[]`** (latest-first)
   so the dashboard can render the selector from data it already has — no extra tool, no
   second round trip.
4. The Dashboard owns `selectedSprintId`; the board AND the huddle digest follow it.

## Consequences

- The "wrong sprint" bug disappears without configuration; teams with a single active
  sprint see no UI change (selector hidden for 0–1 active sprints).
- `activeSprints` slightly widens the payload of every board load (a few hundred bytes).
- Copilot users get the same fix transparently; they can say "use sprint 1234" because
  the tool schema now documents `sprintId`.
- Tests must cover the sort (startDate ordering, null handling, id tiebreak) and the
  invalid-`sprintId` error.

## Alternatives considered

- **`values[last]`** — Jira's ordering is not contractually guaranteed; sorting by
  `startDate` is explicit and testable.
- **A separate `list_active_sprints` tool** — second round trip and more surface; the
  list rides along on `get_active_sprint` instead.
- **Allowing any sprint id (incl. closed/future)** — turns the board into a sprint
  history browser; that is the Phase 3 reports feature, not the live dashboard.
- **Server-side sticky selection** — the bridges are stateless by design (ADR-001);
  selection state belongs to the UI.
