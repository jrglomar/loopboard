# ADR-021: Dev ticket for an existing PO story + velocity that includes active sprints

## Status: Accepted (2026-06-17)

## Context

Two user requests against the real board:

1. **"From a PO board sprint, create/generate a Dev ticket linked to that existing PO
   ticket, and target a Dev board sprint."** The existing `TicketGen` only creates a *new*
   PO Story + Dev Task **pair**. There was no flow to attach a *new* Dev Task to an
   **already-existing** PO Story and drop it into a chosen Dev sprint.

2. **"In Reports, velocity is getting the wrong sprints ŌĆö it should fetch the latest even if
   it's an active sprint."** `get_velocity` pooled only **closed** sprints. This team's Dev
   board rarely *formally closes* sprints (Ōēł30 sit in `active` indefinitely), so the closed
   pool is sparse/old. With `beforeSprintId` (Reports context), the velocity window returned
   the few most-recent *closed* sprints ŌĆö which can be months old ŌĆö while the recent delivered
   work lived in never-closed `active` sprints that were excluded. Hence "wrong sprints."

## Decision

### 1. `LinkDevTicketCard` (frontend only ŌĆö no backend change)

`create_dev_ticket` already accepts `{ summary, description, linkedPoTicketKey?, sprintId? }`
(link + add-to-sprint are non-fatal, returning `linkWarning`/`sprintWarning`). So this is a
pure UI composition:

- **PO source:** a PO **board** sprint select (`list_sprints` on `boards.po.id`,
  active+future+closed) ŌåÆ list that sprint's tickets via `get_active_sprint(po.id, sprintId)`
  (all buckets) ŌåÆ a **PO ticket** select (`KEY ŌĆö summary`).
- **Dev draft:** `summary` + `description`, pre-seeded from the chosen PO story and editable.
  Optional **"Generate with AI"** reuses `POST /api/ai/draft-tickets` seeded with the PO
  story and takes the `.dev` half; when AI is off it pre-seeds from the deterministic
  template ŌĆö same dual-mode contract as TicketGen.
- **Dev target:** a Dev **board** sprint select (`list_sprints` on `boards.dev.id`).
- **Create:** `create_dev_ticket({ summary, description, linkedPoTicketKey: <PO key>,
  sprintId: <dev sprint> })`. Success shows the new Dev keyŌåÆlink and "linked to <PO key>".

Kept as a **separate card** (not folded into TicketGen) because the intent differs: TicketGen
authors a brand-new pair; this card extends an existing PO story. Overloading TicketGen would
muddy both flows.

### 2. `get_velocity` gains `includeActive`

- New input `includeActive?: boolean` (**default `false`** ŌĆö preserves the documented
  closed-only behavior for Copilot/other callers and existing tests).
- When `true`, the candidate pool is `closed` **+** `active` sprints (never `future`), sorted
  latest-first by `completeDate` (fallback `endDate`, since active sprints have no
  `completeDate`). The `beforeSprintId` filter is unchanged, so the selected/in-progress
  sprint is still excluded and not double-counted.
- The UI (`useVelocity` in Reports and the capacity plotter) passes `includeActive: true`, so
  the chart reflects the **latest** sprints "even if active."

## Consequences

- Velocity on boards that don't close sprints now reflects recent throughput instead of stale
  closed sprints. Boards that *do* close sprints are unaffected when `includeActive` is true
  (closed sprints are a subset of the pool and sort by the same recency key).
- An `active` sprint that is genuinely *in progress* (incomplete) could enter the pool, but
  only when it is **not** the selected sprint and is strictly *before* the anchor ŌĆö i.e. a
  prior, never-closed sprint, which is exactly the delivered work we want. The current sprint
  (the one being reported on) remains excluded by `beforeSprintId`.
- No new MCP tool; `create_dev_ticket` and `get_active_sprint` are reused as-is. The only tool
  IO change is the additive `includeActive` flag.

## Alternatives considered

- **Make `includeActive` the default (true).** Rejected: changes the documented "last N
  *closed* sprints" semantics for all callers (incl. Copilot) and breaks existing tests for no
  benefit ŌĆö the UI opt-in fixes the user-visible problem precisely.
- **Count the current in-progress sprint in velocity.** Rejected: partial/incomplete points
  understate a "completed velocity" trend; the report already shows the selected sprint's
  progress separately.
- **Extend TicketGen to also link to an existing PO.** Rejected: conflates "new pair" with
  "extend existing"; a focused card is clearer and independently testable.
- **A new `create_linked_dev_ticket` tool.** Unnecessary ŌĆö `create_dev_ticket` already does
  exactly this; a new tool would duplicate surface area.
