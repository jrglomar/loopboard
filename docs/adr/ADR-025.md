# ADR-025: Draft the Dev task from the PO story's description on the Linking page

## Status: Accepted (2026-06-24)

## Context

The Linking page bulk-creates a Dev task for each selected PO story. Until now its AI plan
(`POST /api/ai/plan-dev-tickets`) was given only each PO's one-line **`summary`** — because that is
all the sprint board read (`get_active_sprint` → `IssueSummary`) carries. PO stories in real use
have a substantive **description** (user story, acceptance criteria, scope), and that is exactly the
context a good Dev task should be derived from. Drafting from the title alone produced generic Dev
descriptions that ignored the PO's actual ACs.

Two relevant facts about the existing surface:

- The AI side was **already built for this**: `plan-dev-tickets` accepts `poStories[].description?`
  and `draftService.planDevTickets` already injects a `Details:` line per story into the prompt
  (v1.11). The field was simply never populated by the frontend.
- `jiraClient.getIssue(key)` already fetches an issue and flattens its description to plain text via
  `adfToText`. There was no **batch** read for descriptions, and the board read deliberately omits
  description (it would bloat every sprint/board fetch).

## Decision

Add one small read tool and wire the frontend to use it — no change to the AI contract surface.

### `get_issue_descriptions` (READ)

- New MCP tool `get_issue_descriptions({ keys: string[] (1..50) })` →
  `{ descriptions: Record<string, string> }`.
- For each key, fetch the issue and return its description as plain text (reuses `getIssue` →
  `adfToText`). Fetches run in parallel; a missing/unreadable key contributes `""` (non-fatal),
  mirroring `get_linked_issues`' per-key resilience. Read-only; **no new jiraClient surface**.

### Linking page wiring (frontend-only)

- On **Generate**, the page first calls `get_issue_descriptions` for the **selected** PO keys
  (not the whole sprint — leaner), then sends `poStories: [{ key, summary, description? }]` to
  `plan-dev-tickets`. Each description is **capped at ~4000 chars** to bound prompt size; empty
  descriptions are omitted.
- The fetched descriptions are kept in a `descMap` so per-draft **Regenerate** (v1.12) also passes
  the PO description as context.
- When **AI is off**, the deterministic fallback prepends a `## Source PO story` block built from
  the description, so the PO context still flows into the Dev draft.

## Consequences

- Dev drafts are now grounded in the PO's real description/ACs instead of just the title — the
  central ask of this enhancement.
- Cost is one extra batch read per Generate (selected keys only), behind the existing "Planning…"
  spinner. No change to `create_dev_ticket` or `plan-dev-tickets` IO; the only new surface is the
  read tool. jira tools 19 → 20.
- Copilot gains a batch description read it can use for its own grooming flows.

## Alternatives considered

- **Add `description` to `get_active_sprint`'s `IssueSummary`.** Rejected — it would enlarge every
  board/Dashboard read with long ADF text that those surfaces never use. A targeted batch read for
  the selected POs is leaner and keeps the board read cheap.
- **Extend `get_linked_issues` to also return descriptions.** The page already calls it for every
  PO ticket, so this would be zero extra round-trips — but it would fetch descriptions for *all*
  tickets in the sprint (most never selected) and overload a tool named for links. Fetching only
  the selected keys on Generate is more proportionate.
- **Fetch descriptions server-side inside `plan-dev-tickets`.** Rejected — it would couple the
  (pure) AI endpoint to Jira reads and complicate its tests. Keeping the read in a read tool and
  letting the frontend orchestrate matches how the page already composes `get_linked_issues` +
  `plan-dev-tickets`.
- **Per-key `get_ticket` calls from the client (up to 20).** Chattier than one batch call and
  returns far more than the description. A focused batch tool is cleaner.
