# ADR-080: Draft card ticket actions + breakdown + capacity-source transparency (v1.69)

## Status: Accepted (2026-07-21) — implemented in v1.69

## Context

Live use of the v1.68 Draft Capacity Plan card (ADR-079) surfaced three gaps:

1. **The chips dropped the usual per-ticket actions.** The Assign Tickets table (ADR-026/047)
   lets the user edit points inline, change status, and move a ticket to another sprint — the
   draft card's chips had none of that, so grooming bounced between the card and the table. The
   user also asked for a NEW action neither surface had: **rename** (edit the summary).
2. **No way to break a too-big story down.** Capacity matching makes oversized stories obvious
   (a 13-pointer fits nobody); the natural next step — split it into smaller PO stories in the
   same sprint — required leaving the card for the ticket generator, once per piece.
3. **Live bug report: "capacity for all members shows 8."** Every developer tile showed exactly
   `requiredPoints` — i.e. the leaves read for the paired Dev sprint came back EMPTY and the card
   rendered full capacity with total confidence. Three distinct causes produce this identical
   silent state: (a) the auto-paired Dev sprint isn't where the team plotted (the Offset Tracker
   defaults to the ACTIVE sprint; pairing prefers FUTURE on ties); (b) v1.68 **persisted the
   auto-pairing as a side effect of any draft mutation**, so one wrong guess stuck as if chosen;
   (c) a per-user store-scope mismatch (ADR-056 storeUserId) makes another scope's leaves
   invisible. The card gave no signal distinguishing "no leaves exist" from "looking in the
   wrong place."

## Decision

All react-app; **zero backend changes** — `update_ticket` has accepted `summary` since v1.0
(§4.5) and `create_po_ticket` has accepted `storyPoints` + `sprintId` since v1.4/ADR-042.

**1. Shared ticket action cells (`components/ticketCells.tsx`).** `PointsCell`, `StatusCell`,
`MoveSprintCell` move verbatim out of `AssignmentList.tsx`; a new `SummaryCell` adds inline
rename (collapsed text + "Rename" affordance → input, Enter/save Esc/cancel, 1–255 chars, via a
new `updateTicketSummary` client beside `updateTicketPoints`). `PointsCell` gains an optional
`onSaved` callback. Both surfaces import the cells: the draft card's chips get an inline points
input plus a per-chip "Edit" expander (rename / status / move / break down), and the Assign
Tickets table keeps its exact UI while gaining rename and — via `onSaved` → sprint refetch — a
fix for its pre-existing stale points summary after an inline points edit. In the draft card
every field edit refetches the PO sprint so tile totals and capacity deltas stay truthful. A
move initiated FROM the draft card also removes the ticket's draft entry in the same action — a
deliberate move must not leave a "Needs attention" stale row behind. The expanded editor carries
the note "These edits change the real Jira ticket."; the card's "Draft only" badge continues to
mean ASSIGNMENTS (`assign_issue` is still never called — unchanged ADR-079 invariant).

**2. Breakdown dialog.** From a chip's editor, "Break down" opens a dialog seeded with the
original's summary/points: 2–6 rows of (summary, points), created SEQUENTIALLY in the same PO
sprint via `create_po_ticket`, per-row success/error with succeeded rows locked (the Linking
page's resilient-bulk pattern — a retry never duplicates), optional "set the original's points
to 0 after creating" via `update_ticket`, and a sprint refetch on success so the new stories
appear as unplanned chips ready to draft. No PO↔PO issue link: the configured `JIRA_LINK_TYPE`
encodes PO→Dev dependency semantics (§4.2), so provenance is recorded in each new story's
description ("Broken down from <KEY>") instead of overloading the link type.

**3. Capacity-source transparency + non-sticky pairing.** (a) Under the Dev-sprint select the
card shows "N leave/offset day(s) found across M member(s)" when the paired sprint's leaves read
returns data, and a warning-toned hint when it returns empty ("No leaves or offsets recorded
under this Dev sprint — pick the sprint where the team plotted them (Dev-board Planning or the
Offset Tracker)."). (b) Auto-pairing is EPHEMERAL: draft mutations pass through the STORED
`devSprintId` (null until chosen) instead of the effective one; only an explicit pick in the
select persists. An "Auto-paired" label shows when the default is in use; "Reset to auto" clears
a stored choice (persists `devSprintId: null`). (c) While the Dev sprint's leaves are loading,
tiles show a loading state for capacity instead of flashing full-capacity numbers.

## Consequences

- The draft card becomes the PO's one grooming surface: draft, re-point, rename, transition,
  move, and split — with the assignment side still strictly draft-only.
- The "everyone shows 8" state is now self-diagnosable in the UI, whichever of the three causes
  produced it — and the sticky-wrong-pairing cause is eliminated outright. A store-scope
  mismatch (ADR-056) still yields empty reads by design; the warning now says where to look.
- Legacy drafts that already captured an auto-paired `devSprintId` keep honoring it (it is
  indistinguishable from a chosen one); "Reset to auto" is the one-click recovery.
- `AssignmentList` behavior is preserved through the extraction (its test suite runs unchanged
  except for the new rename cases), and its points summary no longer goes stale after inline
  edits.
- Tests: react-app 1031 → **1066** (+35: ticketCells 13 — SummaryCell save/cancel/Escape/
  empty-and-whitespace-revert/error/disabled-while-saving + PointsCell onSaved on-success/
  not-on-failure/optional; DraftPlanCard +18 — expander, rename/status flows, move removes the
  draft entry with the STORED devSprintId, non-sticky pairing persists null, Reset-to-auto
  shown/hidden + persists null, Auto-paired label, leaves-found/empty-warning/loading indicator,
  breakdown happy path + partial-failure lock (retry never duplicates) + zero-original checkbox,
  drag affordance disabled while the editor is expanded; AssignmentList +4 — rename
  open/save-refetch/cancel/error-revert; Planning assertions extended in place); mcp-jira/
  mcp-github unchanged (644/57) — total **1,767**; smoke unchanged **60/60**.

## Alternatives considered

- **Duplicating the action cells inside DraftPlanCard** — rejected; the repo's core rule is one
  implementation consumed by every surface (the tool-registry pattern applied to UI cells).
- **A modal editor per ticket instead of an inline expander** — rejected; the table's existing
  cells are inline and the expander matches `StatusCell`'s collapsed→expanded precedent
  (ADR-026), keeping both surfaces idiomatically identical.
- **Linking broken-down stories to the original with `JIRA_LINK_TYPE`** — rejected; that type's
  direction is defined as PO-depends-on-Dev (§4.2, ADR-046); reusing it PO↔PO would render
  misleading dependency text in Jira. Description provenance is honest and cheap.
- **Auto-picking the Dev sprint that HAS leaves** ("smart" pairing) — rejected; data-dependent
  magic would flip the pairing when someone plots a day off, and hides the real problem the
  warning now surfaces. Deterministic date pairing + visible evidence beats guessing.
- **Deleting a stale-looking stored `devSprintId` on load** — rejected; a stored choice may be
  legitimate, and silently un-choosing it repeats the exact class of invisible behavior this
  phase removes. The label + reset button keep the user in charge.
