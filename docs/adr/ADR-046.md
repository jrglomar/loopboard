# ADR-046: PO→Dev dependency link, assignable Dev create, and point-driven breakdown

## Status: Accepted (2026-07-01) — Phase A + Phase B implemented

## Context

The Linking feature creates one Dev task per PO story and links them with the generic **"Relates"**
type. The team wants the link to express a real **dependency**, wants a PO story to be **split into
multiple Dev tasks** (by the point scale) that can go to **different developers**, and therefore needs to
create a Dev task **already assigned**.

## Decision

### Phase A — backend (this ADR, implemented)
- **Dependency link.** `JIRA_LINK_TYPE` default changes `"Relates"` → **`"Depends"`**. The existing
  `create_dev_ticket` link payload is unchanged (`inwardIssue: <dev>, outwardIssue: <po>`); with an
  asymmetric `Depends` type the OUTWARD issue is the depender, so this reads **"PO story depends on Dev
  task"** — the confirmed direction. The type must exist in the Jira instance (else the link 400s; the
  failure is already non-fatal → `linkWarning`). Configurable, so a differently-named type can be set.
- **Assignable Dev create.** `create_dev_ticket` gains optional `assigneeAccountId`. After the create +
  link + sprint steps, when present it calls the shared `assignIssue` (the PUT-assignee already behind
  `assign_issue`) **non-fatally** → `assignWarning` on failure; the output echoes `assigneeAccountId`.
  This lets the Linking breakdown create each Dev task pre-assigned to a (possibly different) developer.

### Phase B — Linking breakdown (frontend, implemented, same ADR)
A PO story's points drive how many Dev tasks it becomes: a pure `suggestBreakdown(total)` (`lib/points.ts`)
splits a total that is not a single valid estimate into a balanced pair of allowed values (4→[2,2],
6→[3,3], 8→[3,5], 5→[5]), allowed set `{0.2,0.3,0.5,1,2,3,5,7}`. The Linking plan is grouped by PO with
1–2 editable Dev-task rows (`PlanRow` = the AI draft shape + a stable `id`, per-task `storyPoints`, and
`assigneeAccountId`), each with a free-numeric **Points** and an **Assignee** picker (Dev-board roster via
`useTeamMembers`), plus per-group add/remove. "Create all" loops `createLinkedDevTicket` with each row's
points + `assigneeAccountId` (the Phase-A `create_dev_ticket` fields). No new tool; IO shapes unchanged.

## Consequences

- The PO↔Dev relationship now models delivery ("PO depends on Dev"), and a Dev task can be born assigned —
  the two backend seams the breakdown needs. No new tool; jira tools stay 36; smoke unchanged (IO shapes
  unchanged apart from two optional fields). Assignment/link failures never fail the create (warnings).
- The `Depends` type is instance-specific — if a Jira lacks it, set `JIRA_LINK_TYPE` to an existing type.
  The exact inward/outward label direction should be eyeballed once live (swap the two keys if reversed).

## Alternatives considered

- **Assign in a second `assign_issue` call from the frontend** instead of inside `create_dev_ticket`.
  Rejected — an extra round-trip per Dev task and a partially-created state on failure; folding it in
  (non-fatal) keeps the Linking "create all" loop simple.
- **Hardcode "Blocks".** Rejected — the user chose `Depends`; keeping it configurable covers both.
