# ADR-062: The board selector + offset policy read the PER-USER context, not global `/api/health` (v1.51)

## Status: Accepted (2026-07-13) — implemented in v1.51

## Context

Multi-tenancy (ADR-055/056) made every tool, store, and AI call resolve through the signed-in user's
merged config (`.env` ← admin global ← shared-credential source ← the user's own overrides), request-scoped
via `AsyncLocalStorage`. The intent: an admin (or the user) can point one user at a different board than the
`.env` default, and the whole app follows.

It didn't follow. A user borrowed the shared connection and set a **different PO/Dev board** in their
per-user override, but the dashboard kept showing the `.env` board.

**Root cause.** The React board selector reads boards through `useBoards()` in `lib/boards.ts`, and the
offset views read the policy through `usePolicy()`. Both hooks fetched **`GET /api/health`** — the GLOBAL,
unauthenticated health endpoint, which computes `getProjects()` / `getOffsetPolicy()` with **no** ALS
context and therefore always returns the raw `.env` boards/policy. The per-user override was resolved
correctly on the backend for every tool call, but the one thing that chooses *which board the UI shows* was
bypassing it. So the override silently had no visible effect.

## Decision

- **Serve boards + policy from the per-user context endpoint.** `GET /api/me/context` now computes both
  inside the caller's request context: `resolveUser(userId)` → `runWithUser({ userId, config, storeUserId,
  canWriteJira }, () => ({ boards: getProjects(), policy: getOffsetPolicy() }))`. It already returned
  `boards`; it now also returns `policy = { requiredPoints, offsetThreshold }`. When a user can't be
  resolved, it falls back to the global default (unchanged behaviour for that edge).
- **The hooks read `AuthContext`, not `/api/health`.** `useBoards()` and `usePolicy()` now read
  `context.boards` / `context.policy` from `AuthContext` (which already fetched `/api/me/context` once for
  the gate) — a single source of truth, no extra round-trip. The pure normalizer was extracted as the
  exported `normalizeBoards(raw)` (handles per-side arrays, the legacy single-object shape, and
  invalid/empty sides → `null`) and unit-tested directly. `usePolicy()` returns `context.policy` or the
  `{ requiredPoints: 8, offsetThreshold: 2 }` default until the context loads.
- **`/api/health` stays global and keyless.** It still returns the `.env` boards/policy and is used only by
  the smoke check and health probes — never again as the UI's board source.

## Consequences

- A per-user (or admin-set) board/policy override now actually reaches the dashboard: the board selector and
  offset views reflect the signed-in user's effective config, matching every other per-user surface.
- One fewer client fetch: the board/policy no longer round-trips to `/api/health`; both come from the
  context the gate already loaded.
- **Board switching requires overriding the board _id_.** `getProjects()` keys off `JIRA_DEV_BOARD_ID` /
  `JIRA_PO_BOARD_ID`; overriding only the project **key** changes labels, not which board's data is fetched.
  Documented in CONTRACTS §9.5 so the override that failed here is set correctly next time.
- Related latent gap (NOT fixed here, flagged as follow-up): `aiClient.getAiStatus` still reads
  `/api/health .ai` (global), so the assistant can show "AI disabled" for a user who has their own AI token
  but no global `.env` AI. Same class of bug; separate change.
- Gate: typecheck + build clean, 1,355 tests (github 57, mcp-jira 535, react-app 763), smoke 53/53.
  Pill → 1.51.0. The +3 mcp-jira tests (delegation.test.ts) prove a per-user `JIRA_DEV_BOARD_ID`
  override reaches `getProjects()` inside `runWithUser` (the exact `/api/me/context` computation) and
  document the **board-ID-vs-project-key trap** (a key-only override is cosmetic — same id → same board).
- **Field caveat found while verifying:** the reporting user's per-user override had never actually been
  saved — the Admin "Manage" panel has three separate forms with independent submit buttons (Access,
  Password, **Board & env overrides → "Save user config"**); saving Access does not save the board
  override. This read-path fix is necessary but only takes effect once the override is actually persisted
  AND the board **id** (not just the key) is set.

## Alternatives considered

- **Keep `/api/health` but make it context-aware.** `/api/health` is intentionally global and
  unauthenticated (keyless smoke, liveness) — scoping it to a session would break that contract. The
  per-user question already has a per-user endpoint (`/api/me/context`); boards/policy belong there.
- **A dedicated `GET /api/me/boards`.** Redundant — `AuthContext` already fetches `/api/me/context` for the
  gate; adding boards/policy to it costs nothing and keeps one source of truth.
- **Leave `usePolicy` on `/api/health`, move only boards.** Inconsistent (policy is per-user too, ADR-038)
  and would reintroduce the same bug class for the offset threshold; both moved together.
