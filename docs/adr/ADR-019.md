# ADR-019: Curated per-board team roster (seeded from recent sprints) instead of the org-wide assignable list

## Status: Accepted (2026-06-15)

## Context

v1.7 populated the leaves plotter and the ticket-assignment dropdown from
`get_assignable_users` → `GET /rest/api/3/user/assignable/search?project=KEY`. Live testing
showed this returns **people who aren't on the team** — because that endpoint returns
*everyone with the "Assignable User" permission* on the project, which on the team's cloud
site is org-wide groups (50+ users). The product owner asked for a **fixed list they can
add to / remove from**, suggesting it be **seeded from the usual members assigned in
previous sprints** — applied to both leaves plotting and assignment.

**Jira-expert finding (REST API):** there is no portable "team" primitive.
`user/assignable/search` is permission-scoped, not a team. **Project roles**
(`/project/{key}/role/{id}` → actors) and **groups** can act as a team but are
instance-specific, frequently unmaintained, and need elevated permissions. The
authoritative, portable signal for "who actually works this board" is the **assignee on
recent sprint issues** (`issue.fields.assignee` → `{ accountId, displayName }`) — and
`accountId` is exactly what `PUT .../assignee` requires.

## Decision

1. **Maintain a curated team roster per board**, persisted to a bridge-side JSON file
   (`JIRA_TEAM_FILE`, default `<mcp-jira pkg>/.loopboard-team.json`, git-ignored — same store
   pattern as leaves, ADR-016). Keyed by boardId (Dev and PO teams can differ). Members are
   `{ accountId, displayName }`.
2. **Seed from recent-sprint assignees.** `get_recent_assignees` scans the last N sprints'
   issues for distinct assignees (with ticket counts) — the "usual members." First run shows
   a one-click "set up your team from recent sprints"; thereafter "Add from recent sprints"
   refreshes suggestions.
3. **Editable.** `get_team_members` / `set_team_members` (replace-the-list = add/remove). An
   optional add-by-name typeahead may use `get_assignable_users?query=` for someone not seen
   recently — that's a scoped search, not the unfiltered org list.
4. **Both consumers use the curated team.** The leaves plotter and the assignment dropdown
   roster from `get_team_members(boardId)`, not `get_assignable_users`. `assign_issue` still
   assigns by `accountId`; the dropdown pre-selects the current assignee via the new
   `IssueSummary.assigneeAccountId`, and shows an off-roster current assignee as a disabled
   "(not on team)" option so it isn't lost.
5. **Capture `assigneeAccountId`** on `IssueSummary` (from `fields.assignee.accountId`) —
   needed for both recent-assignee derivation and correct pre-selection.

## Consequences

- The roster is the team, not the org — the reported "strangers" problem is fixed.
- Second small stateful store (after leaves); host-local JSON, git-ignored. Acceptable for a
  POC; a real DB if multi-tenant later.
- The roster is per board and seeded from data, so it works out-of-the-box and stays
  editable. Renamed/departed members are pruned by editing; new joiners are added from recent
  sprints or by search.
- `get_assignable_users` stays in the API (Copilot/search) but is no longer the UI roster.

## Alternatives considered

- **Jira project roles / groups as the team** — the "correct" Jira concept, but
  instance-specific, often unmaintained, and needs elevated perms; not portable for a POC.
- **Atlassian Teams API** — separate product, not cleanly tied to a Jira board; overkill.
- **Keep `assignable/search` but filter client-side** — no reliable team signal to filter on.
- **Recent-assignees with no persistence (compute every time)** — can't add someone who
  hasn't been on a recent sprint, and can't remove a stale member; persistence gives the
  "fixed list you can add/remove" the owner asked for. The derived list seeds it.
