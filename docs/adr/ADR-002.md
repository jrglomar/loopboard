# ADR-002: In-UI Chat Panel is a Deterministic Command Router, Not a Direct Claude API Call

## Status: Accepted

## Context

The React dashboard includes a `ChatPanel` component that allows users to interact with sprint and ticket data from the browser UI. A naive implementation would forward the user's natural-language message to the Claude API (via `@anthropic-ai/sdk` or a proxy endpoint) and let Claude decide which tools to call.

This is explicitly prohibited by spec §8:

> **Why GitHub Copilot instead of a direct Claude API integration?**  
> "The team already has GitHub Copilot licenses. Using Copilot avoids additional subscription cost, keeps the toolchain inside the existing GitHub ecosystem, and allows developers to interact with the AI companion without leaving VS Code."

Calling Claude directly from the React app would:
- Require an Anthropic API key loaded into the browser or a proxy server (neither exists in this architecture).
- Introduce a direct dependency on the Anthropic API that the spec explicitly says is not part of this system.
- Add cost that the spec says is avoided by using Copilot licenses.
- Create a second AI reasoning path that could behave differently from the Copilot path, confusing users about what the system can do.

At the same time, the spec calls for a chat panel that lets users "ask about the sprint inside the UI" (spec §7, Phase 2). This capability must be delivered without a direct LLM call.

## Decision

The `ChatPanel` is implemented as a **deterministic command router** (`src/lib/chatRouter.ts`):

- A finite set of commands is recognized and mapped directly to HTTP bridge calls:
  - `help` — display command reference
  - `huddle` — call `get_daily_huddle`, render result
  - `sprint` — call `get_active_sprint`, render sprint summary card
  - `ticket <KEY>` — call `get_ticket`, render ticket card
  - `enhance <KEY> [notes]` — call `get_ticket` then `update_ticket` with notes
  - `create <description>` — call `create_po_ticket` + `create_dev_ticket`, render pair
  - `prs` — call `list_prs`, render PR list
  - `link pr <n> [KEY]` — call `link_pr_to_ticket`

- Any input that does not match a known command returns a help message that explicitly explains:
  - This panel handles structured sprint commands.
  - Free-form natural language questions about the sprint belong in GitHub Copilot Chat in VS Code.
  - This is a deliberate scope decision, not a bug.

- `chatRouter.ts` is a **pure function** (`router(input: string): RouterAction`) with no side effects. It parses input, returns an action descriptor, and is fully unit-testable without network access.

- The frontend `ChatPanel` component calls `router()`, then executes the resulting action via `mcpClient.ts`, and renders the response as a message card.

## Consequences

**Positive:**
- No Anthropic API key required in the browser or in any server.
- The chat panel is fully deterministic and unit-testable.
- The command set is documented and finite — users know exactly what works.
- No additional subscription cost introduced.
- The boundary between "structured sprint UI" and "free-form AI assistant in Copilot" is explicit and user-visible.

**Negative / trade-offs:**
- Users cannot ask open-ended questions in the chat panel ("Why is DEV-99 blocked?"). They must use Copilot for that.
- New commands require code changes (not just prompt tuning).
- The chat panel may feel less "AI-like" than users expect from an "AI companion" product — this must be managed with clear UX copy.

**UX requirement:** The panel must display a visible label (e.g., "Sprint commands — type `help`") and the unknown-command handler must proactively direct users to Copilot for natural-language queries. This is not optional — it prevents the panel from feeling broken.

## Alternatives considered

### Alternative A: Direct Claude API call from the browser

Call `https://api.anthropic.com` from the React app, providing an API key via environment variable.

**Rejected because:** (a) violates spec §8's decision to not use the Claude API directly; (b) exposing an Anthropic API key in a Vite app (even via `VITE_*` env var) means it is visible in the browser bundle — a security violation for any key with spend permissions; (c) introduces a cost center the spec explicitly avoids.

### Alternative B: Proxy server that holds the API key and forwards to Claude

A small backend proxy (e.g., Express endpoint on :4003) holds the Anthropic API key server-side and forwards prompts to Claude, returning natural language responses.

**Rejected because:** This still adds a direct Claude API dependency that spec §8 rules out. It also adds a third server process to manage alongside the two MCP bridges, and requires a new API key provisioning step — all overhead the spec explicitly avoids by using Copilot.

### Alternative C: Call the stdio MCP server's registered prompts from the browser

Have the browser talk to a prompt-execution endpoint on the HTTP bridge that invokes Claude via the MCP prompts registered in `src/lib/prompts.ts`.

**Rejected because:** MCP prompts are instructions *to* Claude (the external AI reasoning engine); they cannot be executed server-side without a running Claude process. The HTTP bridge has no access to a Claude model — it only calls Jira/GitHub REST APIs. This conflates two different layers of the architecture.
