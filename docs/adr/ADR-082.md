# ADR-082: Streaming Ask assistant ‚Äî SSE + markdown + rich cards + tool transparency (v1.71)

## Status: Accepted (2026-07-22) ‚Äî implemented in v1.71

## Context

The in-app assistant (`ChatPanel` ‚Üí `askService.askAssistant`, ADR-029/030/050) is capable but feels
thin next to the reasoning experience it evokes:

1. **No streaming.** The agentic loop runs several tool turns and only then returns one JSON envelope.
   A multi-tool answer leaves the panel showing a spinner for seconds with nothing to read.
2. **No markdown.** Assistant answers render through a `split("\n")` loop ‚Äî bold, lists, tables, and
   inline links all come out as literal text, even though the model emits markdown.
3. **The tool trace is discarded.** `askAssistant` already returns `toolsUsed`, but the UI drops it, so
   the user can't see what the assistant looked at ‚Äî the answer reads as an unsourced assertion.
4. **Free-form answers are prose-only.** The deterministic commands (`sprint`, `ticket ‚Ä¶`, `huddle`)
   render rich result cards; an equivalent AI answer about the same ticket renders as a paragraph.

The user asked to "deepen the chat assistant." The goal is to make it feel alive and trustworthy
without adding new AI capabilities or new surfaces ‚Äî one panel, the existing Ask path.

## Decision

Four coordinated changes, all riding on streaming. Contract updated first (`CONTRACTS.md` ¬ß4.9).

**1. Streaming over Server-Sent Events.** A new `POST /api/ai/ask/stream` runs the *same* loop but
emits SSE frames ‚Äî `step` (a read-tool batch is about to run), `delta` (a chunk of the final answer),
`cards`, `proposed` (a write the model wants ‚Äî stream then ends, confirmed in the modal, never
executed), `done`, `error`. Pre-flush failures (validation / provider-config / `AI_UNAVAILABLE`) use
the ordinary JSON envelope + HTTP status, identical to `/api/ai/ask`, so the client can fall back. The
non-streaming `/api/ai/ask` stays for that fallback and for tests.

**2. `chatWithToolsStream` on the `AiProvider` port.** Same return contract as `chatWithTools`, plus an
`onDelta(chunk)` callback for streamed final-answer text. **Anthropic** implements true token streaming
via `messages.stream` (forward `text` deltas, collect `tool_use` blocks, resolve to the same
`final`/`tool_calls` result). **GitHub Models** uses a **buffered fallback** ‚Äî it delegates to the
existing `chatWithTools` and emits the final text as a single `delta`. GitHub Models is the secondary
provider here and streaming its fragmented `tool_calls` deltas is fiddly and error-prone; the buffered
path still shows live `step`s and keeps both providers first-class without that risk (YAGNI).

**3. Deterministic rich cards.** The loop already runs the read tools in-process, so their results are
in hand. For a **card-able** subset whose output shape matches an existing renderer
(`get_ticket`‚Üí`ticket`, `get_active_sprint`‚Üí`sprint`, `get_daily_huddle`‚Üí`huddle`) we capture
`AskCard = { kind, data }` (‚â§3, most-recent distinct) onto the response. **No extra model call.** The
ChatPanel renders them with the same result-card components the deterministic commands use. PR cards
are deferred ‚Äî the jira `get_pull_requests` output shape differs from the `pr-list` renderer, so mapping
it would misrender; it's added only if the shapes are reconciled.

**4. Markdown + tool transparency in the UI.** A small `<Markdown>` component converts the (possibly
partial, streaming) answer with `marked`, sanitizes with the **already-present** DOMPurify, and renders
‚Äî the same markup‚Üísanitize‚Üírender pattern `RichTextEditor` uses (ADR-051). While streaming, `step`
events drive a live "Looking at‚Ä¶" line; on `done`, a muted "Looked at: active sprint ¬∑ impediments"
trace (a `toolName`‚Üílabel map) persists under the answer.

`marked` is the one new dependency ‚Äî tiny, zero-dependency, and it reuses DOMPurify rather than pulling
in `react-markdown`'s remark tree.

## Consequences

- The assistant streams token-by-token (Anthropic), renders real markdown, shows its sources, and
  attaches ticket/sprint/huddle cards ‚Äî a materially better experience from the same loop and data.
- The write-proposal safety model (ADR-030) is unchanged: a write still short-circuits the loop into a
  `proposed` event and is only executed after the modal confirmation. The AI never mutates Jira.
- Two Ask code paths now exist (streaming + JSON). They share the loop body; the JSON path is the
  automatic fallback when the stream fails (old browsers, buffering proxies), so no environment loses
  the assistant.
- Tests stay keyless/offline: providers are mocked, the SSE parser is unit-tested against a fake stream,
  and `Markdown` is tested for both rendering and `<script>` sanitization.

## Alternatives considered

- **Stream only "thinking"/step events, not answer tokens.** Rejected ‚Äî the user explicitly asked for
  token-by-token streaming; steps alone still freeze during the final synthesis turn.
- **Model-directed cards** (ask the model to emit structured references). Rejected ‚Äî an extra round-trip
  and less reliable than reusing tool results the loop already fetched deterministically.
- **`react-markdown`.** Rejected for a POC ‚Äî larger dep tree for the same result; `marked` + the
  existing DOMPurify matches the app's established sanitize pattern.
- **Full SSE streaming for GitHub Models too.** Deferred ‚Äî accumulating fragmented `tool_calls` deltas
  by index is materially more code and edge cases for the secondary provider; the buffered fallback is
  indistinguishable for short answers and keeps risk down.
- **Replace `/api/ai/ask` with the stream endpoint.** Rejected ‚Äî keeping the JSON endpoint preserves a
  clean fallback and the existing test surface.
