# ADR-063: Admin "Manage" panel hardened — one save for access + board overrides (v1.52)

## Status: Accepted (2026-07-13) — implemented in v1.52

## Context

ADR-062 fixed the read path so a per-user board override reaches the UI. But the field report that
prompted it had a second, more basic cause: **the override had never been saved.** In the admin console's
per-user "Manage" panel, access/credentials and board/env overrides were **two separate forms with two
separate submit buttons** ("Save access" and "Save user config"). An admin set the shared connection, saved
*that* form, and reasonably assumed the whole panel was saved — but the board override sat unsaved. The
store confirmed it: the borrower's `credentialSourceUserId`/`allowWrites` were persisted while `userConfigs`
was empty.

Two adjacent footguns compounded it:
- **Silent no-op on the wrong field.** `getProjects()` selects a board by its numeric **id**
  (`JIRA_DEV_BOARD_ID`); the project **key** is only a label (ADR-037/062). Editing the key alone changes
  nothing visible, with no cue that it wouldn't.
- **No unsaved-changes signal.** Nothing indicated the panel held edits that hadn't been persisted.

## Decision

Rework the per-user "Manage" panel (`packages/react-app/src/pages/Admin.tsx`):

- **One save for the whole setup.** A new `UserSetupForm` folds access (credential source + `allowWrites`)
  and the board/env overrides into a single form with **one "Save changes"** button. Password reset and
  the danger zone stay separate — they're discrete, opt-in actions. The parent handler fires only the parts
  that changed (`updateUser` for access, `putUserConfig` for the config, config last so its response
  reflects both), so a one-field edit is still one request. **No API/contract change** — the same two
  endpoints, just never split across two buttons the admin can half-use.
- **Unsaved-changes guard.** The form reports its dirty state up; the row shows an **"unsaved"** pill and an
  "Unsaved changes" hint by the button, "Save changes" is **disabled until something changes**, and clicking
  **Close** with unsaved edits arms an inline **"Discard unsaved changes? — Discard / Keep editing"** confirm
  instead of silently dropping them.
- **Board-ID-vs-key hint.** The Boards & projects group now carries the line "The board ID selects which
  board loads — set this to switch boards. The project key is only a label," and the field labels say so
  too ("Dev board ID (switches board)" vs "Dev project key (label only)"). The hint renders wherever the
  fields do (per-user, global defaults, template creation).

Shared plumbing was extracted so the global-defaults form and the per-user form stay in lockstep:
`collectConfig` (string map → `AdminConfig`), `configDiffers` (dirty check), and `ConfigFields` (the grouped
inputs + hint). `ConfigForm` (global defaults + template create) now wraps `ConfigFields`.

## Consequences

- The exact failure that motivated ADR-062 is now structurally prevented: an admin can't save the shared
  connection and forget the board — it's one save, and the panel won't let unsaved edits vanish quietly.
- Setting the wrong field is far less likely: the label and hint say the **id** is what switches boards.
- Fewer requests are possible per save (only changed sections fire), and the global/template forms are
  unaffected because they reuse the same extracted pieces.
- Behaviour-only, presentation-layer change — no server, tool, endpoint, or store change. The admin API
  contract (§9.6) is untouched; only the client's save orchestration changed.
- Gate: typecheck + build clean, react-app 763 → 767 tests (Admin.test.tsx 17 → 21: combined save, both-
  changed save, disabled-until-dirty, discard-guard, ID-vs-key hint), mcp-jira 535, github 57 → **1,359
  total**, smoke 53/53. Pill → 1.52.0.

## Alternatives considered

- **Keep separate forms, just add a save-all button.** Still leaves two half-usable submit buttons and the
  same "which one did I click" ambiguity; folding access + config into one form removes the trap outright.
- **Auto-save on blur.** Surprising for an admin editing several fields, and it would fire partial saves;
  an explicit, clearly-enabled "Save changes" with a dirty guard is more predictable.
- **A modal for the whole panel.** Heavier than warranted; the inline dirty pill + discard confirm give the
  same protection without a context switch.
- **Rename the store field / drop the project key.** Out of scope and higher-risk (touches live data and the
  multi-project model); a clear label + hint solves the confusion without a migration.
