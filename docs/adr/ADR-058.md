# ADR-058: Journal notes as a quick-add feed + the CORS PATCH fix

## Status: Accepted (2026-07-10) — implemented in v1.48

## Context

Two things surfaced right after the v1.47 journal shipped (ADR-057):

1. **The to-do toggle failed in the browser** with an error on `PATCH /api/me/journal/todos/:id`, even
   though the route worked in tests. The bridge's `cors({ methods })` listed `GET, POST, PUT, DELETE` —
   **not `PATCH`**. Toggling a to-do is the only `PATCH` in the whole app, so the browser's CORS preflight
   rejected it. Same-process `fetch` in the test suite never issues a preflight, so nothing caught it.
2. The **daily notes** UI (a date navigator + one note per day + an explicit Save button) was more
   ceremony than the user wanted. The ask: "a simple way to save a note."

## Decision

- **Notes are a quick-add feed.** Type a line, press **Enter**, it saves with a timestamp and appears at
  the top of a running list; **Shift+Enter** inserts a newline for multi-line notes. No date navigator, no
  per-day bucket, no Save button. Each entry can be deleted individually. To-dos likewise drop their `date`
  field and are simply sprint-scoped.
  - Data model changes from `notes: { [sprintId]: { [YYYY-MM-DD]: text } }` to `notes: JournalNote[]`
    (`{ id, sprintId, text, createdAt }`). The store **migrates** any v1.47 per-day notes into feed entries
    on first read — dated midday UTC for stable ordering — and **persists** the conversion so entry ids are
    stable (delete-by-id then works). Routes: `PUT /note` → `POST /notes` + `DELETE /notes/:id`.
- **CORS lists every verb the app uses.** Added `PATCH` to `cors({ methods })`, and added an explicit
  **preflight** test (`OPTIONS` with `Access-Control-Request-Method`) asserting `GET/POST/PUT/PATCH/DELETE`
  are all advertised. This is the one class of bug the same-process integration tests structurally cannot
  catch, so it gets its own guard.

## Consequences

- Saving a note is now one gesture (type → Enter). The feed doubles as a lightweight work log.
- The migration means nobody loses the notes they already wrote under v1.47; it runs once, transparently.
- The preflight test documents the invariant "every verb must be in `cors({ methods })`" so the next new
  verb (if any) fails a unit test rather than only failing live.
- Gate: 1,347 tests (mcp-jira 532, react-app 758, github 57), smoke 53/53, build clean. Pill → 1.48.0.

## Alternatives considered

- **One autosaved sprint note** — simplest, but loses the per-entry history that makes the feed useful as a
  daily log; the user picked the feed.
- **Keep one-note-per-day, drop only the navigator** — still imposes a date model the user didn't want.
- **Allow all methods in CORS (`*`)** — `cors` can't reflect a specific credentialed origin and use `*`
  methods cleanly, and an explicit allowlist is the safer, greppable choice consistent with the rest of the
  bridge.

## Follow-up (planned, not in this change)

A dedicated **UI/UX review phase**: an expert pass over every surface (accessibility, keyboard nav,
contrast, empty/error/loading states, responsive breakpoints, copy) producing a prioritised findings report
with fixes. Tracked as the next phase.
