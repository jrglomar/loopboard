# ADR-048: Formatted sprint-review export — styled Excel + printable PDF + per-member table

## Status: Accepted (2026-07-01) — implemented (react-app, frontend-only)

## Context

The full sprint-review export (ADR-045) produced a flat two-column Field/Value **CSV**. The user
wanted a **good-looking, formatted** review document, and — critically — a **per-member breakdown**:
each individual's committed/completed points plus their leaves (by type) and offset standing. A CSV
cannot carry formatting, and none of that member data was being gathered on the Reports page.

## Decision (all frontend; reuses existing tools)

### One shared model
- `buildSprintReviewMeta(report, form, flyIns)` (extracted in `reportMarkdown.ts`) is the single source
  of the Field/Value pairs; the CSV, Excel, and print renderers all consume it so they never drift.
- `buildMemberReviewTable(report, leaves, ledger, requiredPoints, roster)` (`lib/sprintReview.ts`, pure)
  builds one row per member — the **union of the dev roster** (`useTeamMembers(sprint.boardId)`) and anyone
  with points (`report.byAssignee`), leaves, or an offset standing. Including the roster means every
  developer's capacity counts even with no tickets. **`committed` is a CAPACITY commitment =
  `max(0, requiredPoints − leaveDays)`** (N = `health.policy.requiredPoints`, e.g. 8; each leave day of any
  type subtracts one) — **not** the member's assigned tickets, which is what the user's model means by
  "committed". `completed` = `donePoints`. Plus leave days per type (`VL/EL/Holiday/Offset`, over working
  days via `leaveDaysByType`) and offset balance. Sorted by name, "Unassigned" last (commitment 0), TOTAL row.
- **Summary "Commitment points" = total team capacity** (Σ committed over the roster) — prefilled into the
  export form (still editable), replacing the old prefill from Jira's committed points. E.g. 7 devs × 8 with
  no leaves → 56.
- **Sprint-goal normalization.** Jira stores the goal as one string but users put several goals on separate
  lines; `normalizeGoal` collapses newlines to " · " so the full goal shows on one line — fixing the
  on-screen run-on (newlines→spaces) and the export's first-line-only truncation. Used on-screen + in export.

### Three export formats (from the same dialog)
- **Styled Excel (.xlsx)** — `sprintReviewXlsx.ts` uses **`xlsx-js-style`** (Vite-friendly, styling-capable
  SheetJS fork). A pure `sprintReviewAoa(...)` builds the cell matrix + row/merge metadata (unit-tested);
  a thin pass applies the title band, section headers, member-table borders, and a bold TOTAL row.
  Member numeric cells are real numbers so Excel can sum them.
- **Printable report → PDF** — `buildSprintReviewHtml(...)` (pure) returns a self-contained, print-ready
  HTML document (title band, summary grid, member table, retro blocks; HTML-escaped). A thin `printHtml`
  helper opens it in a new window and calls `print()` (guarded — a no-op when the window is unavailable).
- **CSV** — unchanged (`buildSprintReviewCsv`), still offered.

### Data wiring
- `SprintReportView` (Reports) now reads `useLeaves(selectedSprintId)` + `useOffsetLedger()` and threads
  the typed `LeavesMap` + `OffsetLedger` through `ExportBar` → `SprintReviewExport`.

## Consequences
- **No new tool; jira tools stay 36; smoke unchanged.** The member table reuses `get_sprint_report`
  (points), `get_leaves` (typed leaves), and `get_offset_ledger` — all read-only, all pre-existing.
- Adds one runtime dependency, `xlsx-js-style` (~SheetJS-sized); acceptable for the POC and it bundles
  cleanly under Vite. The pure builders keep the testable logic library-independent.
- Everything is a client-side download / print — no writes to Jira; safe to run live without gating.

## Alternatives considered
- **exceljs** for the workbook — heavier and more prone to Vite/Buffer-polyfill friction; `xlsx-js-style`
  covers the needed styling (fonts, fills, borders, widths, merges) with less weight.
- **HTML-table-as-.xls** (no dependency) — opens in Excel but triggers a "format/extension mismatch"
  warning; rejected in favour of a real `.xlsx`.
