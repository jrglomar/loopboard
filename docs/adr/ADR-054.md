# ADR-054: Task Helper — per-user accounts + AI ticket→prompt (additive multi-account)

## Status: Accepted (2026-07-09) — planned + approved; implementing in phases A→D

## Context

Loopboard is single-tenant by construction: `getConfig()` is a global singleton (one Jira/GitHub/AI),
`jiraClient.getClient()` is a module-level singleton axios instance used by all ~40 tools, the bridge's
`tool.handler(req.body)` has no identity, and state is JSON files keyed by board/sprint. There is no auth.

The team wants a **team-internal** feature: each developer teammate signs up, connects **their own** Jira +
GitHub (pasted tokens), fetches **their** sprint tickets, and gets an AI-produced **refined ticket** + a
**ready-to-paste coding-agent prompt**. Question posed: the best way to handle multiple accounts here.

## Decision

**Introduce identity only for the new section** — do not retrofit multi-tenancy into the existing stack.

- **Auth**: signup/login with an httpOnly, HMAC-signed session cookie; `requireAuth` middleware guards every
  Task Helper route. Passwords hashed with **scrypt** (salted, timing-safe verify).
- **Per-user connection vault**: each user's Jira/GitHub tokens are **AES-256-GCM encrypted at rest** (key
  from `TOKEN_ENC_KEY`), stored in a git-ignored JSON user store, **decrypted only in-memory per request** to
  build a client, and **never returned to the client or logged** (only `{connected, baseUrl/login, hint}`).
- **Request-scoped client**: a small `userJira` factory builds an axios instance from the logged-in user's
  decrypted creds — the Task Helper needs only a couple of reads (`/myself`, my-sprint-issues, one issue), so
  the singleton `jiraClient` and the 40 tools stay untouched.
- **AI**: reuse the existing `AiProvider` port + `getAiProvider()` (the app's shared key). A new
  `taskHelperService` runs refine → technical-plan → prompt-engineer personas via `provider.complete()`.
- **Placement**: on the existing mcp-jira HTTP bridge (already the app's backend, already hosts non-MCP
  `/api/ai/*`), in modules imported **only** by `http.ts`/the router — never by `tools/index.ts` or the
  stdio entry. The MCP tool registry and Copilot's server are unaffected.
- **Zero new dependencies**: Node built-in `crypto` (AES-GCM, HMAC, scrypt) + a JSON store (mirrors the
  established `.loopboard-*.json` pattern) + Express's built-in `res.cookie`. Rejected `better-sqlite3`
  (native build) / `bcryptjs` / `cookie-parser` as unnecessary for a small team tool.

## Consequences

- Enabled only when `TOKEN_ENC_KEY` + `SESSION_SECRET` are both set; otherwise routes 503
  `TASK_HELPER_UNAVAILABLE`, so keyless tests + the shared board keep working untouched.
- The JSON store's read-modify-write has the same (accepted) concurrency tradeoff as the other stores;
  fine for a handful of teammates. A future migration to SQLite/Postgres + OAuth is a clean follow-on that
  can adopt these same primitives (auth, vault, request-scoped client).
- GitHub is connected + validated but not deeply used in v1 (the fetch flow is Jira-centric); no per-user
  run history is persisted in v1.
- Security surface is new (auth, cookies, secrets) — hence: encrypt-at-rest, never-return-tokens, scrypt,
  timing-safe compares, httpOnly+SameSite cookies, basic login rate-limiting, CORS with credentials + a
  specific origin.

## Alternatives considered

- **Full multi-tenant rewrite** (every page on each user's own Jira): retrofits ~40 tools + all JSON stores
  to per-user via AsyncLocalStorage — high risk, unnecessary for this feature. Rejected for v1.
- **Public SaaS with Clerk + Postgres**: overkill for a team-internal tool where teammates just set up their
  own account. Rejected.
- **OAuth (Atlassian 3LO + GitHub App)**: cleaner UX but needs registered apps + token refresh; deferred in
  favor of encrypted pasted tokens for v1.
