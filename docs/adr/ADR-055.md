# ADR-055: Multi-tenant Loopboard — per-user Jira/GitHub/AI via request-scoped config

## Status: Accepted (2026-07-10) — Phases A–F implemented (complete)

## Context

The team pivoted Loopboard from a shared-config tool to a **multi-tenant app**: login required to
enter, every feature runs on the signed-in user's **own** Jira/GitHub token, the **AI runs on each
user's own token**, and a super-admin configures board keys + supervises users. This builds on the
v1.44 Task Helper (auth + encrypted per-user connection vault).

## Decision

- **Request-scoped config (the backbone).** `getConfig()` is the single funnel every tool +
  `jiraClient` + the AI layer flow through. A new `lib/requestContext.ts` (`AsyncLocalStorage`) carries
  the per-user config; `getConfig()` returns it when present, else the global `.env` (so stdio/Copilot
  and the keyless test suite are unchanged). A bridge middleware (`perUserContext`) on `/api/tools` +
  `/api/ai` resolves the session → the user's merged config → `runWithUser(...)`, so **all 40 tools run
  on the user's own Jira with zero tool changes.** `jiraClient` now caches its axios client **per
  credential set** (a global singleton would have reused user A's client for user B). Per-user JSON
  stores live under `.loopboard-user-stores/<userId>/`.
- **Per-user config = merge** of the global `.env` base ← (Phase B) admin per-user overrides ←
  per-user encrypted secrets. `resolveUserConfig(userId)` overrides the Jira creds with the user's
  connection and folds in their **AI** connection (provider + token + model), so `getAiProvider()`
  becomes per-user automatically.
- **Per-user AI (fixes the shared-token pain).** A new `ai` connection (`validateAi` does a tiny live
  call at connect time so a bad/expired token is caught then, clearly). The Task Helper's `/help` runs
  inside `runWithUser`, so it uses the user's own AI token.
- **App-wide gate (frontend).** `AuthProvider` at the app root + `AppGate`: not signed in → login;
  signed in but not **ready** (Jira + GitHub connected, from `GET /api/me/context`) → onboarding
  (connect accounts); ready → the app. `callTool` + the AI client now send `credentials: "include"`,
  so the board/huddle/reports render the user's own data.
- New endpoints: `PUT /api/me/connections/ai`, `GET /api/me/context` (connections + readiness + boards).

## Consequences

- **Existing suites stay green** because of the `.env` fallback (no ALS context in unit tests). The
  frontend test migration was small: mock `AppGate` as pass-through in `App.test`; the gate is tested
  separately. Gate at completion (v1.46): 1,297 tests, smoke 53/53, build clean.
- The stdio/Copilot server is untouched (no ALS → global `.env`).
- **Phase B (roles + super-admin)** and **Phase E (admin console UI)** are implemented: `ADMIN_EMAILS`
  bootstrap, `requireAdmin`, `routes/admin.ts` (users/config/role), and an admin-only **Admin** tab
  (`pages/Admin.tsx`) for user supervision + global/per-user board-env config. The AI json-mode bug
  (GitHub Models 400 on the Task Helper's Markdown output) was fixed alongside — `complete()` is now
  free-form; JSON mode is `completeWithSchema`-only.
- **Phase F (done, v1.46):** the Task Helper consumes App's shared board+sprint context
  (`SharedSprintProps`) and scopes its ticket list to the **selected sprint** —
  `GET /api/me/tasks/issues?sprintId=<id>` → JQL `assignee = currentUser() AND sprint = <id>`, defaulting
  to the board's ACTIVE sprint. Omitting `sprintId` preserves the v1.44 `openSprints()` behavior. The
  board selector is now shown on the Task Helper tab.
- Security unchanged from ADR-054: tokens AES-256-GCM at rest, decrypted only in-memory per request,
  never returned to the client or logged.

## Alternatives considered

- **Rewrite all 40 tools to take credentials** — rejected; AsyncLocalStorage + the `getConfig()` funnel
  achieves per-user scoping with near-zero tool changes.
- **Per-request axios (no cache)** — the per-credential-set cache keeps efficiency while staying correct.
