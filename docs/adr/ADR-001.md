# ADR-001: Dual Transport — stdio MCP + HTTP Bridge over a Shared Transport-Agnostic Tool Registry

## Status: Accepted

## Context

The system must serve two distinct consumers:

1. **GitHub Copilot (Claude)** — communicates with MCP servers exclusively via **stdio** (the MCP protocol spec standard for local server processes). Copilot spawns the server as a child process and exchanges JSON-RPC frames on stdin/stdout. There is no way to configure VS Code Copilot to call an HTTP endpoint as an MCP server for local development without the experimental StreamableHTTP transport.

2. **React dashboard (browser)** — runs in a browser security sandbox where Node.js `child_process` and raw stdin/stdout are unavailable. The only inter-process communication mechanism available to a browser is HTTP (fetch). The Vite dev server at `:5173` must call outbound HTTP.

Both consumers need access to the same tool functionality (sprint data, ticket creation, PR linking). Without a shared implementation, the same business logic would have to be written and maintained twice — once for the MCP stdio path and once for HTTP route handlers. This would double the surface area for bugs and create divergence risk between what Copilot calls and what the dashboard calls.

The spec (§5, §6) explicitly shows `src/http.ts` as a sibling to `src/index.ts` in the same package, confirming the intent to support both transports from one codebase.

## Decision

Each MCP package (`mcp-jira`, `mcp-github`) implements a **transport-agnostic tool registry** as its core:

- `src/tools/index.ts` exports `const tools: ToolDef[]` where `ToolDef` is:
  ```ts
  interface ToolDef {
    name: string;
    description: string;
    schema: z.ZodObject<z.ZodRawShape>;
    handler: (input: unknown) => Promise<unknown>;
  }
  ```
- Each tool handler validates its input using its own `schema` and returns a plain domain object.
- `src/index.ts` is the **stdio adapter**: it iterates `tools`, calls `server.registerTool()` for each, and wraps handler results in `{ content: [{ type: "text", text: JSON.stringify(result) }] }`. It logs startup exclusively to **stderr** (stdout carries MCP protocol frames — any stdout write corrupts the framing and breaks Copilot).
- `src/http.ts` is the **HTTP adapter**: an Express app exposing `POST /api/tools/:name`, iterating the same `tools` array, and wrapping results in `{ ok: true, data: result }` or the error envelope defined in `docs/CONTRACTS.md §2`.

Both adapters consume the same `tools` array. Tool handlers are tested independently of both adapters.

## Consequences

**Positive:**
- Tool business logic is written and tested once. Copilot and the dashboard always call identical code paths.
- Adding a third transport (WebSocket, Server-Sent Events, StreamableHTTP for remote Copilot extensions) requires only a new adapter file — no handler changes.
- The stdio entry can be registered in VS Code `settings.json` with `"command": "npx tsx ..."` exactly as the spec shows.

**Negative / trade-offs:**
- Each package runs as two processes simultaneously during development (`npm run dev` for stdio; `npm run dev:http` for HTTP). Developers must start both if they want both consumers.
- The `ToolDef.handler` signature takes `unknown` and must perform its own zod parse internally; this is a minor repetition but keeps the registry protocol simple and prevents adapter-specific types from leaking into handlers.

**Normative rule (enforced by quality gate):** The stdio adapter entry MUST NOT write to stdout at any point (including during startup, logging, or error handling). All diagnostic output goes to stderr. Tests that capture stdout in stdio mode must assert it is empty.

## Alternatives considered

### Alternative A: HTTP-only, using StreamableHTTP MCP transport in Copilot

The MCP SDK supports a `StreamableHTTP` transport. In principle, VS Code Copilot could be configured to connect to an HTTP MCP server.

**Rejected because:** The StreamableHTTP transport requires the Copilot MCP config to specify a URL rather than a command. This is not the `"command"/"args"` stdio model described in spec §6.2 and widely documented for local Copilot MCP setup. For a POC targeting local development with standard VS Code Copilot configuration, stdio is the proven path. Introducing StreamableHTTP would add configuration complexity and reduce compatibility across Copilot versions.

### Alternative B: Duplicate handlers — separate implementations per transport

Implement tool logic directly in both `src/index.ts` (for MCP) and `src/http.ts` (for Express routes).

**Rejected because:** This doubles the code under test, doubles maintenance burden, and introduces the risk of behavioral divergence between what Copilot calls and what the dashboard calls. Any bug fix or output format change would require two edits.

### Alternative C: React app calls stdio server via a Node.js bridge process (IPC)

A Node.js companion process in the React app could forward calls from the browser to the stdio server.

**Rejected because:** This adds a third process layer with IPC complexity (child_process, message passing) while providing no advantage over a simple HTTP express server. HTTP is the natural protocol for browser-to-process communication.
