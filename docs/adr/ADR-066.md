# ADR-066: Offset amounts are decimal-capable (v1.55)

## Status: Accepted (2026-07-13) — implemented in v1.55

## Context

The offset ledger (ADR-038/044/061/065) treated every amount as an integer: `earned`/`spent` on the
per-sprint snapshot, the single `manualAdjust` (opening balance), and each `adjustments[].amount` were all
zod `.int()`, and the Leaves opening-balance input `Math.trunc`ed its value. But teams credit **half-points**
(0.5, 0.25, …) — a plain integer offset couldn't represent them, so a "0.5" opening or a "+0.5" manual
adjustment was silently truncated or rejected (`400 VALIDATION`).

## Decision

- **Drop `.int()` from every offset amount.** `add_offset_adjustment.amount` (keeps the non-zero rule),
  `set_offset_adjustment.manualAdjust`, and `set_offset_for_sprint`'s `earned`/`spent` are now plain
  `z.number()` (non-negativity kept where it applied). `sprintId` stays an int (it's an id). The store,
  `summarizeOffset`, and the wallet math already used ordinary `+`/`−`/`reduce`, so decimal balances just
  work.
- **The Leaves opening input accepts decimals.** `Math.trunc(...)` → `Math.round(v * 100) / 100` (2 dp to
  keep the store tidy), and the input gains `step="0.5"`. The History dialog's add form parses with
  `Number.parseFloat` (was `parseInt`) and its amount input is `step="0.5"` / `inputMode="decimal"`.
- **Every offset figure renders through `formatPoints`** (≤ 2 dp, trailing zeros trimmed — the same helper
  Reports/velocity use). This is what makes decimals safe on screen: a fractional sum like
  `0.1 + 0.2 = 0.30000000000000004` displays as `0.3`, and `2` still shows as `2`. Applied to the offset
  table (Done / Total / Earned / Balance), the wallet card (earned / used / opening / adj / balance), and
  the History dialog (tiles, earned list, the signed adjustment amounts).

## Consequences

- Half-point (and any decimal) credits now round-trip end to end: opening balance, per-sprint banking, and
  the manual-adjustment log all accept and display fractions; balances stay exact-looking via `formatPoints`.
- No data migration — existing integer values are valid numbers; older store files read unchanged.
- Float summation noise is a display-only concern, fully handled by `formatPoints`; stored inputs are
  rounded to 2 dp at the two entry points (opening input, and callers pass already-parsed floats).
- The binary earned *rule* (`(done + leaveDays) ≥ N + N2 ? 1 : 0`, ADR-038) is unchanged — this ADR only
  makes the *amounts* the ledger can hold/display fractional; earned still banks 0 or 1 per sprint by policy
  (but the snapshot tool would now accept a fractional earned if a future feature sends one).
- Gate: typecheck + build clean, mcp-jira 542 → 543 (offset decimal test), react-app 774 → 777 (wallet +
  dialog decimal tests), github 57 → **1,377 total**, smoke 55/55. Pill → 1.55.0. **Not live-eyeballed**
  (Offset Tracker behind the login gate) — proven by the decimal unit/jsdom tests + `formatPoints` coverage.

## Alternatives considered

- **Store cents (integer ×100).** More invasive (migration + every read/write scales) for no real gain in a
  JSON store; plain floats + `formatPoints` on display are sufficient here.
- **Round to whole numbers in the UI.** Defeats the purpose — the user explicitly credits 0.5.
- **Allow decimals only on the manual adjustment.** Inconsistent — opening and per-sprint amounts would
  still truncate; relaxing all four keeps the ledger uniformly decimal-capable.
