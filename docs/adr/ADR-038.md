# ADR-038: Typed leaves + offset-points engine + dedicated Leaves page

## Status: Accepted (2026-06-26)

## Context

The leaves tracker stored untyped dates (`{ sprintId: { assignee: string[] } }`) and lived as cards on
the Planning + Reports pages. The user wants:
- A **dedicated Leaves page** for visibility.
- Leave **types** when plotting: Vacation (VL), Emergency (EL), Holiday, Offset.
- An **offset-points engine**: each member has `N` required points/sprint; if `completedPoints +
  leaveDays ≥ N + N2` they earn **at most 1** offset point that sprint (each leave day = 1 pt). `N`/`N2`
  configurable. An **Offset** leave **spends** from a per-developer **running balance**, and there must
  also be a **manual** way to adjust the balance.

## Decision

### Typed leaves (store v2)
`LeaveType = "VL" | "EL" | "Holiday" | "Offset"`. Store →
`{ sprintId: { assignee: { "YYYY-MM-DD": LeaveType } } }`. `readLeaves` **normalizes on read** (legacy
`string[]` → `{ date: "VL" }`), so the existing `.loopboard-leaves.json` keeps working with no
migration. `set_leaves` takes `entries: [{ date, type }]` and **also** accepts legacy `dates: string[]`
(→ VL) so the pre-v1.26 frontend keeps working during the transition. `get_leaves` returns the typed map.

### Offset engine (config + ledger, two adjustment paths)
- Config: `JIRA_REQUIRED_POINTS` (N, default 8) + `JIRA_OFFSET_THRESHOLD` (N2, default 2), exposed at
  `GET /api/health .policy`.
- Pure earn rule (frontend `offset.ts`): `earned = (donePoints + leaveDays) ≥ (N + N2) ? 1 : 0` —
  **capped at 1 per sprint**. `donePoints` from `get_sprint_report.byAssignee`; `leaveDays` = count of
  VL+EL+Holiday+Offset on sprint working days.
- Ledger store (`offsetStore`) keyed by assignee:
  `{ bySprint: { [sprintId]: { earned, spent } }, manualAdjust }`. **Two ways** (per the user):
  `set_offset_for_sprint` writes the **auto** per-sprint snapshot (idempotent — re-recording a sprint
  never double-counts); `set_offset_adjustment` sets a **manual** absolute delta. `get_offset_ledger`
  returns the computed `{ earned, spent, manualAdjust, balance }` per developer, where
  `balance = Σ earned − Σ spent + manualAdjust`. jira tools 32 → 35.

### Frontend (next slice)
A new **Leaves** tab: a leave-type painter, the working-day grid with typed colored cells, and the
per-developer offset table (Required N · Done · leaves by type · Total · Earned · Spent · Balance) with
the manual-adjust control — per the approved mockup. Reports keeps its read-only capacity view.

## Consequences

- Leaves carry meaning (VL/EL/Holiday/Offset) and feed a configurable offset engine with both
  automatic accrual and manual override; existing data + the running frontend keep working through the
  `set_leaves` back-compat shim.
- The offset balance is derived from per-sprint snapshots (written when the Leaves page records a
  sprint) + a manual delta — a pragmatic POC ledger that avoids re-scanning all history.

## Alternatives considered

- **Keep untyped dates, add a parallel "type" store.** Rejected — two stores to keep in sync; typing
  the existing store with a read shim is simpler and lossless.
- **Compute the balance by scanning every sprint's report + leaves on demand.** Accurate but heavy;
  the snapshot ledger is cheaper and good enough for a POC.
- **Auto-only or manual-only offset.** The user explicitly wants both.
