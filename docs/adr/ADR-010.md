# ADR-010: Design-system refresh + Scrum affordances (data-grounded, no Phase-3 fabrication)

## Status: Accepted (2026-06-12)

## Context

After the shadcn/Tailwind overhaul (ADR-009), the product owner commissioned a two-
specialist live review — a senior UI/UX designer and a Scrum Master — of the running
dashboard, then asked to apply the recommendations. The review found: a flat grey-on-white
palette with weak status signaling and ad-hoc semantic colors (`green-50`/`amber-50`
literals); soft typographic hierarchy; and several Agile-ceremony gaps — no sense of sprint
time remaining, progress shown as a bare fraction, blockers invisible until hunted, a
status-first huddle that fights person-first standups, and no one-click "my work" focus.

## Decision

1. **Refine the token system, don't add color.** Promote the shadcn CSS variables to a
   real semantic set (`success`/`warning`/`error`/`info`, each with `-bg`/`-border`),
   a slate neutral ramp, a named status-color group (todo/inprogress/codereview/done/
   blocked), standardized radius/shadow/spacing/type scales, lucide iconography (already a
   dependency), and motion guarded by `prefers-reduced-motion`. Add a `.dark` variable
   block now (no toggle UI yet) so dark mode is near-free later.
2. **Add Scrum affordances ONLY from data already in the tool outputs** — sprint progress
   bar + issues count (`totals`), days-remaining timeline (`sprint.startDate`/`endDate` +
   client clock), a **pace heuristic** (% time elapsed vs % points done → on-track/behind/
   ahead, explicitly a heuristic not a forecast), a blocker banner + "Show blocked" filter
   (`totals.blocked`), a remembered "My issues" quick filter (`localStorage`), and a huddle
   "By status / By person" regroup of the same `get_daily_huddle` data. **No MCP tool,
   HTTP, or env contract changes** — this is presentation + client-side derivation.
3. **Do not fabricate Phase-3 data.** Velocity averages, burndown charts, and a retro board
   need historical/persisted sprint data the bridges do not expose. They stay Phase 3; the
   Reports page gets clearly-labeled DISABLED placeholders rather than invented numbers.
   The pace heuristic is the honest, live stand-in for burndown using only current data.
4. Preserve all behavior, props, state, every prior contract, exact contract-critical copy,
   a11y, and the keyless/offline test gates. Delivered as a single applied pass against the
   §6.1 spec.

## Consequences

- The board reads at a glance: time left, progress, blockers, and "my work" are now
  first-class — the questions a daily scrum actually opens with.
- Semantic tokens kill the ad-hoc color literals and make future theming/dark-mode a
  variable swap.
- The pace chip is a heuristic; it is labeled as such so no one mistakes it for the
  Phase-3 velocity forecast. This is a deliberate honesty boundary (cf. ADR-004's
  "no security/▒ theater" spirit — don't show numbers you can't stand behind).
- "My issues" identity is remembered client-side (no auth, no current-user lookup, no Jira
  round trip) — a pragmatic POC choice; a real `myself` lookup is a future option.
- localStorage adds a tiny persistence surface (one key); it degrades gracefully when
  absent/blocked.

## Alternatives considered

- **Fake a velocity/burndown to "apply all recommendations"** — rejected; fabricated Agile
  metrics are worse than absent ones (they mislead planning). Phase-3 placeholders + the
  honest pace heuristic instead.
- **Add a `get_current_user` tool / `myself` call for "My issues"** — a Jira round trip and
  contract surface for something localStorage solves for a single-user POC. Deferred.
- **Server-side days-remaining/pace in `summaryText`** — would change the tool contract and
  couple a clock into the server; kept client-side to leave the contract stable.
- **A heavier component/charting library for progress/burndown** — overkill; a CSS bar and
  a heuristic chip suffice and stay dependency-light.
