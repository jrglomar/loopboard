# ADR-081: Draft Capacity Plan — pure-draft multi-developer point split + full-width redesign (v1.70)

## Status: Accepted (2026-07-21) — implemented in v1.70

## Context

Live testing of v1.68/v1.69 produced two clear pieces of user feedback that correct the direction:

1. **The draft card must not write to Jira — at all.** v1.69 wired inline points edits, rename,
   status change, move-to-sprint, and a ticket-creating "breakdown" onto the draft chips, and every
   one of those hits Jira. The user's model is cleaner: the Draft Capacity Plan is *only* for
   seeing and checking whether the team's capacity matches the PO sprint's tickets. Real edits
   belong to the **Assign Tickets** card ("Assign Tickets should handle those actual changes in
   JIRA"). So the two surfaces separate by intent: Assign Tickets = real Jira writes; Draft
   Capacity Plan = a non-committing overlay.

2. **"Breakdown" meant something different than what was built.** The user clarified: real Jira
   points stay read-only, "but we should be able to make some breakdown ticket, points manipulation
   just for visibility, because 1 PO ticket can be assigned to multiple developers with breakdown
   points." That is not creating sub-tickets in Jira (what v1.69's `BreakdownDialog` does) — it is
   splitting ONE PO story's points across SEVERAL developers, as a draft, so a big story shared by
   two people loads each person by their slice instead of dumping the whole estimate on one tile.

3. **The layout read as cramped.** The v1.68/v1.69 card used a narrow-left (tickets) / wide-right
   (developers) split. The user asked for developers on top so both tickets and developers get full
   width.

## Decision

React-app is the bulk; one backend shape change. **Nothing on this card writes to Jira.**

**1. Draft model → per-developer point shares (§4.30).** An issue key maps to `DraftShare[]`
(`{ accountId, displayName, points }`), not one member. A ticket can be split across developers,
each holding a DRAFT points slice. `points` defaults to the ticket's real Jira points when a ticket
is drafted whole to one person; splitting divides it; over/under vs the real estimate is allowed and
never enforced (capacity is advisory — unchanged ADR-079 stance). The store, tools, and the read-
migration (pre-v1.70 single object → one-element array, `points: 0`) live in mcp-jira; the tools
still never touch Jira. This is the only backend change — no new tools, count stays 45.

**2. Draft card becomes strictly read-only toward Jira.** Removed: inline points WRITE, rename,
status, move, and the ticket-creating breakdown. Real Jira points render read-only. Kept and now
central: the draft split (server-side draft store), per-share DRAFT point manipulation, per-developer
capacity vs drafted load (Σ of that dev's share points) with over/under, the v1.69 leaves/offset
capacity-source transparency + non-sticky Dev-sprint pairing (the fix for "everyone shows 8"), and
the embedded dev `TeamManager`.

**3. Assign Tickets absorbs the real edits.** It already owns points/status/move (v1.15/v1.37) and
rename (v1.69); it now also hosts the real `BreakdownDialog` (creates new PO stories via
`create_po_ticket`) relocated from the draft card. One surface, one place where planning writes to
Jira.

**4. Full-width two-tier redesign (frontend-design pass, within the existing shadcn/token system —
this is an internal planning surface, so the win is a disciplined capacity board, not a rebrand).**
- TOP — developers, full-width responsive grid of capacity cards. Each card is a drop target and
  shows: name, capacity pts (or a loading placeholder while the Dev sprint's leaves load, or "—"
  when the paired sprint has no dates), drafted load (Σ share points), an over/under chip, and its
  share chips (ticket key + summary + an inline DRAFT points input + remove). The capacity match —
  drafted-vs-capacity — is the card's thesis, so it leads visually.
- BOTTOM — PO tickets, full width. Each ticket shows key, summary, real points (read-only), an
  "N of M pts drafted" allocation indicator, small chips of the developers it is split across, and
  two ways to draft a slice: native HTML5 drag onto a developer card, and a native "Draft to… / Add
  developer" `<select>` (the keyboard/screen-reader path, ADR-009). A ticket stays reachable in the
  bottom tier until fully allocated, so adding a second developer to a partly-drafted story is a
  first-class action, not a hidden one.
- Microcopy stays in the interface's voice: the "Draft only — nothing is assigned in Jira." badge
  is retained verbatim and now literally true of every control on the card; empty states invite the
  next action ("Drag a ticket onto a developer to draft it").

Default share points on draft: the first share of a ticket gets the ticket's full real points; a
subsequent share defaults to the remaining unallocated points (floored at 0). The user then
manipulates each share freely. Per-ticket "allocated N of M" makes the split legible.

## Consequences

- The draft card does exactly one job — check capacity against the sprint's tickets — and can never
  surprise anyone by mutating Jira. Real changes have exactly one home (Assign Tickets).
- Splitting a story across developers models real shared work, so per-developer load and the
  over/under signal are meaningful for stories bigger than one person.
- The v1.69 draft-card action work is largely superseded (its cells return to Assign Tickets only);
  `ticketCells.tsx` stays the shared implementation, now consumed only by Assign Tickets. This is
  deliberate churn driven by testing feedback, not a revert of the whole phase — v1.69's capacity-
  transparency fix is the load-bearing part and it stays.
- Migration is lossless except for the (previously nonexistent) point split on legacy drafts, which
  read back as a single share at 0 pts and are trivially re-drafted.
- Tests: mcp-jira 644 → **654** (+10, all in draftPlan.test.ts: multi-share round-trip, per-ticket
  accountId de-dupe, points + empty-array validation, share-count bounds, legacy single-object →
  array migration), react-app 1066 → **1099** (+33: draftPlan rollups over shares, useDraftPlan/
  client share shape, DraftPlanCard two-tier multi-share/point-split/read-only-points, AssignmentList
  breakdown relocation, Planning wiring), mcp-github unchanged 57 — total **1,810**; smoke 60/60.
  react-app version pill 1.69.0 → 1.70.0.

## Alternatives considered

- **Draft-only points as a per-ticket single override (not per-developer).** Rejected — it can't
  express "5 pts to Alice, 3 to Bob," which is the whole point of the user's clarification.
- **Keep the real breakdown on the draft card but make it draft-only** (create no Jira tickets,
  just draft splits). This is effectively what the per-share model IS; a *second*, Jira-writing
  breakdown on the same card is exactly the confusion the user flagged, so the ticket-creating one
  moves to Assign Tickets and the draft split lives on the card.
- **Kanban lanes (developers as columns, tickets inside lanes).** Rejected — it collapses the
  ticket pool into the lanes, so tickets don't get their own full-width tier; the user explicitly
  wants both developers and tickets at full width, which the stacked two-tier board delivers.
- **Deleting `BreakdownDialog` outright** (assume the user no longer wants Jira sub-tickets).
  Rejected — splitting a story into real sub-stories is a legitimate real action; relocating it to
  Assign Tickets preserves it where real edits live, and it's one unobtrusive row action the user
  can veto later.
- **Forcing shares to sum to the ticket's points.** Rejected — ADR-079 established capacity here is
  advisory (over/under allowed); enforcing exact allocation would fight the "just for visibility"
  intent.
