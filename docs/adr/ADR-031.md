# ADR-031: Huddle daily sections — auto-PRs, post-scrum tracking, meeting goal

## Status: Accepted (2026-06-25)

## Context

The Huddle (daily standup) sidebar has two manual cards (impediments + pending PRs, ADR-027). The
user wants the daily ceremony better supported:

1. **PRs linked to Jira should appear in code review automatically** — not just hand-pasted links.
2. A **post-scrum section** to capture per-person "parking-lot" follow-ups after the standup, so they
   are tracked rather than lost.
3. A **meeting-goal** component — the focus for *today's meeting* (distinct from the Jira sprint goal).
4. The daily widgets should be **compact, always-visible sidebar cards** (chosen over a popup).

`list_prs` (mcp-github) already returns `PrSummary[]` with detected `jiraKeys`. The impediments/PR
stores establish the bridge-side JSON store pattern (per-sprint, config-driven path, git-ignored).

## Decision

### Auto-PRs (current sprint) — frontend-only

The code-review card (`PullRequestsCard`) gains an **auto** list: it calls `list_prs` and shows open
PRs whose `jiraKeys` intersect the **current sprint's** ticket keys (passed down from the loaded
sprint board). Scope is deliberately "linked to the current sprint" (not all open PRs). The **manual**
store stays as a supplement (PRs without a key, or cross-repo links). When github is unavailable the
card degrades to manual-only — no error blocks the Huddle. No new tool; no contract IO change.

### Post-scrum store + card (`get/set_post_scrum`)

New `postScrumStore.ts` (mirrors `impedimentsStore`) keyed by sprintId → `PostScrumNote[]`
(`{ id, person, note, createdAt, resolved? }`). Full-replace `set_post_scrum` fills id/createdAt.
New `PostScrumCard` adds a note (person + text) and lists them grouped by person — the "tracking".

### Meeting-goal store + card (`get/set_meeting_goal`)

New `meetingGoalStore.ts` keyed by sprintId → `{ goal, updatedAt }`. `set_meeting_goal` stamps
`updatedAt`. New `MeetingGoalCard` is an inline editable goal (like the sprint-goal editor). This is
the *standup's* focus and is intentionally separate from the Jira sprint goal (§4.10b).

### Read-allowlist + compact redesign

`get_post_scrum` + `get_meeting_goal` join the AI Q&A read-allowlist (§4.9) so the assistant can
answer "what's today's meeting goal?". The four Huddle cards are restyled tighter (smaller headers,
denser rows) toward the app-wide compact direction (ADR-032). jira tools **27 → 31**.

## Consequences

- The Huddle reflects real PR activity automatically while keeping manual control; daily follow-ups
  and the meeting focus are first-class, persisted per sprint, and answerable by the assistant.
- Two more bridge-side stores + env vars (`JIRA_POST_SCRUM_FILE`, `JIRA_MEETING_GOAL_FILE`), both
  git-ignored and keyless-testable; smoke validates the four new tools reject empty input.
- Consistent with ADR-002 (no LLM in servers) and ADR-027 (manual stores for ceremony state).

## Alternatives considered

- **Auto-link ALL open PRs** regardless of sprint. Rejected — noisy; the user wants current-sprint
  focus. (The manual store still covers the long tail.)
- **Reuse the Jira sprint goal for the meeting goal.** Rejected — they answer different questions
  ("what are we delivering this sprint" vs "what's the focus of today's standup").
- **One combined "ceremony notes" store.** Rejected — per-concern stores mirror the existing pattern
  and keep the AI read-allowlist granular.
- **A popup/drawer for the widgets.** The user chose compact always-visible sidebar cards.
