# ADR-078: Granular per-provider credential sharing + admin email edit + self-service password change (v1.67)

## Status: Accepted (2026-07-21) â€” implemented in v1.67

## Context

Three gaps surfaced in the Task Helper's admin console and shared-credentials model (ADR-056),
bundled into one phase because they all touch the same "an admin sets up a user's access" surface:

1. **Sharing was all-or-nothing per user, not per provider.** `getEffectiveConnection` (ADR-056)
   already resolves a shared-credential user's Jira/GitHub/AI independently â€” a user with their
   own Jira but no GitHub already gets their own Jira plus the source's GitHub. What it could NOT
   do was let an admin **restrict** that fallback: today, if `credentialSourceUserId` is set,
   EVERY provider the user hasn't connected themselves silently borrows from the source. An admin
   had no way to say "only share GitHub; if this user hasn't set up their own Jira or AI, show
   those as disconnected â€” don't silently hand them the admin's Jira board." The admin console
   also only exposed each user's OWN connection flags (`connections.{jira,github,ai}` booleans),
   so an admin looking at the user list could not tell "borrowing a token" apart from "genuinely
   disconnected" â€” both showed as `false`.
2. **No email-edit field in the admin console.** `PUT /api/admin/users/:id` has accepted `email`
   since ADR-056 (v1.46) â€” the server side was never the gap â€” but `UserSetupForm` in `Admin.tsx`
   never grew an input for it, so an admin could reset a password or move a user between shared
   credential sources but couldn't fix a typo'd or changed email without direct store access.
3. **No self-service password change.** The Task Helper's `/api/auth/*` routes cover signup,
   login, logout, and `/me` â€” nothing lets the CURRENTLY SIGNED-IN user (any role) change their
   own password. The admin console's "reset password" (ADR-056) resets *someone else's* password
   as an admin action and doesn't require knowing the old one â€” a different operation from a user
   changing their own credential, which should require proving the current password first.

## Decision

**Granular sharing (`StoredUser.sharedProviders?: ConnectionProvider[]`)**: undefined/absent means
"share ALL providers the user doesn't own" â€” the exact legacy behavior, so every existing shared
account keeps working with zero migration. An explicit array restricts fallback-sharing to only
the listed providers; `getEffectiveConnection` checks it right after resolving the credential
source and BEFORE falling back â€” a provider left off the list resolves to `null` (shows as
disconnected) even though the source genuinely has that connection. This is a pure narrowing: a
user's OWN connection for a provider always still wins regardless of `sharedProviders` (unchanged
from ADR-056), and the one-hop-only invariant (a credential source must own its own Jira
connection) is untouched â€” `sharedProviders` restricts WHAT is borrowed, never WHO it's borrowed
from.

**Admin view: own vs inherited vs none.** `userView()` keeps the existing `connections` (own-only
booleans â€” other code may depend on their exact shape) and adds `effective: { jira, github, ai }`,
each a `{ status: "own" } | { status: "inherited"; via: string } | { status: "none" }` computed via
`getEffectiveConnection`. The admin console's per-user connection badges become a 3-state
indicator ("Jira" green = own, "Jira via admin@co.com" amber = inherited, "Jira" muted = none) â€”
same visual language as the old on/off badge, one more state. `sharedProviders` itself is also
surfaced (`null` = share-all) so the console can round-trip the restriction.

**Admin UI: three checkboxes, default all-checked.** Wherever a credential source is picked
(`AddUserCard`, `UserSetupForm`), three checkboxes â€” "Share Jira" / "Share GitHub" / "Share AI" â€”
appear alongside the existing "allow Jira writes" checkbox, each defaulting to CHECKED. `AddUserCard`
omits `sharedProviders` from the create payload entirely when all three stay checked (so the
default path is byte-for-byte the legacy request shape); `UserSetupForm` always sends it as part of
the combined access save (ADR-063's "one Save changes" pattern), using `null` for "all checked" â€”
the same clear-back-to-default convention `credentialSourceUserId: null` already uses in this file.

**Admin UI: email edit.** `UserSetupForm` gains an `Email` input, pre-filled from `user.email`,
folded into the same `accessDirty` tracking and the same single "Save changes" action the form
already uses for `credentialSourceUserId`/`allowWrites` â€” no new save button, no new endpoint (the
server already accepted `email` on `PUT /api/admin/users/:id` since ADR-056).

**Self-service password change: `PUT /api/auth/password`.** Requires an existing session
(`requireAuth`, any role). Body `{ currentPassword, newPassword(â‰Ą8) }`; verifies
`currentPassword` against the stored hash (`verifyPassword`) before accepting the change â€” this is
the operation's entire reason for existing as a DIFFERENT route from the admin's reset
(`PUT /api/admin/users/:id { password }`), which is an admin acting on someone else and has no
"prove you know the old one" step. `401 INVALID_PASSWORD` on mismatch, `400 VALIDATION` when the
new password is under 8 characters. Surfaced in the React app as an "Account" card on the
already-app-wide (not role-gated) Connections page, so both "user" and "admin" roles can reach it.

## Consequences

- An admin can now say "share only GitHub" and the user's Jira/AI correctly show disconnected
  instead of silently inheriting the admin's â€” closing the exact gap reported: granular,
  provider-by-provider trust rather than all-or-nothing.
- The admin console's connection badges are now diagnostic, not just status lights: "inherited via
  X" is visibly different from "not connected," which was previously indistinguishable from the
  outside.
- Zero migration for existing shared-credential accounts: `sharedProviders` absent behaves
  identically to before this phase.
- Fixing the email-edit gap and adding self-service password change round out the admin console's
  user-lifecycle story without inventing new endpoints beyond the one password route â€” both reuse
  either an already-accepting server field (`email`) or a narrowly-scoped new one
  (`/api/auth/password`).
- Tests: mcp-jira 613 â†’ **626** (+13: 5 delegation restriction cases, 4 admin-view own/inherited/
  none + sharedProviders CRUD, 4 self-service password-change route); react-app 950 â†’ **958**
  (+8: 5 Admin console sharing-checkbox/email-edit/inherited-badge cases, 3 Connections "Account"
  card password-change cases).

## Alternatives considered

- **A separate `PATCH /api/admin/users/:id/sharing` endpoint** for `sharedProviders` â€” rejected;
  it belongs with the rest of a user's access fields on the existing combined-save `PUT
  /api/admin/users/:id`, exactly like `credentialSourceUserId` and `allowWrites` already do.
- **Reusing the admin's password-reset endpoint for self-service change** â€” rejected; reset is an
  admin acting on someone else's account and intentionally skips "prove the old password," which
  is precisely the property self-service change must have.
- **A boolean per user ("this user may share AI") instead of a list** â€” rejected; a list is no
  harder to model or render and generalizes cleanly if a fourth provider is ever added, without a
  fourth boolean.
