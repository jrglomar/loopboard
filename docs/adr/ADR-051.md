# ADR-051: Huddle meeting notes — WYSIWYG rich text per sprint

## Status: Accepted (2026-07-04) — implemented; editor + save + render verified live

## Context

The team wants free-form **meeting notes on the Huddle** — deployment notes, runbook links,
reminders — richer than the one-line meeting goal (§4.24). Requirement: a **WYSIWYG** editor
(bold, lists, clickable links), per sprint, visible to everyone at standup.

## Decision

- **Store:** the established per-sprint JSON-store pattern (`meetingNotesStore.ts`,
  `JIRA_MEETING_NOTES_FILE`, default `.loopboard-meeting-notes.json`). Value = an **HTML string**
  + `updatedAt`. Tools `get_meeting_notes` / `set_meeting_notes` (§4.27); empty html clears.
  jira tools 36 → 38; smoke lists updated (48 → 50 checks).
- **Editor:** **TipTap v3** (`@tiptap/react` + StarterKit, which bundles Link in v3 — configured
  with `openOnClick: false, autolink: true`). A thin `RichTextEditor` component with a toolbar:
  bold / italic / strike / H2 / bullet / ordered / link / unlink. Typing a URL autolinks.
- **Safety:** the server stores HTML blindly (no DOM on Node); the app runs **DOMPurify on save
  AND on render**, and a DOMPurify hook forces `target="_blank" rel="noopener noreferrer"` on
  every anchor. The store is treated as untrusted input end-to-end.
- **Placement:** `MeetingNotesCard` on the Huddle sidebar directly under Meeting goal; view mode
  renders sanitized HTML with an "Updated …" stamp; `useMeetingNotes(sprintId)` mirrors
  `useMeetingGoal`.
- **Assistant:** `get_meeting_notes` joins the Q&A read-allowlist (17 read tools) — "what are the
  deployment notes?" now works in chat.

## Consequences

- Two new deps in react-app: TipTap (+ ProseMirror transitives) and DOMPurify. ProseMirror needs
  real-DOM APIs, so **tests stub `RichTextEditor`** (a textarea forwarding HTML); the card's
  sanitize/save/render logic is fully tested in jsdom, the editor itself is exercised live.
- HTML in a JSON file stays greppable and portable; if notes outgrow one blob per sprint
  (attachments, history), that's a new ADR.

## Alternatives considered

- **`contentEditable` + `document.execCommand`** — zero deps but deprecated and quirky
  (inconsistent markup across browsers); rejected for a feature meant for daily use.
- **Markdown textarea with preview** — simpler, but the user explicitly asked for WYSIWYG.
- **Storing ProseMirror JSON instead of HTML** — richer round-trip, but HTML keeps the store
  human-readable and the assistant/export paths trivial.
