# ADR-012: Phase 3 sprint reports — data-first, real velocity, optional AI summary, dependency-light export

## Status: Accepted (2026-06-12)

## Context

Phase 3 of the spec is the sprint report generator/viewer + velocity forecasting. Until
now Reports was a stub, and ADR-010 deliberately left velocity/burndown as *disabled
placeholders* because the bridges couldn't read historical sprint data. With `list_sprints`
+ `get_sprint_report` reading **closed** sprints (ADR-011), that constraint is lifted —
velocity can be computed from real data. The user asked for the full set: per-sprint
report, velocity + forecast, export (copy/Markdown/PDF), and an optional AI summary.

## Decision

1. **Data-first report.** `get_sprint_report(sprintId)` computes everything from the
   sprint's issues: committed vs completed story points, completion rate, completed vs
   carryover lists, blocker count, and a per-assignee breakdown. The report renders fully
   **without AI** — AI is additive, never required (consistent with ADR-004/006).
2. **Real velocity replaces the placeholder.** `get_velocity` averages completed points
   over the last `JIRA_VELOCITY_SPRINTS` (default 6) **closed** sprints and offers a
   forecast = that average, **explicitly labeled a heuristic/"suggested capacity," not a
   commitment** (same honesty boundary as ADR-010's pace chip — but now grounded in real
   history, so it's no longer a faked number). Rendered as a dependency-free CSS bar chart.
3. **Optional AI executive summary.** `POST /api/ai/sprint-summary` (bridge-only,
   dual-provider, same `AI_UNAVAILABLE`/error semantics as the other AI endpoints) drafts a
   review narrative from the report payload. Hidden/disabled when AI is off; AI errors never
   break the data report.
4. **Dependency-light export.** Copy-to-clipboard and `.md` download are built from a pure
   `reportMarkdown.ts` (unit-tested). PDF is **print-to-PDF** via `window.print()` + a
   scoped `@media print` stylesheet — no `jspdf`/charting library added. Good enough for
   sharing in reviews/retros; a programmatic PDF lib can come later if needed.
5. **Scope landed now:** per-sprint report, velocity+forecast, export, AI summary.
   **Deferred (Phase 3+):** burndown chart (needs daily snapshots, which Jira's Agile API
   doesn't expose without greenhopper/historical capture), shareable links, retro board.

## Consequences

- Reports is a real, useful page; velocity is honest and live.
- `get_velocity` makes N+1 issue fetches (one per closed sprint in the window) — acceptable
  for an on-demand report at N=6; if it becomes slow, cache or batch later.
- No new runtime dependency (charts = CSS, PDF = print). Keeps the bundle lean and the
  offline/keyless test posture intact (report math + markdown are pure, unit-tested;
  the AI endpoint is mocked).
- The ADR-010 "disabled velocity/burndown placeholder" is now superseded for velocity;
  burndown remains a labeled future item.

## Alternatives considered

- **Jira greenhopper velocity/sprint-report endpoints** — richer, but unofficial, need
  `rapidViewId`, and vary by instance. Computing from `sprint/{id}/issue` is portable and
  testable. Rejected for the POC.
- **A charting library (recharts/chart.js) + jspdf** — heavier bundle for a few bars and a
  PDF; CSS bars + print-to-PDF meet the need dependency-free. Revisit if the reporting grows.
- **Server-rendered AI summary baked into `get_sprint_report`** — would couple an LLM into a
  data tool and break keyless tests; kept as a separate optional bridge endpoint.
- **Storing report history / burndown snapshots** — needs persistence the POC doesn't have;
  out of scope this round.
