# ADR-057: Personal sprint journal, a dedicated Connections page, and reusable config templates

## Status: Accepted (2026-07-10) — implemented in v1.47

## Context

Three requests landed together, each small on its own but touching the same surfaces:

1. **Task Helper** should carry a developer's **day-by-day notes** and a **to-do checklist** for their
   sprint tickets — the "what did I do today / what's left" companion to the AI prompt builder.
2. **Connections** (Jira/GitHub/AI tokens) were wedged into the top of the Task Helper as step 1. Account
   setup is not a daily task and does not belong in a work surface.
3. **Admins** repeatedly retype the same board/env block for each new user. They want to save a
   configuration once and **reuse** it.

## Decision

### Personal journal (§8.6)

A per-sprint store of `notes: { "YYYY-MM-DD": string }` plus a flat `todos[]`, each to-do carrying its
`sprintId`, `date`, `done`, and an optional `ticketKey` linking it to one of the user's sprint tickets.

**The journal is keyed by the user's REAL id, not `storeUserId`.** This is the one place we deliberately
break from ADR-056's store sharing: leaves/retro/meeting-notes are *team* state and are correctly shared
with the credential owner, but a journal is *personal*. A shared-credential viewer keeps their own notes
rather than writing into the token owner's. Because the Task Helper routes never enter the AsyncLocalStorage
context, `journalStore` simply takes an explicit `userId` and resolves
`.loopboard-user-stores/<userId>/journal.json` — no ALS involvement, and the distinction is impossible to
get wrong by accident.

The UI is one day at a time: a date navigator (‹ / date / › / Today), that day's note, and that day's
to-dos, with a sprint-wide "N open · M done" counter. Dates are computed in **local** time — `toISOString()`
would land on the wrong day for anyone east or west of UTC at the edges.

### Connections page

`ConnectionsPanel` moves to a first-class **Connections** tab. The Task Helper reads readiness from the
app-wide auth context (`context.connections.jira`) instead of owning a status fetch, so the two surfaces
stay decoupled. The onboarding gate (ADR-055) keeps using the same panel.

### Config templates (§9.9)

`configTemplates` in the user store: a named `AdminConfig` bundle. Apply is a **server-side** operation
(`POST …/apply-template`) rather than a UI-side form prefill, so "replace" vs "merge" semantics are
defined in one place, are testable, and can't drift between the two apply targets (a user's overrides and
the global defaults). The config forms remount on apply (`key={…formEpoch}`) so the new values are visible.

`ConfigForm` gained an `idPrefix`: several forms now coexist on the Admin page and were emitting duplicate
DOM ids. Likewise the template pickers name their scope in the buttons' `aria-label`
("Replace dev@team.com config with the selected template") — two identically-named buttons on one page are
ambiguous for screen readers, and the fix made the tests unambiguous too.

## Consequences

- A developer's daily standup material lives next to the ticket they're working on, and to-dos can point at
  real Jira keys without Loopboard writing anything back to Jira.
- Journals never leak across users — including the shared-credential case, which is the subtle one.
- Onboarding a new user is now: create them (optionally on shared credentials), then apply a template.
- Templates are a *copy*, not a live link: editing a template does **not** retroactively change users who
  had it applied. That keeps the §9.2 merge order simple and avoids a hidden fourth layer of config. If
  live-linked templates are wanted later, that's a deliberate follow-up, not an accident.
- Gate: 1,335 tests (mcp-jira 499→524, react-app 741→754, github 57), smoke 53/53, build clean. Pill → 1.47.0.

## Alternatives considered

- **Store the journal under `storeUserId`** (reusing ADR-056's sharing) — rejected: a stakeholder borrowing
  the admin's token would silently write into the admin's private notes.
- **Apply templates purely client-side** (prefill the form, let the admin press Save) — simpler, but
  "merge" semantics would live only in the UI and the two apply targets would drift.
- **Keep connections in the Task Helper** — rejected; the request was explicit and the coupling was real
  (the page owned a connections fetch just to know whether Jira was connected).
