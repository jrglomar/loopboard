# ADR-037: Multiple PO/Dev projects, switchable from the UI

## Status: Accepted (2026-06-26)

## Context

The app is wired to exactly one PO project (`JIRA_PO_PROJECT_KEY` + `JIRA_PO_BOARD_ID`) and one Dev
project (`JIRA_DEV_PROJECT_KEY` + `JIRA_DEV_BOARD_ID`). The team works across **several** PO and Dev
projects and wants to switch between them in the UI, while keys stay env-configurable.

The whole app leans on a strict **PO ↔ Dev pairing** (ticket-gen creates a PO story + linked Dev task;
Linking maps PO→Dev). So "multiple projects" must preserve "there is always an active PO and an active
Dev project," not collapse into one flat board list.

## Decision

**Lists per side, with an active selection.**

- **Config:** new `JIRA_PO_PROJECTS` / `JIRA_DEV_PROJECTS`, each a `KEY:boardId,KEY2:boardId2` list. A
  pure `parseProjects(raw, fallbackKey, fallbackBoardId)` returns `Array<{ projectKey, boardId }>`; when
  the new var is empty it yields a **1-element** list from today's single key+board id — fully
  back-compatible, no config migration required.
- **Health:** `GET /api/health` `.boards` becomes `{ dev: ProjectRef[]; po: ProjectRef[] }`
  (`ProjectRef = { id, projectKey }`); element 0 is the default for that side.
- **React:** `Boards` type → arrays; `boards.ts` validator + `useBoards` updated. `App` (the shared
  board+sprint owner) gains `activeDevIdx` / `activePoIdx` (default 0); the resolved active board id +
  project key = `boards[boardKey][activeIdx]`. The top-right shell keeps the Dev/PO toggle and adds a
  **project dropdown** for the active side. Pages already accept a numeric `boardId` from shared
  context — they just receive the resolved active id; no tool signatures change.
- **Graceful downgrade:** an older bridge that still returns object-shaped `boards` is normalized to a
  1-element list by the client validator.

## Consequences

- Teams can run several PO/Dev projects and switch live; single-project setups are unchanged (the new
  env vars are optional and default to the existing single project).
- One breaking shape change (`health.boards` object → arrays) handled entirely within this phase:
  `boards.ts` + every `boards[key].id`/`projectKey` read moves to the active-index form together.
- No Jira tool signatures change — all board-scoped tools already take `boardId`.

## Alternatives considered

- **One flat board list + a single "Select board" dropdown** (like the reference image). Simpler UI,
  but loses the guaranteed PO+Dev pairing the ticket-gen/linking flows depend on. Rejected.
- **Separate env var per extra project** (`JIRA_PO_PROJECT_KEY_2`, …). Doesn't scale and is awkward to
  parse; a single delimited list is cleaner.
- **Per-board project metadata fetched from Jira.** Unnecessary — board id + key is all the app needs,
  and keeping it pure-config avoids a Jira call on health.
