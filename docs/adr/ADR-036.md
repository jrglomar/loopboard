# ADR-036: Left-sidebar shell + top-right board selector

## Status: Accepted (2026-06-26)

## Context

Follow-up UI enhancements (with a reference Kanban mockup): (1) the Fly-in tracker should show each
ticket's **status** and get **wider/horizontal** visibility; (2) the assistant popup should be
**750px** with a **dim backdrop**; (3) move the app **header into a left sidebar** and put **board
selection in the top-right**, matching the reference.

The board (PO/Dev) selector currently lives **inside** each board-scoped page (Dashboard, Planning,
Reports) and is exercised directly by those pages' tests (clicking "PO"/"Dev"). The shared board
context, however, is already owned by `App` and threaded to pages as controlled props.

## Decision

- **Fly-in**: `FlyInCard` becomes a full-width strip rendered above the board grid in the Dashboard
  (only when there are fly-ins), as a wrap of chips — each with the ticket key (link), summary, a
  tinted **status pill** (by `statusCategory`), and assignee. Still derived from the loaded sprint
  issues; no new data.
- **Assistant**: panel `w-[440px] → w-[750px]`, `shadow-2xl`, and a fixed **scrim** (`bg-black/40`,
  click-to-dismiss) rendered while open; the dialog gets `aria-modal`.
- **Shell**: `App` is restructured to a **left `<aside>` sidebar** (brand + version + a vertical
  `role="tab"` nav inside `<nav aria-label="Main navigation">` — unchanged a11y, so `App.test` passes)
  plus a right column with a **top `<header>` bar**. A single `BoardToggle` lives in the top-bar,
  right-aligned, wired to the shared board context (hidden on Linking, which is dual-board).
- **Single source of board UI**: to avoid two board selectors in the live app *and* keep the page
  tests (which render pages standalone and click the in-page toggle) green, each page renders its own
  `BoardToggle` **only when uncontrolled** (`!onBoardChange`). In the real app the pages are
  controlled by `App`, so they hide their toggle and the shell's selector is the only one; in
  standalone tests the prop is absent, so the in-page toggle still renders and those tests are
  unaffected.

## Consequences

- The app matches the reference layout (sidebar + top-right board picker) and the Fly-in/chat asks,
  with **no contract/tool change** and the full suite still green. Two Dashboard "controlled" tests
  were re-pointed from the (now shell-owned) in-page toggle to prop→hook behavior + an `onSelectSprint`
  assertion (the SprintBoard mock gained a sprint-pick button).
- A small duplication of intent remains (each page still *can* render a toggle for standalone use); it
  is inert in the live app. If pages are later never used standalone, the in-page toggles can be
  deleted and the tests moved to `App.test`.
- The shell is desktop-first (fixed-width sidebar); a future responsive/collapsible sidebar is out of
  scope.

## Alternatives considered

- **Rip `BoardToggle` out of every page** and host only in the shell. More correct long-term but
  churns 3 pages + 4 page test files; the uncontrolled-gate achieves the same live result with far
  less risk.
- **Keep the top header**, just add a sidebar. Rejected — the user explicitly wanted the header moved
  into the sidebar.
