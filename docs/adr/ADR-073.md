# ADR-073: Trends hardening + styled exports, DoD-true aging, Task Helper UI retired (v1.61)

## Status: Accepted (2026-07-18) — implemented in v1.61

## Context

Live-testing feedback on v1.60, in three areas:

1. **Trends & KPIs** threw an error on date ranges ("when some other members or devs are not yet on
   that sprint"). Root cause found by inspection, not the reported theory: the per-dev join was
   already fully guarded — but `sprintIdsInDateRange` had **no cap**, and the user's real board
   keeps ~30 sprints perpetually active, so a wide range selected more than the §4.29 limit of 26
   sprint ids → `get_multi_sprint_report` returned 400 VALIDATION → the error alert. Additionally:
   "Unassigned" (a ticket state, not a person) appeared in the developer listings; the exports had
   no styled treatment; and the user wanted Trends to be the Reports default.
2. **Ticket aging** contradicted the team's own Definition of Done: code-review tickets were aged
   and flagged, but ADR-014 established code review counts as done. And a ticket carried into the
   sprint mid-flight showed its full historical age — alarming for work that is new to *this*
   sprint.
3. **Task Helper** — the user asked to remove the feature and its Guide documentation. Chosen
   depth (AskUserQuestion): **UI-only retirement** — the §7/§8 backend endpoints and stores stay
   dormant; login, Connections, Admin, and `GET /api/me/context` are app-wide infrastructure and
   are untouched.

## Decision

**Trends window capped client-side at the contract limit.** `capSprintWindow` keeps the NEWEST 26
ids (`slice(-26)` over the chronological list) for range and pick modes, with a visible
"showing the latest 26" note (`role="note"`, polite live region) and pick-mode checkboxes
disabling at the cap. The backend contract is unchanged — the client simply never sends an
invalid request. Recent mode was already input-clamped.

**"Unassigned" is filtered from every per-developer surface**: both sides of the kpiAdjust name
union, the developer picker, and the per-assignee sections of the markdown/CSV/xlsx exports. Team
committed/completed totals are untouched — they sum issues, not assignees, so unassigned tickets'
points still count where they are real.

**Two styled workbooks** (xlsx-js-style, the ADR-048 idioms — same palette, border, and
number-format treatment as the sprint-review export): a **team trends** workbook from the export
bar (per-sprint rows + TOTAL and AVERAGE rows) and a **developer KPIs** workbook from the
Developer KPIs section (per dev: header with avg done and met-N-of-M, then per-sprint rows with
Done / Total / Leaves / Target (adj) / Met). Markdown and CSV exports remain — the styled files
are additions, not replacements. Two separate downloads was the user's explicit pick over a
single two-sheet workbook.

**Reports opens in Trends & KPIs mode.** The toggle still returns to the single-sprint report;
tests that exercise sprint mode navigate to it explicitly.

**Aging is DoD-true and sprint-scoped.** Contract §4.3 step 8 amended: `withAging` enriches the
`inprogress` bucket ONLY — code-review issues never fetch changelogs (cost drops to ≤2 calls per
genuinely in-progress issue) and never carry `inProgressSince` (undefined = never touched,
distinct from the enriched-but-unresolvable `null`). The chip is also structurally scoped to the
In Progress column. Display age clamps to the sprint: shown age = days since
`max(inProgressSince, sprint.startDate)` — presentation-side only, so the tool's
`inProgressSince` stays the raw changelog fact and any future "total age" view still has it.

**Task Helper UI deleted; backend dormant.** Removed: the tab, `pages/TaskHelper.tsx`, the
sprint-journal card, `journalClient`/`taskHelperClient`, the Guide section, and the USER-GUIDE
feature section. Kept with reworded copy only: `task-helper/ConnectionsPanel.tsx` and
`LoginForm.tsx` (they power the app-wide login gate and Connections tab — the folder name is now
historical), `authClient`, `connectionsClient`, `Connections.tsx`, `AppGate.tsx`. The
USER-GUIDE's "Enabling the Task Helper" setup section was **retitled, not deleted**
("Enabling accounts (login)") — it is the only documentation of `TOKEN_ENC_KEY`/`SESSION_SECRET`,
which the surviving login gate requires; a compact "Accounts & Connections" blurb replaces the
feature tour.

## Consequences

- Wide date ranges can no longer error; the trade is silent truncation to the newest 26 sprints,
  made non-silent by the note. If a >26 window is ever a real need, the §4.29 cap is the thing to
  revisit — not the client.
- Aging now matches how the team actually defines done, and a carryover ticket reads as "in
  progress since this sprint started", which is the standup-relevant number.
- The journal (daily notes/to-dos) UI is gone with the Task Helper tab; its backend store remains
  readable via the dormant endpoints if the data ever needs recovering.
- The AI assistant's tool surface is unchanged (toolCatalog had no Task Helper references; counts
  hold at 48/43/19).
- Tests: mcp-jira 577 (content rescoped), react-app 911 → **943** (−16 with the deleted TH tests,
  +48: trendsXlsx 17, TrendsView 3, aging +7, sprintRange +6, picker +4, reportMarkdown +3,
  DeveloperKpiSection +3, SprintBoard +2, AgingCard +1, Dashboard +1, Guide +1). Total 1,545 →
  **1,577**; smoke 55/55 (no surface change). Pill → 1.61.0.
- jsdom cannot test "clicking a disabled checkbox does nothing" (fireEvent bypasses the browser's
  disabled gate) — the `disabled` attribute is asserted directly instead, plus an
  unchecking-at-the-cap case.
- **Not live-eyeballed** (login gate) — but this release exists *because* of live testing, and
  each reported symptom maps to a now-pinned regression test.

## Alternatives considered

- **Raise the §4.29 cap** instead of capping the client. The 26 ceiling bounds the backend's
  parallel fan-out (26 × getSprintIssues); the client truncating to the newest window preserves
  that guarantee and fixes the UX without a contract change.
- **Split oversized ranges into multiple backend calls** client-side. Complexity for a window
  larger than any trend question asked so far; deferred until someone actually needs >26.
- **Keep "Unassigned" as a pseudo-dev row.** It made the developer list lie about who the
  developers are; unassigned points remain visible in team totals where they belong.
- **Backend-side sprint-start clamp.** Would overwrite the raw changelog fact; the display clamp
  keeps data honest and the presentation rule swappable.
- **Full Task Helper removal (backend too).** Offered and declined — the user chose UI-only;
  the dormant endpoints cost nothing and keep the journal data recoverable.
