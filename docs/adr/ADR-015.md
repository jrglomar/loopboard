# ADR-015: Velocity is relative to the selected sprint

## Status: Accepted (2026-06-12)

## Context

The v1.4 `get_velocity` always averaged the latest N closed sprints globally. On the
Reports page you pick a sprint to look at — and velocity is most meaningful as "the trend
*leading into* this sprint," not "the most recent sprints regardless of what I'm viewing."
The product owner asked for velocity to use the sprints *before* the selected one.

## Decision

- `get_velocity` gains an optional `beforeSprintId`. When present, it considers only closed
  sprints that come **before** that sprint — excluding the sprint itself and any closed
  sprint whose `completeDate` (fallback `startDate`) is not earlier than the selected
  sprint's `startDate` (fallback `completeDate`) — then takes the first `sprintCount`.
- The Reports `useVelocity` passes the currently-selected sprintId, so the velocity chart
  and the forecast track the picker and refetch on change.
- With no `beforeSprintId`, the prior "latest N closed" behavior is unchanged (backward
  compatible for Copilot / direct tool calls).

## Consequences

- Velocity reads as the run-up to the chosen sprint — the right context when reviewing or
  planning that specific sprint.
- The forecast (and the v1.5 capacity-adjusted "possible committed velocity") become
  sprint-relative, which is what a planner expects when selecting a future/next sprint.
- Ordering relies on `completeDate`/`startDate`; sprints with null dates sort conservatively
  (treated as not-before), so they're excluded from a "before X" window rather than
  guessed into it.

## Alternatives considered

- **Keep global latest-N** — simpler but ignores which sprint the user is analyzing;
  rejected per the explicit request.
- **A separate `get_velocity_before` tool** — unnecessary surface; an optional parameter on
  the existing tool keeps it backward compatible.
- **Compute the window client-side from `list_sprints`** — would duplicate the per-sprint
  point math in the frontend; the server already has it, so the param stays server-side.
