# ADR-033: Approver status of Jira-linked PRs on the Huddle

## Status: Accepted (2026-06-25)

## Context

ADR-031 made the Huddle code-review card auto-list open GitHub PRs whose detected `jiraKeys` are in
the current sprint. The user then asked to also **monitor the approver status** of those linked PRs —
i.e. surface whether each PR is approved, has changes requested, or still needs review, right in the
daily standup view.

`list_prs` (the REST `GET /pulls` list endpoint) does not include review state. GitHub exposes review
state two ways: GraphQL `pullRequest.reviewDecision` (one field, factors in CODEOWNERS/required
reviewers) or REST `GET /pulls/{n}/reviews` (the raw per-reviewer reviews). The existing
`githubClient` is REST/axios-based.

## Decision

A new **batch** read tool `get_pr_reviews { numbers[], repo? }` on mcp-github, called by the Huddle
card with only the PR numbers it is actually displaying (the small set of current-sprint-linked open
PRs) — so review fetching stays targeted, not an N+1 over every open PR.

- `githubClient.listReviews(owner, repo, n)` → `GET /pulls/{n}/reviews` (REST, consistent with the
  rest of the client).
- Pure `summarizeReviews(reviews)` reduces a PR's reviews to a `PrReviewStatus`: each reviewer's
  **latest meaningful** vote wins (`APPROVED` / `CHANGES_REQUESTED` / `DISMISSED` override earlier;
  `COMMENTED` / `PENDING` ignored; `DISMISSED` clears). `decision` = changes_requested → approved →
  review_required, plus an `approvals` count and approving-reviewer logins. Being pure, it is fully
  unit-tested without network.
- The tool fetches each PR's reviews in parallel and **omits** any PR that 404s (resilient, like
  `get_linked_issues`). Read-only — no writes.
- Frontend: a `getPrReviews` client + `usePrReviews(numbers)` hook; `PullRequestsCard` fetches review
  status for its auto-PR numbers and renders a per-PR badge (✓ approved [+count] · ✗ changes
  requested · ⏳ review required). github-down or an unreviewed PR simply shows no badge.

## Consequences

- The standup view shows, at a glance, which linked PRs are cleared to merge vs still blocking — the
  approver status the user asked for, scoped to the sprint's PRs.
- One more mcp-github tool (4 → 5); Copilot can also answer "is PR #123 approved?".
- The REST aggregate is a dependency-free approximation of GitHub's official review decision — it does
  not apply CODEOWNERS or required-reviewer rules. Documented as such; sufficient for a dashboard cue.
- No change to `list_prs`/`get_pr`; the card makes one extra batch call only when it has auto-PRs.

## Alternatives considered

- **GraphQL `reviewDecision`.** Most "correct" (honours required reviewers) and one call for many PRs,
  but introduces a GraphQL path into an otherwise REST client. Deferred — the REST aggregate is enough
  for a glanceable badge and keeps the client uniform.
- **Fold review state into `list_prs` (`withReviews?`).** Would fetch reviews for *every* open PR then
  discard non-sprint ones — wasteful N+1. The batch-by-number tool fetches only what's shown.
- **Per-PR (non-batch) tool called once per row.** More round-trips from the browser; the batch tool
  is one request for all displayed PRs.
