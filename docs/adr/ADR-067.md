# ADR-067: The Guide documents the MCP tools — how-to + full 47-tool reference (v1.56)

## Status: Accepted (2026-07-13) — implemented in v1.56

## Context

Loopboard's 47 MCP tools, split across `mcp-jira` (42) and `mcp-github` (5), are the product's
headline capability — everything else in the app is a UI over them. They're reachable two ways:
VS Code Copilot Chat gets all 47 over stdio, and the in-app floating assistant gets a curated
18-read / 7-propose-with-confirm subset (ADR-029/030). Yet the in-app Guide (`pages/Guide.tsx`,
ADR-060) had zero MCP content — a user landing on the Guide had no way to discover what they could
ask Copilot for, which tools have no dashboard button at all, or which of the assistant's answers
came from a read vs. required confirmation. The user asked for a **full reference**, not a
highlights list, so people can maximize what the tools can do.

## Decision

- **Two new `SECTIONS` entries**, inserted after `"assistant"` and before `"tips"` (11 → 13
  sections). `"mcp-tools"` ("Using the MCP tools") explains the two paths to the tools — Copilot
  Chat (all 47, including the three with no dashboard surface: `sync_pr_links`, `get_pr`,
  `get_pr_reviews`) and the floating assistant (18 read / 7 propose, confirmation-gated, no
  GitHub tools) — plus the Jira/GitHub-vs-Local-store distinction and the shared-credentials
  read-only caveat (ADR-056). `"tool-reference"` ("Tool reference") is nine grouped tables
  covering all 47 tools: Tool / What it does / Type badge / AI / Used in the app.
- **A typed data module, not hand-written JSX.** `packages/react-app/src/lib/toolCatalog.ts`
  exports `TOOL_CATALOG` (47 entries) and `TOOL_GROUPS` (9 groups), rendered by page-local
  `ToolGroupTable`/`ToolTypeBadge` components in `Guide.tsx`. A typed array — not 47×5 hand-built
  table cells — is what lets unit tests assert counts, name-uniqueness, and per-group totals
  instead of trusting hand-authored JSX to stay in sync.
- **Hand-synced `aiAssistant` flags, on purpose.** Each catalog entry's `aiAssistant` field
  (ask / propose / none) is a deliberate duplicate of `askService.ts`'s `READ_TOOLS`/`WRITE_TOOLS`
  allowlists, hand-copied rather than imported, because `react-app` must not pull `mcp-jira`'s
  server-only code into the browser bundle (the module boundary in `ARCHITECTURE.md` §7.2).
  `toolCatalog.test.ts` keeps literal copies of the three allowlists it must track (`READ_TOOLS`
  18, `WRITE_TOOLS` 7, and `delegation.ts`'s `JIRA_WRITE_TOOLS` 8) and asserts set-equality
  against the catalog, so a future allowlist edit that forgets the catalog fails a test instead of
  silently going stale.
- **Reuse over new primitives.** Table rendering reuses the `overflow-x-auto` table pattern
  already proven in `Reports.tsx` and the existing info/warning semantic color tokens for the two
  badge families. No new UI primitives, no `App.tsx` change — this is new content on an existing
  page, not new surface.
- **No `CONTRACTS.md` change.** This feature adds zero new tools, routes, fields, or env vars —
  it's a user-facing gloss of tools that are already fully specified in `CONTRACTS.md`. The Guide
  was never a contract surface (the same reasoning ADR-060 used for the original Guide page), so
  documenting existing tools on it doesn't touch the contract.

## Consequences

- The Guide is now the discovery surface for all 47 tools — a user can go from "what can I ask
  for" to a concrete Copilot prompt or an assistant question without leaving the app.
- The catalog↔`askService` drift risk is the same accepted class of risk as ADR-060's
  Guide↔`USER-GUIDE.md` drift: two hand-maintained descriptions of one truth. It's mitigated here
  (unlike ADR-060's, which stays accepted-and-unmitigated) by the anti-drift tests in
  `toolCatalog.test.ts` — a changed allowlist without a matching catalog update fails CI.
  `docs/USER-GUIDE.md` was updated in parallel, per ADR-060's maintenance note that both copies
  should change together.
- Gate: typecheck + build clean, react-app 777 → 793 tests (toolCatalog.test 11 new, Guide.test 3 → 8),
  mcp-jira 543, github 57 → **1,393 total**, smoke 55/55 (no server edits). Pill → 1.56.0.

## Alternatives considered

- **Hand-written JSX rows for all 47 tools.** What ADR-060's original section bodies do for prose,
  but 47 rows × 5 columns of hand-built table markup is unreviewable and untestable — no way to
  assert a tool wasn't dropped or miscounted except eyeballing the diff.
- **Import `askService.ts`'s allowlists directly into the catalog.** Would guarantee zero drift,
  but `askService.ts` is server-only code (Node built-ins, the AI provider port) — importing it
  into `react-app` breaks the browser/server module boundary and pulls Node-only code into the
  Vite bundle. The hand-synced-plus-tests approach was chosen specifically to avoid this.
- **One flat 47-row table.** Simpler to build, but unscannable at that length and ignores the
  user's explicit choice of grouped tables mirroring the tool's functional area.
- **Document only the highlights (a handful of example tools).** Rejected by the user — they
  asked for a full reference so people can maximize the tools, not a teaser.
