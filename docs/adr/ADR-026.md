# ADR-026: Ticket-level actions on the Planning list — status transition + move sprint

## Status: Accepted (2026-06-24)

## Context

During grooming/planning the team needs to (a) advance a story through its workflow (e.g. move a PO
story To Do → In Progress → Done) and (b) move a ticket from one sprint to another. The Planning
page already has a per-ticket table (`AssignmentList`) with an assignee dropdown per row — the
natural home for these actions (the user chose "Planning ticket list" over the board cards).

Two Jira facts shape the design:

- **Status is not a writable field.** A Jira status change goes through a **transition**, and the
  set of valid transitions depends on the issue's current status and the project workflow. So the
  available transitions must be fetched per issue, not assumed.
- **Moving to a sprint already has an endpoint.** `addIssuesToSprint(sprintId, keys)`
  (`POST /rest/agile/1.0/sprint/{id}/issue`) already exists (used by create-with-sprint); adding an
  issue to a sprint moves it out of any prior sprint, so "move" needs no new Jira call — only a thin
  tool wrapper.

## Decision

### `get_transitions` (READ) + `transition_issue` (WRITE)

- `get_transitions({ ticketKey })` → `GET /rest/api/3/issue/{key}/transitions` →
  `{ ticketKey, transitions: [{ id, name, to: { name, category } }] }`.
- `transition_issue({ ticketKey, transitionId })` → `POST .../transitions { transition: { id } }`,
  then re-read the issue → `{ ticketKey, status, statusCategory }`.
- New jiraClient `getTransitions` + `transitionIssue`; 404 → UpstreamError.

### `move_issue_to_sprint` (WRITE)

- `move_issue_to_sprint({ ticketKey, sprintId })` → reuses `addIssuesToSprint(sprintId, [ticketKey])`.
  Returns `{ ticketKey, sprintId }`.

### Frontend — `AssignmentList`

- An **assignee filter** above the table (reuses the native-`<select>` pattern from `SprintBoard`)
  with a **"N of M · P pts"** summary (sum of `storyPoints` over the filtered rows).
- A per-row **Status** dropdown showing the current status; it **lazy-loads** the issue's
  transitions on first open (so we don't fire N transition reads up front), then applies the chosen
  one optimistically and refetches the sprint.
- A per-row **move-to-sprint** dropdown of the board's active+future sprints (passed down from
  `Planning`, which already loads them), applying `move_issue_to_sprint` then refetching.

## Consequences

- The Planning list becomes a light "sprint ticket manager": assign, set status, move sprint, filter
  — without leaving the page. Copilot also gains generic status/move tools.
- Transitions are fetched lazily per row, so a large sprint doesn't pay N upfront reads.
- These are **real Jira writes**; they're exercised against real tickets only with explicit user
  go-ahead. Tests mock the Jira client and stay keyless/offline.

## Alternatives considered

- **A status field write** (e.g. `update_ticket { status }`). Rejected — Jira rejects direct status
  writes; transitions are mandatory and workflow-dependent.
- **A dedicated move endpoint / backlog support.** `addIssuesToSprint` already moves between sprints;
  backlog moves (`POST /rest/agile/1.0/backlog/issue`) can be added later if needed — out of scope.
- **Put the actions on the Huddle board cards.** The user chose the Planning list; the board stays a
  read-only daily view.
