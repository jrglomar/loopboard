# ADR-030: Floating assistant + chatbot write-actions with modal confirmation

## Status: Accepted (2026-06-24)

## Context

The v1.18 AI assistant (ADR-029) answers read-only questions, embedded in the Huddle sidebar. The
user wants it to (a) be a **floating chatbot** available everywhere, and (b) **perform actions** —
"update points of VRDB-2700 to 2pts", "move VRDB-2700 to next sprint", "create a new sprint" — but
with a **confirmation** before anything writes (fetches still answer directly).

Constraints: the assistant must never silently mutate Jira; the user chose a **modal** confirmation;
and there's already a tool-calling loop (`askService.ts`) plus write tools (`update_ticket`,
`move_issue_to_sprint`, `create_sprint`, `transition_issue`, `set_sprint_goal`, `assign_issue`).
`update_ticket` doesn't yet set story points, though the create path already knows the field.

## Decision

### Floating widget

Move `ChatPanel` into a global `AssistantWidget` — a FAB at the lower-right that toggles a popup
panel, mounted once in `App.tsx`. Lazy-mounts the panel (and its context fetches) on first open and
keeps it mounted after, so the conversation persists. Removed from the Huddle sidebar.

### Write-actions via "propose → confirm → execute"

The AI **proposes**; the human **confirms** in a modal; the **frontend executes**. The AI never
writes.

- `update_ticket` (+ `jiraClient.updateIssue`) gains optional `storyPoints` → `JIRA_STORY_POINTS_FIELD`.
- `askService` offers READ tools **and** a curated `WRITE_TOOLS` set. Read tool_calls run in-process
  as before. A **write tool_call is not executed** — the loop returns `proposedAction: { tool, args }`.
  Reads can precede a write (e.g. `list_sprints` to resolve "next sprint", then propose
  `move_issue_to_sprint` with the resolved id), so args arrive fully resolved.
- `/api/ai/ask` output gains optional `proposedAction`. The frontend (`AssistantWidget`) opens a
  modal `ConfirmActionDialog` — a summary + Confirm/Cancel for simple actions, an **editable form**
  for `create_sprint` (reuse `CreateSprintDialog`'s fields). On confirm it executes the write via the
  existing tool (`callTool`) and reports the result in the chat; cancel dismisses.

## Consequences

- The assistant can both inform and act, while every mutation is human-gated behind a modal — no
  silent writes. Read questions stay one-shot.
- The AI's role is unchanged in spirit: it plans + fills arguments; execution is an explicit user
  action in the UI (consistent with ADR-006's "AI on the bridge, human/UI in control").
- New: `proposedAction` on `/api/ai/ask`; `storyPoints` on `update_ticket`; `ConfirmActionDialog`.

## Alternatives considered

- **Let the AI execute writes directly** (just expose write tools to the loop). Rejected — unsafe;
  the user explicitly wants confirmation.
- **Inline confirm card in the chat** (vs modal). The user chose a modal dialog; it also gives room
  for the create-sprint form.
- **A separate "intent classifier" endpoint.** Unnecessary — the existing tool-calling loop already
  surfaces the intended tool + args; we just intercept writes instead of running them.
