# ADR-022: Dedicated Linking page — bulk PO→Dev ticket creation with AI planning

## Status: Accepted (2026-06-17)

## Context

v1.10 (ADR-021) added a single-PO `LinkDevTicketCard` on the Planning page: create one Dev
task linked to one existing PO story. The user wants to scale this up and give it room:

- **Separate page.** Move linking off the busy Planning page onto its own tab.
- **Bulk.** Select **multiple** PO stories and create a Dev task for each, all into one target
  Dev sprint.
- **See what already exists.** For each selected PO, show its existing linked Dev ticket(s)
  ("one or none") so we don't create duplicates.
- **AI plan.** Have the AI propose the Dev tasks for all selected POs in one go.
- **Status log.** Report which Dev tickets were created successfully (and which failed).

## Decision

### A new `Linking` page/tab (`Dashboard · Planning · Linking · Reports`)

A guided workflow in `pages/Linking.tsx`:

1. Pick a **PO board sprint** and the **target Dev sprint** (`list_sprints` per board).
2. **Multi-select** the PO sprint's tickets (`get_active_sprint(po.id, sprintId)`), each row
   badged with its existing linked Dev ticket from the new `get_linked_issues` tool. Rows that
   already have a Dev link are **deselected by default** (avoid duplicates) but still selectable;
   a "select all without a Dev link" helper.
3. **Generate the plan with AI** (`POST /api/ai/plan-dev-tickets`) → one editable Dev draft
   (`devSummary` + `devDescription`) per selected PO. AI-off → deterministic template per PO.
4. **Create all** → the client **loops** `create_dev_ticket({ summary, description,
   linkedPoTicketKey, sprintId })` per item, rendering a live **status log** (⏳→✓ `DEV-xxx` /
   ✗ error) and a final "N created, M failed" summary.

The single-PO `LinkDevTicketCard` (v1.10) is **removed** — the bulk page subsumes it (selecting
one PO is the single case).

### Two new backend surfaces

- **`get_linked_issues` (MCP tool).** `{ keys[], projectKey? }` → `{ links: { poKey:
  LinkedIssue[] } }`. For each PO key, read `fields=issuelinks` and keep the linked issues in
  the Dev project (default `JIRA_DEV_PROJECT_KEY`). Fetches run in parallel; one bad key yields
  `[]`, never throws. New jiraClient `getLinkedIssues(key)`.
- **`POST /api/ai/plan-dev-tickets` (bridge-only).** `{ poStories[], instructions? }` → one
  provider call returning `{ items: [{ poKey, devSummary, devDescription }], assistantMessage }`.
  Lives in `lib/ai/draftService.ts` (`planDevTickets` + `PlanDevTicketsOutputSchema`, zod/v4),
  reusing the existing `AiProvider` dispatch. NEVER an MCP tool (circular for Copilot, ADR-006).

### Bulk-create stays client-side (no new bulk tool)

Looping the existing `create_dev_ticket` keeps each PO's result independent, which is exactly
what a per-item status log needs, and reuses the already-non-fatal link/sprint handling. A
server-side bulk tool would either be all-or-nothing or re-implement the same per-item result
shape for no real gain at POC scale.

## Consequences

- One AI call drafts the whole plan (cheaper + more coherent than N `draft-tickets` calls); the
  UI reconciles `items` to selected POs by `poKey` and falls back to a template for any omitted.
- `get_linked_issues` is generic (any project filter) and reusable beyond this page.
- Bulk create is N sequential writes; the log shows progress. Partial failure is fine and
  visible — the user can retry the failed rows.
- `create_dev_ticket` and the v1.10 `createLinkedDevTicket` client wrapper are reused unchanged.

## Alternatives considered

- **Keep linking on Planning, add a bulk mode there.** Rejected — Planning is already dense
  (context, team, new sprint, ticket gen, leaves, assignment); linking deserves its own space.
- **A server-side `bulk_create_dev_tickets` tool.** Rejected — loses clean per-item status,
  duplicates `create_dev_ticket`, and adds atomicity questions for no POC benefit.
- **Loop `draft-tickets` per PO for the plan.** Rejected — N round-trips and the model can't see
  the set as a whole; a single `plan-dev-tickets` call is faster and more coherent.
- **Derive existing links from `get_active_sprint` data.** Rejected — `IssueSummary` carries no
  issue links; a focused `get_linked_issues` (fields=issuelinks) is the right read.
