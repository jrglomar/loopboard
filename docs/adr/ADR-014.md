# ADR-014: Definition of Done = "Done OR In Code Review" (reports + dashboard)

## Status: Accepted (2026-06-12)

## Context

The team's Definition of Done treats a ticket in **Code Review** as effectively complete —
the work is finished and awaiting review, so for progress/velocity purposes it should count
as done. The v1.4 report and the dashboard progress counted only the strict `done`
status category, so review work looked "not done," understating completion. The product
owner asked to treat done-or-code-review as completed in the report completion summary and
on the dashboard.

## Decision

1. **One DoD predicate:** an issue is *completed* if `statusCategory === "done"` **OR** it
   matches the existing code-review detection (`JIRA_CODE_REVIEW_STATUSES`, §4.3).
2. **Reports** (`get_sprint_report`): `completedPoints`/`completedCount`, the `completed`
   list, and `byAssignee.donePoints`/`doneCount` all use this predicate. Carryover =
   everything else. `get_velocity` uses the same predicate for its per-sprint completed
   points.
3. **Dashboard:** `get_active_sprint.totals` gains `storyPointsCodeReview`; the progress
   bar and pace chip treat `storyPointsDone + storyPointsCodeReview` as completed. The
   **board still shows Code Review as its own column** — this changes *measurement*, not
   the board's bucketing (Code Review remains visible and distinct).

## Consequences

- Completion %, by-assignee done points, velocity, and the dashboard pace now reflect the
  team's real DoD; review work no longer reads as incomplete.
- `storyPointsDone` keeps its strict meaning (the Done column total); the DoD-completed
  number is `storyPointsDone + storyPointsCodeReview`, computed where progress is shown.
  Adding a separate field (rather than redefining `storyPointsDone`) avoids breaking the
  Done-column count and keeps the change explicit.
- Velocity over closed sprints is largely unaffected in practice (closed sprints rarely
  retain code-review issues), but the predicate is applied uniformly for correctness.

## Alternatives considered

- **Redefine `storyPointsDone` to include code review** — would conflate the Done-column
  count with DoD-completion and silently change an existing field's meaning. Rejected in
  favor of an explicit `storyPointsCodeReview` companion.
- **Dashboard unchanged, reports only** — the owner explicitly wanted both; DoD is a
  team-wide definition, so applying it only to reports would be inconsistent.
- **Make the DoD statuses a separate env list** — unnecessary; the code-review status list
  already captures "in review," which is exactly the DoD addition.
