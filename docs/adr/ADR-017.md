# ADR-017: Separate PO and Dev boards (dashboard, ticket targeting, reports)

## Status: Accepted (2026-06-15)

## Context

The app has always been Dev-board-centric: the dashboard, sprint selection, and reports all
used `JIRA_DEV_BOARD_ID`. But the team also runs a **PO board** (`JIRA_PO_BOARD_ID` /
`JIRA_PO_PROJECT_KEY`) with its own sprints, and the product owner wants to:
1. View the PO board's sprint board on the dashboard, just like the Dev board.
2. In TicketGen, target a **PO board sprint** for the PO story and a **Dev board sprint**
   for the Dev task when creating the linked pair.
3. See **separate PO reports** (report + velocity) alongside Dev reports.

All board-scoped tools already accept `boardId`, so this is mostly a matter of letting the
UI choose a board — but the frontend doesn't know the configured board ids (they're env on
the server).

## Decision

1. **Expose board config via health.** `GET /api/health` (mcp-jira) returns
   `boards: { dev: { id, projectKey }, po: { id, projectKey } }` from config — no Jira call.
   The app reads it once (`boards.ts` / `useBoards()`); no tool signature changes.
2. **Board toggle (Dev/PO) on the Dashboard and Reports**, default Dev. The selected board's
   numeric `id` is threaded as `boardId` into every board-scoped call on that page
   (`get_active_sprint`, `get_daily_huddle`, `list_sprints`, `get_sprint_report`,
   `get_velocity`, `create_sprint`). Switching boards resets sprint selection + filters.
3. **Graceful no-sprint boards.** A PO board may be Kanban (no sprints); the
   "No active or future sprint" upstream case maps to a friendly empty state, not an error.
4. **TicketGen targets both boards.** Two "Add to sprint" selects — PO sprint (from
   `list_sprints` on `boards.po.id`) for the PO story, Dev sprint (on `boards.dev.id`) for
   the Dev task. The linked pair is created with each ticket's own `sprintId`. Falls back to
   the single Dev select when `boards` is absent (older bridge).

## Consequences

- The dashboard and reports become board-aware; PO and Dev are first-class and separate.
- Zero new tools — `boardId` already existed on every relevant tool; only health grew and
  the UI gained a selector. Backend change is tiny and keeps the stateless/keyless posture.
- Linked PO+Dev creation can now place each ticket in the right board's sprint in one go,
  matching how the team plans (PO stories on the PO cadence, Dev tasks on the Dev cadence).
- A ticket only joins a sprint whose board filter includes its project; the existing
  non-fatal `sprintWarning` covers mismatches (e.g. trying to add a Dev task to a PO sprint).

## Alternatives considered

- **A `board: "po"|"dev"` alias param on every tool** — would resolve to the configured id
  server-side, but changes six tool signatures for no real gain; exposing the ids in health
  and passing `boardId` (which already exists) is smaller and backward compatible.
- **A separate PO dashboard/route** — more surface and duplicated UI; a board toggle reuses
  the existing board/report views with one piece of state.
- **Hardcoding board ids in the frontend** — leaks env config into the client and breaks
  per-environment configuration; health is the right channel.
- **Auto-detecting boards from Jira** (list all boards) — unnecessary; the two boards the
  team cares about are already configured, and listing/▒choosing arbitrary boards is scope
  creep for this round.
