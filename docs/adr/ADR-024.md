# ADR-024: Shared board+sprint context across tabs + first-class sprint goal

## Status: Accepted (2026-06-17)

## Context

From the Scrum-Master review of Loopboard, the two highest leverage-per-effort gaps:

1. **Sprint context didn't carry across tabs.** Dashboard, Planning, and Reports each owned
   their own `selectedBoardKey` / `selectedSprintId` and re-defaulted independently, so a Scrum
   Master working one sprint across the ceremonies had to re-pick the board (and sometimes the
   sprint) on every tab.
2. **The sprint goal wasn't trackable.** `create_sprint` could set a goal, but there was no way
   to edit it later, and the Dashboard — the at-a-glance surface — never showed it. The goal is
   the sprint's north star; it should be visible and current.

## Decision

### Shared board + sprint (lifted to `App`, threaded as optional controlled props)

`App` owns `selectedBoardKey: BoardKey` and a shared `sprintId: number | null`, and passes
`{ boardKey, sprintId, onBoardChange, onSprintChange }` to Dashboard / Planning / Reports.

- **Controlled when the props are present, uncontrolled otherwise.** A page rendered standalone
  (e.g. in its existing unit tests) keeps its own internal state — zero test churn — while under
  `App` it uses the shared state.
- **The shared sprint is an *explicit pick*, not a forced value.** Each page shows
  `sharedSprintId ?? itsOwnPerCeremonyDefault` (Dashboard → active, Planning → next future,
  Reports → latest closed). So the sensible per-ceremony defaults are preserved, but the moment
  the user explicitly picks a sprint it **follows them across tabs**. A page's computed default is
  kept in local state and is **never written back** to the shared sprint (that would contaminate
  the other ceremonies' defaults) — only an explicit user pick calls `onSprintChange`.
- **Changing the board resets the shared sprint to `null`,** so each page re-defaults for the new
  board.
- **Linking is excluded** — it is a dual-board tool (PO source + Dev target) and keeps its own
  selectors.

### `set_sprint_goal` (WRITE) + goal UI

- New MCP tool `set_sprint_goal({ sprintId, goal })` → `POST /rest/agile/1.0/sprint/{id}` with
  `{ goal }` (the Agile API treats POST as a partial update). New jiraClient `updateSprintGoal`.
- **Planning** gets an inline goal editor (textarea + Save) — where the SM keeps the goal current.
- **Dashboard** gets a goal banner: the goal + `% points done (DoD)` + `days left`, all from the
  already-loaded `get_active_sprint` data (no extra fetch). No goal → a "set one in Planning" hint.

## Consequences

- One coherent sprint workspace: pick the board (and optionally the sprint) once, and it follows
  you Dashboard → Planning → Reports.
- Per-ceremony defaults are retained, so Planning still opens on the next sprint to plan unless
  the SM has chosen otherwise.
- The goal is now a tracked, visible artifact; `set_sprint_goal` is also available to Copilot.
- The optional-props pattern means no existing page test needed changing; new tests cover the
  controlled wiring and the goal flows.

## Alternatives considered

- **React Context for shared state.** Cleaner call sites, but a missing provider throws in the
  standalone page tests; optional props keep tests trivial and the data flow explicit.
- **Force a single shared sprint everywhere (no per-ceremony default).** Simpler, but regresses
  Planning's "next sprint to plan" default; the explicit-pick-overrides-default rule keeps both.
- **A general `update_sprint` (name/dates/goal).** Broader surface than needed now; a focused
  `set_sprint_goal` matches the feature and is clearer for Copilot. Can generalise later.
- **Write the page default back to the shared sprint.** Rejected — it would leak one ceremony's
  default into the others.
