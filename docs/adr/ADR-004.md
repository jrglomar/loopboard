# ADR-004: Deterministic Server-Side Logic — No LLM Calls Inside MCP Servers

## Status: Accepted

## Context

The MCP servers (`mcp-jira`, `mcp-github`) receive tool calls from Claude (via Copilot) and return structured data. A tempting implementation pattern is to have the servers themselves call an LLM API (e.g., Claude via `@anthropic-ai/sdk`) to generate content — for example, to produce the `summaryText` in `get_daily_huddle`, to write acceptance criteria in `create_po_ticket`, or to "enhance" a description in `update_ticket`.

This pattern is common in LLM application frameworks but creates several concrete problems for this system:

1. **Testability**: The spec quality gate requires `npm run test` to pass with no `.env` and no network access. Any tool handler that calls an LLM API at runtime cannot be unit-tested without mocking the Anthropic client — adding a test dependency on the Anthropic SDK, API key shapes, and response formats.

2. **Key management**: An Anthropic API key would be needed in the MCP server's `.env`. The spec has no provision for `ANTHROPIC_API_KEY`; adding it contradicts spec §8's explicit decision not to use a direct Claude API integration.

3. **Non-determinism**: LLM outputs are stochastic. If the MCP server generates content via LLM, the same input can produce different tool outputs on different calls. This makes integration testing unreliable (no snapshot tests) and makes debugging difficult ("why did the digest say X today but Y yesterday?").

4. **Latency and cost**: The MCP server is already making one or more Jira/GitHub REST calls. Inserting an LLM call in series multiplies total latency and introduces per-call API cost. For a POC used by a small team, this is wasted overhead.

5. **Role confusion**: Claude is the *caller* of these tools, not a dependency of them. The architecture is: Claude → MCP server → Jira/GitHub. Inserting Claude into the middle of that chain (MCP server → Claude → MCP server's response) creates a circular dependency and defeats the separation of concerns.

The `summaryText` field in `get_daily_huddle` could be a candidate for LLM generation. However, its purpose is to be a structured digest that Claude then presents with additional narrative. A deterministic digest (counts, blocked call-outs, sprint name and dates) is sufficient — Claude's narrative layer adds the polish.

## Decision

**No MCP server handler calls any LLM, AI, or generative API at runtime.**

All content generation within the servers is deterministic:

- `summaryText` in `get_daily_huddle` is constructed by a pure template function using sprint data: sprint name, dates, issue counts, blocked issue keys.
- `textToAdf()` in `adf.ts` is a deterministic converter (paragraph/heading/bullet) with no AI logic.
- `src/lib/prompts.ts` contains **string templates** (prompt text that Claude receives as instructions). These are pure functions returning strings — they do not invoke Claude; they are *inputs* to Claude registered via `server.registerPrompt()`.
- `jiraKeys.ts` uses a regex, not an LLM, for Jira key detection.
- Ticket templates in `react-app/src/lib/ticketTemplates.ts` produce structured scaffolds (Given/When/Then, technical checklist) deterministically from user input.

The only intelligence in the system is Claude (via Copilot), which is external to all three packages.

## Consequences

**Positive:**
- `npm run test` passes with no `.env` and no network access for all packages — quality gate met.
- No `ANTHROPIC_API_KEY` or additional LLM credentials needed.
- Tool outputs are fully deterministic and snapshot-testable.
- Handlers are fast: latency is bounded by Jira/GitHub REST API round-trip time only.
- The architecture is coherent: Claude is always the orchestrator, never a hidden sub-call.

**Negative / trade-offs:**
- `summaryText` in `get_daily_huddle` is a structured digest paragraph, not a polished natural-language summary. Claude adds the polish when presenting to the user via the `daily_huddle` prompt template.
- Ticket templates in `ticketTemplates.ts` produce a scaffold, not a fully written story — the user (or Claude via Copilot) fills in the specifics. This is the intended workflow.
- Any future desire to add server-side AI content generation requires revisiting this ADR.

## Alternatives considered

### Alternative A: Call Claude SDK in `get_daily_huddle` to generate the summary text

Use `@anthropic-ai/sdk` to produce a natural-language standup summary from sprint data inside the handler.

**Rejected because:** Breaks the testability quality gate (requires API key + network for tests). Adds non-determinism. Creates a circular role (Claude calls a tool that calls Claude). Adds latency and cost. Violates spec §8.

### Alternative B: Call a local LLM (Ollama) server for content generation

Use a locally running open-source model (no API key required) for content generation in handlers.

**Rejected because:** (a) Introduces a fourth infrastructure dependency not in the spec. (b) Still non-deterministic. (c) Still requires network access in tests (even if `localhost`). (d) Output quality from local models at POC scale is variable. The spec does not mention this option.

### Alternative C: Pre-compute templates at startup with a one-time LLM call

Generate templates from Claude at server startup (not per-request), cache them in memory.

**Rejected because:** Still requires an API key and network access. The templates would need to be regenerated on restart. The deterministic approach (literal string templates) is simpler, faster, and requires no credentials.
