# ADR-049: Header navigation returns, full-width compact shell

## Status: Accepted (2026-07-03) — implemented (react-app, presentation-only)

## Context

v1.24 (ADR-036) moved navigation into a left sidebar. In practice the sidebar costs ~224px of
horizontal space on every page while holding only five links, and the main column was further capped
at `max-w-[1400px]` — on wide monitors the dashboard/board/tables floated in a centered band with
large dead margins. The user asked to put navigation **back in the header**, **maximize the usable
width**, and keep the UI **compact**, on all pages.

## Decision (App.tsx shell only; pages inherit)

- **One sticky top header** (h-12): brand → horizontal nav tabs → right-aligned board/project
  controls + version pill. The v1.24 top-bar's board controls merge into this single bar (still hidden
  on Linking, which is dual-board). Nav keeps the `role="tablist"`/`role="tab"` semantics,
  `aria-pressed`/`aria-current`/`aria-selected`, and the `Main navigation` landmark asserted by
  App.test; the tab row is `overflow-x-auto` so all five tabs stay reachable on narrow viewports.
- **Full width**: `<main>` drops `max-w-[1400px] mx-auto` → `w-full px-3 sm:px-5 py-4`. Because every
  page renders inside this one container, the change applies app-wide with no per-page edits.
- **Compact**: header 56→48px, nav items text-[0.8125rem] px-2.5 py-1.5, main padding py-5→py-4 —
  on top of the existing v1.20 compact pass (ADR-032).

## Consequences

- Pages gain ~224px (sidebar) + whatever the max-w cap was clipping on wide screens; the 4-column
  sprint board and the wide Planning/Reports tables benefit the most.
- The shell keeps the same shared-context wiring (boardKey/sprintId/projectIdx) — no page changes,
  no test-visible behavior changes beyond the layout; App.test still passes unchanged.
- BoardToggle stays shell-controlled (pages render their own toggle only when uncontrolled, per v1.24).

## Refinement (2026-07-04) — after a live full-screen review

Screenshots of the running app at ~1570–1920px showed the shell was sound but **wide sparse
internals** read badly: narrow tables stretched edge-to-edge while truncating their own labels.
Fixes, verified live via HMR:

- `<main>` gains a **soft `max-w-[1800px]` centered cap** — invisible on normal monitors, stops
  ultrawide stretch.
- Leaves calendars (`LeavesCalendarCard`, `LeavesPlannerCard`): `w-full` tables → **`w-auto`**
  (day cells cluster naturally instead of spreading a meter apart); assignee name truncation caps
  raised 80–90px → 200px with a 170px min column (full names now visible); the planner's stray
  `px-12` gutter → `px-4`.
- Per-developer capacity table → `max-w-xl`; Reports sprint `<select>` → `max-w-md` (was
  stretching ~1,480px via flex-column stretch).
- **Shrink-wrapped cards** (second pass — capping tables alone left the WHITE CARD spanning full
  width behind them): the leaves/capacity plotter card, the Offset Tracker leave-planner card, and
  the offset-points card are now `w-fit max-w-full` — each card ends where its content ends and the
  page background fills the rest. (The offset table's interim `max-w-5xl` was dropped as redundant.)

## Alternatives considered

- **Collapsible sidebar.** Rejected — two chrome regions for five links; the header does it in one.
- **Keep the max-width cap.** Rejected — the user explicitly wants edge-to-edge; data-dense tables
  read better with the space, and px padding keeps line lengths sane in text-heavy cards.
