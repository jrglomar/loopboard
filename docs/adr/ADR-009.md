# ADR-009: Adopt Tailwind CSS + shadcn/ui for the dashboard (supersedes the "no framework" rule)

## Status: Accepted (2026-06-11)

## Context

The v1.0 contract (§6) and the original build deliberately used hand-rolled CSS with a
custom-property token set — "no UI framework, no Tailwind" — to keep the POC dependency
surface minimal. After live use, the product owner asked for a full UI overhaul built on
**shadcn/ui + Tailwind CSS**, delivered in two phases: (1) component-level migration of
the existing UI to shadcn primitives with behavior preserved, then (2) a senior-UI
modernization audit and pass. This is a user-directed reversal of the earlier decision,
recorded here per the contract-first rule.

## Decision

1. Add **Tailwind CSS** and **shadcn/ui** (Radix primitives) to `react-app` only.
   shadcn components are generated via its CLI into `src/components/ui/`; only components
   actually used are added (button, input, textarea, card, select, badge, tabs, alert,
   skeleton, and similar). The `@/` alias resolves to `src/` (wired in `vite.config.ts`
   and `tsconfig.json`).
2. The migration is a **presentation-layer change only**. Every behavior, prop, event
   handler, piece of state, and every tool/HTTP contract in CONTRACTS.md is preserved.
   The existing token palette (indigo accent, status colors) maps onto shadcn's CSS theme
   variables so the look stays coherent before the Phase 2 modernization evolves it.
3. **Delivery is two sequential phases** (mirroring the request): Phase 1 swaps elements
   for shadcn components without redesigning layouts; Phase 2 audits the whole app and
   applies minimal/clean modern design — consistent spacing/typography, clear hierarchy,
   hover/focus/loading affordances, soft-UI over heavy shadows.
4. Keyless/offline **test gates are unchanged** and must stay green. RTL tests prefer
   role/label queries and are updated only where the DOM legitimately changed; behavioral
   assertions are never weakened to accommodate the new markup.
5. `react-app`'s `package.json`, `tsconfig.json`, and `vite.config.ts` — frozen in prior
   rounds — are **unfrozen for this change only** (Tailwind/PostCSS/alias wiring requires
   it). The two MCP server packages are untouched.

## Consequences

- Net-new dependencies in `react-app`: `tailwindcss`, `postcss`, `autoprefixer`,
  `tailwindcss-animate`, `class-variance-authority`, `clsx`, `tailwind-merge`, and the
  Radix packages shadcn pulls per component. Heavier than hand-rolled CSS — accepted for
  a markedly better, more maintainable UI and faster future iteration.
- Some shadcn components (notably `Select`) render via Radix portals rather than native
  elements, so a few RTL tests must switch from native-`<select>` assertions to
  user-event interactions (or test the selection handler directly). Where jsdom + Radix
  is hostile, prefer asserting the handler/logic and keep a render smoke test — coverage
  of behavior must not drop.
- `ADR-005`'s "duplicated minimal types, no shared package" stance is unaffected; this is
  about the view layer, not shared contracts.
- Graduation path: a theme token file + dark mode are now trivial via shadcn's CSS
  variables if the team wants them later.

## Alternatives considered

- **Keep hand-rolled CSS, just restyle** — rejected by the product owner; they want the
  shadcn component system and its design language.
- **A different component library (MUI, Chakra, Mantine)** — the owner named shadcn/ui;
  it is copy-in (no runtime lock-in), Tailwind-native, and the team's stated preference.
- **Migrate and redesign in one pass** — riskier for behavior preservation and test
  stability; the two-phase split (migrate, then modernize) isolates the risky structural
  change from the aesthetic one.
- **Do the migration before the v1.2 feature work** — would force the feature agents to
  build against a moving UI base; features land first, then the overhaul migrates the
  final component set.
