# ADR-047: Planning assignment & capacity — bulk assign, inline points, per-developer capacity

## Status: Accepted (2026-07-01) — Phase C + Phase D implemented (react-app, frontend-only)

## Context

During sprint planning the team needs three things the Planning page did not yet offer:
1. **Bulk assignment** — assign many tickets to one developer in a single action (the per-row
   `<select>` is slow for a full sprint).
2. **Editing story points while planning** — the assignment table showed points read-only.
3. **Per-developer capacity** — the plotter showed a single team-level capacity %, not each
   developer's remaining points once their leaves/offset are entered.

All three are achievable with the tools that already exist — no new MCP surface is required.

## Decision (all frontend; reuses existing tools)

### Phase C — Planning assignment table (`AssignmentList.tsx`)
- **Bulk assign.** A leading checkbox column (+ a header select-all over the *visible* rows) and a
  toolbar that appears once rows are selected: **"N selected · Assign to [developer ▾] · Apply"**.
  Apply loops the existing per-row optimistic assign machine (`assignIssue`) over the selected keys,
  then clears the selection. Apply is disabled until a developer is chosen.
- **Inline points.** The read-only Pts cell becomes a small free-numeric `<input>`; on blur/Enter it
  writes via `updateTicketPoints` → the existing **`update_ticket`** tool (`{ ticketKey, storyPoints }`).
  Optimistic with revert-on-failure; a wheel guard (`blur` on wheel) prevents scroll from changing values.

### Phase D — Per-developer capacity (`LeavesPlotterCard.tsx`)
- A pure `computeDevCapacity(requiredPoints, leaveDaysByAssignee)` (`lib/capacity.ts`) yields, per
  developer, `capacity = max(0, requiredPoints − workingLeaveDays)`. `requiredPoints` (N) comes from
  the offset policy (`usePolicy()` → `health.policy.requiredPoints`, default 8); the leave-day counts
  reuse `computeCapacity().byAssigneeLeaveDays`. Rendered as a table under the capacity summary — e.g.
  a developer with 1 VL + 1 Offset shows **6 = 8 − 2**.

## Consequences
- **No new tool; jira tools stay 36; smoke unchanged.** C reuses `assign_issue` + `update_ticket`;
  D reuses `get_team_members` + `get_leaves` + `health.policy`. All IO shapes unchanged.
- Bulk assign and inline points **write real Jira** (per-row, non-fatal — a failed row reverts and shows
  an inline error without blocking the others). Offline gate only; live-verify with explicit go-ahead.
- The capacity model is deliberately simple (1 leave day = 1 point off N), matching the user's stated
  rule; it is a planning heuristic, not a commitment, consistent with ADR-016.

## Alternatives considered
- **A dedicated `bulk_assign` tool.** Rejected — a client-side loop over `assign_issue` needs no new
  surface, streams per-row status, and keeps failures isolated.
- **Points as a click-to-edit control (like the status cell).** Rejected in favour of an always-on
  input — fewer clicks when re-estimating a whole sprint; the wheel guard removes the accidental-change risk.
