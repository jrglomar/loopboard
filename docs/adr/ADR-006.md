# ADR-006: Optional dual-provider AI drafting endpoints on the Jira bridge (amends ADR-002)

## Status: Accepted (2026-06-11)

## Context

ADR-002 ruled out direct LLM calls from this codebase: spec §8 chose GitHub Copilot as
the AI surface to avoid extra subscription cost, and the in-UI ChatPanel was scoped to a
deterministic command router. During live testing the product owner explicitly requested
real AI in the dashboard: the Ticket Generator should behave like an AI chat that
analyses a terse one-liner, enhances it, and produces a detailed, well-structured ticket
pair before anything is created in Jira — and the ChatPanel `create`/`enhance` commands
should get the same treatment. The owner asked for **two providers** — the Anthropic API
and GitHub Models — switched by a config flag, so the team can use the free-tier GitHub
Models path (their PAT already exists) or paid Claude quality as they choose.

This is a user-directed scope change with cause; per the contract-first rule the change
is recorded here and in CONTRACTS.md §4.9 / Changelog v1.1.

## Decision

1. AI drafting lives **server-side in the mcp-jira HTTP bridge** as two REST endpoints
   (`POST /api/ai/draft-tickets`, `POST /api/ai/enhance-ticket`). API keys never reach
   the browser (spec §4.4 still holds).
2. Providers sit behind an **`AiProvider` port**: `anthropic` (official
   `@anthropic-ai/sdk`, `messages.parse` + `zodOutputFormat` structured outputs, default
   model `claude-opus-4-8`) and `github` (GitHub Models REST via fetch — no official SDK
   exists — default model `openai/gpt-4o-mini`, JSON mode + one re-ask retry).
   `AI_PROVIDER` env switches; unset = **disabled**.
3. The endpoints are **NOT MCP tools**: they are absent from the tool registry, the stdio
   server, and `GET /api/tools`. Copilot/Claude keeps drafting via the §4.8 MCP prompts —
   exposing an LLM call as a tool to an LLM caller is circular.
4. **Deterministic fallback is preserved**: with `AI_PROVIDER` unset the endpoints return
   `503 AI_UNAVAILABLE` and the React app falls back to the v1.0 template engine, so the
   POC still runs and all test gates still pass keyless/offline (ADR-004 unchanged for
   tools; the AI module is mocked in tests).
5. The AI endpoints never touch Jira — clients compose them with the existing
   `get_ticket`/`create_*`/`update_ticket` tools, keeping Jira I/O on one code path.

## Consequences

- ADR-002 is **amended, not repealed**: the ChatPanel command set stays finite and
  deterministic in structure; only the *content generation* inside `create`/`enhance` is
  AI-powered when enabled. Free-form NL remains Copilot's job.
- Two new failure modes surface in the UI (`AI_UNAVAILABLE`, provider `UPSTREAM` errors)
  with explicit fallback affordances.
- New env surface: `AI_PROVIDER`, `ANTHROPIC_API_KEY`, `ANTHROPIC_MODEL`,
  `GITHUB_MODELS_TOKEN`/`GITHUB_TOKEN` fallback, `GITHUB_MODELS_MODEL`,
  `GITHUB_MODELS_BASE_URL`.
- Cost: Anthropic path is paid per token; GitHub Models free tier is rate-limited —
  acceptable for a POC, revisit before team-wide rollout.
- zod is consumed through two subpaths in mcp-jira (`zod` v3 API for tools, `zod/v4` for
  AI schemas — required by the SDK helper). Contained to `src/lib/ai/`.

## Alternatives considered

- **Keep AI in Copilot only** (status quo) — rejected by the product owner; the dashboard
  is where POs/SMs work.
- **Smarter deterministic templates** — no real analysis/enhancement; quality ceiling too
  low for the stated goal.
- **Browser-side LLM calls** — leaks keys to the client; violates spec §4.4. Rejected.
- **Single provider (Anthropic only)** — owner explicitly wants a GitHub Models option on
  the existing PAT; the port pattern makes the second adapter cheap.
- **Registering the AI as an MCP tool** — circular for Copilot, pollutes the tool list,
  and would put LLM latency inside tool-call budgets. Rejected.
