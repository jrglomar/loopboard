# ADR-034: Multi-repo PRs + approval from Jira Development Information

## Status: Accepted (2026-06-26)

## Context

ADR-031 made the Huddle code-review card auto-list PRs by enumerating a **single** GitHub repo
(`GITHUB_REPO`) via `list_prs` and matching detected Jira keys; ADR-033 added approval badges by
calling GitHub's reviews endpoint per PR (also single-repo). The team actually works across **multiple
repositories**, and PRs are linked to Jira **automatically** — by putting the uppercase issue key
(e.g. `VRDB-123`) in the branch name, commit message, or PR title, which the *GitHub for Jira* app uses
to attach the PR to that issue's **Development** panel.

So the single-`GITHUB_REPO` model structurally misses PRs in other repos, and requires a GitHub token
with access to each repo. Jira already aggregates the links across every connected repo.

## Decision

Read PRs from **Jira's Development Information**, per sprint issue, instead of enumerating a GitHub
repo. New mcp-jira tool `get_issue_pull_requests { keys[] }`:

- For each key: resolve the numeric issue id (`GET /rest/api/3/issue/{key}?fields=*none`), then
  `GET /rest/dev-status/1.0/issue/detail?issueId={id}&applicationType={JIRA_DEV_STATUS_APP_TYPE}&dataType=pullrequest`.
- Pure `parseDevStatusPullRequests(json)` reduces `detail[].pullRequests[]` to `LinkedPr[]`
  (`url`, `title`, `repo`, `status`, `decision`, `approvals`, `reviewers`). `decision` derives from the
  reviewers' `approvalStatus` (changes_requested → approved → review_required) — so the same
  `ReviewBadge` renders it. Being pure, it is fully unit-tested against a captured fixture.
- Per-key failures (404 / no dev data) → `[]`; fetched in parallel; **read-only**. `applicationType`
  is configurable (`JIRA_DEV_STATUS_APP_TYPE`, default `GitHub`).

Frontend: `getIssuePullRequests` client + `useIssuePullRequests(keys)` hook. `PullRequestsCard`'s auto
section switches from `usePrs` + `usePrReviews` (single-repo) to `useIssuePullRequests(sprintKeys)` —
listing PRs across **all** repos with their approval badge. The manual PR store stays as a supplement.

`get_pr_reviews` (ADR-033) and `list_prs` remain registered tools (single-repo deep checks / Copilot),
just no longer the Huddle's source.

## Consequences

- The Huddle shows every repo's PRs for the sprint's tickets, with approval status, using the linking
  the team already relies on — no per-repo GitHub config, no missed repos.
- **Caveat:** `/rest/dev-status/…` is an **undocumented** Jira endpoint (the one behind the Development
  panel). It is parsed defensively and its exact JSON shape is confirmed against a live Jira before we
  trust it in production. If Jira changes it, only the pure parser + one client method are affected.
- Approval granularity follows Jira's `approvalStatus` (APPROVED / CHANGES_REQUESTED|NEEDS_WORK /
  none), coarser than GitHub's per-review history — acceptable for a glanceable badge.
- jira tools 31 → 32.

## Alternatives considered

- **Multi-repo `list_prs`** (config a list of repos, fan out). Needs a GitHub token per repo, a
  maintained repo list, and still re-derives the key match Jira already did. Rejected.
- **Jira remote links** (`/issue/{key}/remotelink`). Documented, but the team's *automatic* linking is
  via dev info, not remote links, and remote links carry no review/approval data. Rejected.
- **GraphQL across repos.** Same per-repo token/enumeration problem as multi-repo `list_prs`.
