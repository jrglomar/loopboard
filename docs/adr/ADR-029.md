# ADR-029: In-app AI Q&A assistant — tool-calling agent loop over read-only tools

## Status: Accepted (2026-06-24)

## Context

The dashboard `ChatPanel` is a deterministic command router (ADR-002) and explicitly sends free-form
questions to VS Code Copilot. The team wants to ask cross-app questions *inside* the app — "what are
our impediments today?", "who's on leave?", "what's in code review?" — and get a synthesized answer.

The data already exists: each such question maps to an existing mcp-jira **read tool**
(`get_impediments`, `get_leaves`, `get_active_sprint`, `get_pull_requests`, `get_daily_huddle`,
`get_sprint_report`, …). What's missing is **agentic orchestration**: an LLM that picks which tools to
call, calls them, and writes the answer. The existing AI endpoints (`/api/ai/draft-tickets`,
`/api/ai/plan-dev-tickets`) do a *single* structured-output call — not multi-step tool use.

Key reuse: a `ToolDef` (`name` + `description` + zod `schema`, `lib/toolDef.ts`) maps **1:1 onto an
LLM function-calling spec**, so the assistant's tool list is generated from the registry with no
duplication — exactly what MCP itself does for Copilot.

## Decision

Add `POST /api/ai/ask` running a **provider-agnostic tool-calling loop** over a **read-only allowlist**
of the mcp-jira registry. This is consistent with ADR-006 (AI lives only on the mcp-jira bridge, behind
`AI_PROVIDER`, with a deterministic fallback) and amends ADR-002 (the command router stays
deterministic; the AI path is additive when AI is enabled).

- **Port:** add `chatWithTools(system, messages, tools)` to `AiProvider`, returning a **normalized**
  result — `{ type: "final", text }` or `{ type: "tool_calls", calls: [{ id, name, args }] }` — and
  accepting prior tool results in a normalized message shape, so the loop is provider-agnostic.
- **Adapters:** Anthropic via `messages.create({ tools, … })` (`tool_use` ↔ `tool_result`); GitHub via
  OpenAI-style `chat/completions` with `tools` (`tool_calls` ↔ `role:"tool"`). Each tool's zod schema →
  JSON Schema via `zod-to-json-schema` (`$refStrategy: "none"`, inline).
- **`askService.ts`:** the loop — system prompt with current context (boardId, active sprintId, today's
  date), the read allowlist as function specs, capped at ~5 turns, calling `ToolDef.handler`
  **in-process** (same registry `http.ts` already imports). Returns `{ answer, toolsUsed, provider,
  model }`.
- **Read-only safety:** an explicit allowlist constant of `get_*`/`list_*` tools. Writes
  (`create_*`/`update_*`/`set_*`/`assign_*`/`transition_*`/`move_*`) are **never** offered — the
  assistant cannot mutate Jira.
- **Frontend:** `ChatPanel` routes non-command input to `aiAsk` when AI is on; commands stay
  deterministic; AI-off keeps the help fallback.

## Consequences

- A genuine in-app assistant for daily questions, reusing every read tool for free; new read tools are
  available to it the moment they're added to the allowlist.
- First agentic/tool-calling path in the codebase — both providers gain a `chatWithTools` method.
- Read-only by construction, so there's no write-safety surface to worry about in this iteration.
- Cost/latency: a question may take several model+tool round-trips; capped to bound it.

## Alternatives considered

- **Lean on Copilot via MCP only.** Already works in VS Code (and is the documented home for free-form
  Q&A), but doesn't serve users inside the web app — which is what was asked for.
- **Expose write tools too** ("move DEV-1 to In Progress"). Deferred — would need a confirm step;
  out of scope for a Q&A assistant.
- **Cross-server tools (live GitHub PRs via mcp-github).** Different process; the MVP uses the mcp-jira
  PR store (`get_pull_requests`). A later extension can bridge to mcp-github.
- **Hand-rolled zod→JSON-Schema.** Fragile; `zod-to-json-schema` (already transitively present via the
  MCP SDK) is battle-tested — promoted to a direct dependency.
