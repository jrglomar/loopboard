# ADR-028: PO-first ticket generator in a drawer (optional Dev task)

## Status: Accepted (2026-06-24)

## Context

The ticket generator (`TicketGen`) always created a **PO story + Dev task pair**. Since v1.11 the
**Linking** page owns bulk PO→Dev creation (and per-draft refinement), so the generator's job on the
Planning page is really to capture a **new PO story** quickly; creating a Dev task alongside it is now
the exception, not the rule. The user asked for: the generator in a **drawer** opened by a button, and
PO-only by default with **optional** Dev creation.

`TicketGen` already supports two-sprint targets (PO sprint + Dev sprint) and AI drafting that produces
both a PO and a Dev draft. The change is about **what gets created** and **how it's surfaced**, not the
drafting itself.

## Decision

### PO-first with an optional Dev toggle

- `TicketGen` adds a `createDev` state (**default false**) and a checkbox **"Also create a linked Dev
  task"**.
- When **off**: only the PO pane + PO sprint select show; **Create** calls a new `create_po_ticket`
  wrapper (`createPoTicket`) and the success panel shows just the PO link.
- When **on**: the Dev pane + Dev sprint select appear and **Create** uses the existing
  `createTicketPair` (PO then linked Dev) — unchanged behavior.
- AI chat, deterministic fallback, and per-draft Regenerate are untouched (they still draft a pair; the
  Dev draft is simply ignored unless the toggle is on).

### Drawer on Planning

- Planning's inline "Draft Tickets" section becomes a **New ticket** button that opens a shadcn
  **Sheet** (side drawer) hosting `TicketGen`, pre-seeded with the planning sprint (PO board → PO
  sprint). New `sheet.tsx` primitive (Radix Dialog-based, like the existing `dialog.tsx`).

## Consequences

- The common case (log a PO story) is one click and one form; Dev creation is a deliberate opt-in, with
  bulk PO→Dev still on Linking.
- No MCP/tool/endpoint change — purely frontend composition (`create_po_ticket` already exists).
- The PO sprint select benefits from the Phase A `useMCP` race fix, so it shows PO sprints reliably.
- Test impact: `TicketGen` creation tests that assert the pair now enable the toggle first; new tests
  cover the PO-only default. Planning tests open the drawer before asserting `TicketGen`.

## Alternatives considered

- **Keep always-pair, just add a drawer.** Rejected — the user explicitly wants PO-only by default, and
  pair creation now duplicates the Linking page.
- **A brand-new PO-only component.** Would duplicate the substantial AI-chat/fallback/draft logic; a
  small `createDev` branch in `TicketGen` reuses it.
- **A modal dialog instead of a side sheet.** A side drawer fits a longer form better and matches the
  "drawer" request.
